# Quality Gate

実装完了後の品質チェック〜コミットまでの一連のプロセスを実行する。
**3フェーズ検証（標準テスト×2 → 視点違いテスト×2 → 外部監査×1）+ Guardrail 判定 + commit まで一括実行。**

**省略・入れ替え禁止。**

## タスク

$ARGUMENTS

---

## 実行手順

### Phase 0: Four Eyes Test（4観点テスト — 省略禁止）

**Phase A に入る前に、必ず `/four-eyes-test` を実行すること。**

4観点（エンジニア→データ→ユーザー→壊す人）で検証し、全てPASSして初めて Phase A に進む。
詳細: `_common/FOUR_EYES_TEST.md`

Four Eyes Test が FAIL した場合:
1. 失敗した観点の問題を修正
2. Four Eyes Test を観点1からやり直し
3. 全観点 PASS 後に Phase A へ

---

### Phase A: 標準テスト ×2（安定性確認）

同じテストスイートを2回実行して不安定テストを検出する。
1回目で失敗したら**ここで止めて修正する**（Phase B に進まない）。

```bash
npm test
```

1回目通過 → 2回目を実行:

```bash
npm test
```

2回とも PASS したら Phase B へ。

---

### Phase B: 視点違いテスト ×2（セキュリティ・バイパス視点）

同じテストスイートを「**攻撃者・バイパス試行の視点**」で2回評価する。

**B-1: テストカバレッジギャップ確認**

以下を実行して、カバーされていないバイパスパターンがないか確認する:

1. `tests/` の既存テストケース一覧を読む
2. `_templates/hooks/tool-risk.js` の全 BLOCK/HIGH パターンを読む
3. テストされていないパターン変形（unicode混入、シェル演算子、引用符エスケープ等）がないかチェック
4. 未テストのバイパスベクタが見つかった場合 → テストを追加してから `npm test` を再実行

**B-2: 2回目の視点テスト（設定ファイル整合性）**

1. `_templates/settings.json` の deny リストと `tool-risk.js` の BLOCK パターンに重複漏れがないか確認
2. `scripts/auto-repair.js` の health check が現在の全 BLOCK パターンをカバーしているか確認
3. 問題が見つかった場合 → 修正してから Phase A に戻る

---

### Phase C: 外部監査（Sentinel）×1

**前の会話コンテキストをリセットした独立エージェントとして**、以下の3観点でコードレビューを実施する。

> 重要: このステップは「今まで何を作っていたか」を意図的に忘れ、コードを初めて見る第三者として評価する。

**C-1: セキュリティ（S）**
- ハードコードされたシークレット・API キーがないか
- 外部入力のバリデーション漏れ、インジェクション脆弱性
- `tool-risk.js` 自体のロジックに BLOCK を回避できる条件分岐がないか

**C-2: 設計一貫性（D）**
- ビジネス版（business）とエンジニア版の divergence が意図通りか
- `SAFETY_GATE_PATTERNS` と `HIGH_RISK_PATTERNS` に重複・矛盾がないか
- メッセージの blockMsg / highMsg フォーマットが全パターンで統一されているか

**C-3: エッジケース（E）**
- 空コマンド、null、undefined 入力での動作
- 複数フラグ組み合わせ（例: `rm -r -f` vs `rm -rf`）
- パイプ・セミコロン・バッククォートによる連結コマンドでの評価漏れ

**問題が見つかった場合:**
1. 問題を列挙して報告
2. 修正を実施
3. Phase A からやり直す

**問題がなければ:** 「Sentinel: PASS」と宣言して Step 1 へ。

---

### Step 1: Guardrail レベル判定

| Level | 条件 | 対応 |
|-------|------|------|
| **L1** | ドキュメントのみ、軽微な修正 | 簡易確認で通過 |
| **L2** | 通常の機能開発、バグ修正、テスト追加 | 全 Phase 実施済みなら通過 |
| **L3** | 大規模変更、破壊的変更 | Phase C の厳格実施 + 影響分析 |
| **L4** | セキュリティ脆弱性、データ整合性リスク | 即時停止 → ユーザーに報告 |

「Guardrail: L{N} ({理由})」と明示。

---

### Step 2: ドキュメント更新（該当する場合のみ）

フェーズ完了やパターン追加を伴う変更の場合、以下を更新:
- `docs/SECURITY_ARCHITECTURE.md` — 新パターンの設計根拠
- `docs/DESIGN_DECISIONS.md` — ADR 追記

該当しない場合はスキップ。

---

### Step 3: git commit

全 Phase 通過後にコミットする。

- `git status` で変更内容を確認
- `git diff` で差分を確認
- `git log --oneline -5` でコミットメッセージスタイルを確認
- 適切なコミットメッセージで commit する

---

### Step 4: ARIS 成功/失敗パターン記録

**成功パターン記録条件** → `/log-success` を実行:
- フェーズやマイルストーンが完了した
- 想定より大幅に効率的な手法を発見・適用できた

**失敗パターン記録条件** → `/log-failure` を実行:
- 途中で設計ミス・判断ミスがあり修正した
- Phase A/B/C でやり直しが発生した

記録対象がなければスキップ。

---

### Step 5: 完了報告

ユーザーに以下を簡潔に報告する:

- Phase A: テスト結果（pass/fail 件数）×2
- Phase B: カバレッジギャップ・設定整合性確認結果
- Phase C: Sentinel 監査結果
- Guardrail: L{N}
- コミットハッシュ

---

## ルール

- Phase A → B → C → Step 1 の順序は絶対。省略・スキップ禁止
- Phase A で失敗したら B/C に進まず修正する
- Phase C で問題が見つかった場合は Phase A に戻る
- L4 判定は即座にユーザーに報告して停止する
- このスキルは `implement` スキルで実装したものを push する前に必ず実行する
