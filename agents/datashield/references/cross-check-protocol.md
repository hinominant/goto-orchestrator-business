# datashield クロスチェックプロトコル

> 本ドキュメントは datashield（データ保護監視エージェント）が privacy（主査）の成果物を検証する際のプロトコルを定義する。
> privacy の評価・報告を「信頼するが検証する」原則に基づき、技術的実効性とグローバル規制視点で独立検証を行う。

---

## 1. クロスチェックの原則

### 1.1 検証の目的

privacy エージェントは日本の個人情報保護法（APPI）を主軸に評価を行う。datashield はその成果物に対し、以下の観点で独立したクロスチェックを実施する。

1. **GDPR視点での検証**: APPIだけでは見落とす可能性のある国際基準の要件を補完する
2. **技術的実効性の確認**: 規程・ポリシー上の対策が実際にシステムレベルで実装・機能しているかを検証する
3. **盲点の発見**: privacy が見落としたリスク、過小評価したリスクを独立した視点で検出する

### 1.2 検証の姿勢

- **対立ではなく補完**: privacy を否定するためではなく、評価品質を向上させるために行う
- **エビデンスベース**: 主観的な意見ではなく、技術的根拠・法的根拠に基づいて指摘する
- **優先度明確化**: すべての指摘に重要度を付与し、対応の優先順位を明確にする

### 1.3 検証スコープ

| 検証対象 | 検証内容 |
|---|---|
| privacy のリスク評価レポート | リスク評価の妥当性、見落としの有無 |
| 安全管理措置の評価結果 | 技術的実効性の確認 |
| 越境移転の評価 | データフローの実態との整合性 |
| 漏えい対応体制の評価 | 実動可能性のシミュレーション |
| 改善勧告の内容 | 実現可能性、優先度の妥当性 |

## 2. privacy の評価に対する検証手順

### 2.1 安全管理措置の技術的実効性チェック

privacy が「対策済み」と評価した安全管理措置について、技術的に実効性があるかを検証する。

#### 暗号化の検証

```
検証項目:
  □ 保存時暗号化
    - 暗号化アルゴリズムは適切か（AES-256推奨、DES/3DES等の旧式は不可）
    - 鍵管理は適切か（HSM利用、鍵のローテーション、アクセス制御）
    - データベースレベル/ファイルレベル/ディスクレベルのどこで暗号化されているか
    - 暗号化対象に漏れはないか（バックアップ、ログ、一時ファイル）

  □ 通信暗号化
    - TLSバージョンは1.2以上か
    - 証明書の有効性・更新管理は適切か
    - 内部通信（マイクロサービス間等）も暗号化されているか
    - 証明書ピンニングが必要なケースで実装されているか

  □ 鍵管理
    - 暗号鍵はデータと分離保管されているか
    - 鍵のローテーションポリシーが定められ実行されているか
    - 鍵へのアクセスログが取得されているか
    - 緊急時の鍵失効手順があるか
```

#### アクセス制御の検証

```
検証項目:
  □ 認証
    - 多要素認証（MFA）が有効化されているか（全管理者、特権アカウント）
    - パスワードポリシーは適切か（長さ、複雑性、有効期限）
    - サービスアカウント・APIキーの管理は適切か
    - 認証失敗時のロックアウトポリシーがあるか

  □ 認可
    - RBAC/ABACが適切に設計されているか
    - 最小権限の原則が適用されているか
    - 権限の定期棚卸し（最低四半期ごと）が実施されているか
    - 特権アカウントの利用が監視・記録されているか

  □ セッション管理
    - セッションタイムアウトが設定されているか
    - 同時セッション数の制限があるか
    - セッション固定攻撃への対策があるか
```

#### ログ管理の検証

```
検証項目:
  □ 取得範囲
    - 個人データへのアクセスログが取得されているか
    - 管理操作（権限変更、設定変更等）のログが取得されているか
    - 認証ログ（成功・失敗）が取得されているか

  □ 保全
    - ログの保存期間は十分か（最低1年推奨）
    - ログの改ざん防止措置があるか（WORM、ハッシュチェーン等）
    - ログが個人データとは別の安全な場所に保存されているか

  □ 監視
    - 異常検知のアラートが設定されているか
    - ログの定期レビューが実施されているか
    - SIEM等の統合監視ツールが導入されているか
```

### 2.2 越境移転リスクの実際のデータフロー検証

privacy の越境移転評価に対し、実際のデータフローを検証する。

#### 検証手順

1. **データフローマッピングの取得**
   - privacy が作成したデータフロー図を取得
   - システム構成図・ネットワーク図と照合

2. **実際のデータ送信先の確認**
   ```
   検証項目:
     □ 外部API呼び出し先のIPアドレス/ドメインのGeoIP確認
     □ CDN/ロードバランサーの配置先の確認
     □ クラウドサービスのリージョン設定の確認
     □ バックアップデータの保存先リージョンの確認
     □ 障害時のフェイルオーバー先リージョンの確認
     □ ログ集約サービスの保存先の確認
     □ SaaSサービスのデータ処理場所の確認
   ```

3. **契約・同意との整合性確認**
   ```
   検証項目:
     □ 実際のデータ送信先の国が、同意文書に記載されているか
     □ 委託契約に越境移転に関する条項が含まれているか
     □ SCCs（標準契約条項）等の適切な保護措置が締結されているか
     □ 移転先国の法制度変更（政府アクセス要求等）を定期的に確認しているか
   ```

4. **GDPR視点の追加検証**
   ```
   検証項目:
     □ Schrems II判決以降の追加保護措置（補完的措置）が講じられているか
     □ TIA（Transfer Impact Assessment）が実施されているか
     □ 移転先国の政府によるデータアクセスリスクが評価されているか
   ```

### 2.3 漏えい報告体制のシミュレーション

privacy が「整備済み」と評価した漏えい報告体制について、実際に機能するかシミュレーションで検証する。

#### シミュレーションシナリオ

**シナリオ1: 営業時間内の大規模漏えい**
```
状況: 平日14:00、外部からの通報により1万件の顧客データ流出が判明
検証ポイント:
  □ 第一報の連絡先は明確か、連絡がつくか
  □ 対応チームは30分以内に招集可能か
  □ 初動対応（該当システムの遮断等）の手順は明確か
  □ 速報作成の担当者・手順は明確か
  □ 本人通知の手段・テンプレートは準備されているか
```

**シナリオ2: 深夜・休日の不正アクセス**
```
状況: 土曜深夜2:00、IDS/IPSのアラートにより不正アクセスを検知
検証ポイント:
  □ 24時間対応可能な連絡体制があるか
  □ 夜間・休日のエスカレーションルートが定まっているか
  □ 初動対応（ネットワーク切断等）の権限が夜間担当者にあるか
  □ フォレンジック調査の外部委託先との連絡手段があるか
```

**シナリオ3: 委託先での漏えい**
```
状況: 委託先企業から「御社の個人データを含むデータが流出した可能性がある」と連絡
検証ポイント:
  □ 委託先からの連絡ルートが定まっているか
  □ 委託先の調査に協力・監督する体制があるか
  □ 報告義務の主体（委託元）としての対応手順が明確か
  □ 委託契約に漏えい時の対応義務が明記されているか
```

**シナリオ4: AI経由の個人情報漏洩**
```
状況: 社内AIチャットボットが、他部署の従業員の個人情報をユーザーに回答した
検証ポイント:
  □ AI起因の漏えいがインシデント対応手順の対象に含まれているか
  □ AI出力ログの保全手順があるか
  □ 影響範囲の特定方法（どのユーザーに何が表示されたか）が整備されているか
  □ AIシステムの緊急停止手順があるか
```

#### シミュレーション評価基準

| 評価項目 | 合格基準 |
|---|---|
| 初動対応時間 | 事態認知から30分以内に対応チーム招集開始 |
| エスカレーション | 適切な権限者への報告が1時間以内に完了 |
| 証拠保全 | フォレンジック対応が24時間以内に開始可能 |
| 速報提出 | 3営業日以内に個人情報保護委員会へ速報提出可能 |
| 本人通知 | 通知テンプレート・手段が事前準備されている |
| 文書化 | 対応経緯の記録が時系列で残される体制がある |

## 3. 差し戻し基準と承認基準

### 3.1 差し戻し基準（以下のいずれかに該当する場合、privacy に差し戻す）

#### Critical（即時差し戻し）

以下に該当する場合、対応完了まで承認しない。

| 基準 | 具体例 |
|---|---|
| 法令違反の見落とし | 第三者提供の同意未取得を「問題なし」と評価している |
| 技術的措置の重大な欠陥 | 暗号化が実装されていないのに「暗号化済み」と評価している |
| 報告対象漏えいの見落とし | 報告義務がある事象を「報告不要」と評価している |
| 越境移転の未検出 | 実際に海外にデータが送信されているのに未検出 |
| 要配慮個人情報の取扱い不備 | 同意なしの取得・オプトアウト提供を見落としている |

#### Major（修正要求付き差し戻し）

以下に該当する場合、修正を求めて差し戻す。修正後に再検証する。

| 基準 | 具体例 |
|---|---|
| リスク評価の過小評価 | 高リスクを中リスクと評価しており、根拠が不十分 |
| 安全管理措置の検証不足 | 規程の存在は確認しているが、実装・運用の確認がない |
| 改善勧告の実現可能性不足 | 改善勧告がありますが具体的な実装方法・期限が示されていない |
| GDPR要件の未考慮 | EU居住者のデータを扱うのにGDPR要件が検討されていない |
| チェック項目の欠落 | 安全管理措置の4分類のうち一部が未評価 |

#### Minor（コメント付き承認）

以下に該当する場合、コメントを付けて承認する。次回評価時に改善を確認する。

| 基準 | 具体例 |
|---|---|
| 記載の曖昧さ | 評価根拠の記載が不十分だが結論は妥当 |
| 軽微な改善提案 | ベストプラクティスとの差異があるが法令違反ではない |
| 将来リスクの指摘 | 現時点では問題ないが法改正等で対応が必要になる可能性 |

### 3.2 承認基準

以下の全項目を満たす場合に承認する。

```
承認条件:
  □ Critical に該当する指摘がないこと
  □ Major に該当する指摘がすべて解消されていること
  □ 法令要件（APPI全条文）のカバー率が100%であること
  □ 安全管理措置4分類すべてが評価されていること
  □ 技術的実効性の確認が実施されていること
  □ 越境移転の有無が実際のデータフローに基づいて評価されていること
  □ 漏えい対応体制の実動可能性が確認されていること
  □ Minor 指摘がある場合、コメントとして記録されていること
```

### 3.3 判定フロー

```
privacy の成果物受領
  │
  ├─ Phase 1: 形式チェック（所要: 即時）
  │   □ 必須セクションの充足性
  │   □ 評価対象の網羅性
  │   → 不備あり → 差し戻し（形式不備）
  │
  ├─ Phase 2: 法令準拠チェック（所要: 短時間）
  │   □ APPI条文カバー率
  │   □ 要配慮個人情報の取扱い
  │   □ 第三者提供・越境移転の評価
  │   → Critical検出 → 即時差し戻し
  │
  ├─ Phase 3: 技術的実効性チェック（所要: 中程度）
  │   □ 暗号化の実装検証
  │   □ アクセス制御の実装検証
  │   □ ログ管理の実装検証
  │   → Major検出 → 修正要求付き差し戻し
  │
  ├─ Phase 4: データフロー検証（所要: 中程度）
  │   □ 越境移転の実態検証
  │   □ 委託先のデータフロー検証
  │   → Major検出 → 修正要求付き差し戻し
  │
  ├─ Phase 5: シミュレーション検証（所要: 長時間）
  │   □ 漏えい対応のシミュレーション
  │   □ 開示等請求対応のシミュレーション
  │   → Major検出 → 修正要求付き差し戻し
  │
  └─ Phase 6: 総合判定
      □ 全Phaseの結果統合
      □ Minor指摘の記録
      → 承認 / コメント付き承認
```

## 4. レポートフォーマット

### クロスチェックレポート

```yaml
# datashield クロスチェックレポート

meta:
  report_id: "DC-YYYY-MM-DD-NNN"
  対象成果物: "[privacy の成果物名・ID]"
  検証日: "YYYY-MM-DD"
  検証者: "datashield"
  判定結果: "承認 / コメント付き承認 / 差し戻し（Major） / 差し戻し（Critical）"

summary:
  総合評価: "[A: 優良 / B: 合格 / C: 要改善 / D: 不合格]"
  Critical指摘数: N
  Major指摘数: N
  Minor指摘数: N
  主要所見: |
    [2-3行で主要な所見を記載]

findings:
  - id: "F-001"
    重要度: "Critical / Major / Minor"
    カテゴリ: "安全管理措置 / 越境移転 / 漏えい対応 / 法令準拠 / その他"
    タイトル: "[指摘の要約]"
    詳細: |
      [指摘の詳細説明]
    根拠: |
      [法的根拠・技術的根拠]
    privacy_評価: |
      [privacy がどのように評価していたか]
    datashield_検証結果: |
      [datashield の検証結果]
    推奨対応: |
      [具体的な対応方法]
    対応期限: "YYYY-MM-DD"

  - id: "F-002"
    # ... 同様のフォーマットで列挙

verification_details:
  安全管理措置_技術検証:
    暗号化:
      検証実施: "Yes / No"
      結果: "[検証結果の詳細]"
    アクセス制御:
      検証実施: "Yes / No"
      結果: "[検証結果の詳細]"
    ログ管理:
      検証実施: "Yes / No"
      結果: "[検証結果の詳細]"

  越境移転_検証:
    データフローマッピング確認: "Yes / No"
    実際の送信先確認: "Yes / No"
    契約との整合性: "Yes / No"
    結果: "[検証結果の詳細]"

  漏えい対応_シミュレーション:
    実施シナリオ: "[実施したシナリオ名]"
    結果: "[シミュレーション結果の詳細]"

previous_minor_followup:
  - id: "[前回のMinor指摘ID]"
    内容: "[前回の指摘内容]"
    今回の確認結果: "改善済み / 未改善 / 一部改善"

next_actions:
  privacy_対応事項:
    - "[privacy が対応すべき事項1]"
    - "[privacy が対応すべき事項2]"
  datashield_次回検証予定:
    - "[次回検証で確認する事項1]"
    - "[次回検証で確認する事項2]"
```

### レポート記入ルール

1. **findings セクション**: 指摘は重要度の高い順に記載する。同一重要度内では法令準拠 → 技術的実効性 → 運用体制の順に記載する
2. **根拠の明示**: すべての指摘に法的根拠（条文番号）または技術的根拠（規格・ベストプラクティス）を明記する
3. **推奨対応の具体性**: 「改善すべき」ではなく「XXXをYYYに変更する」レベルで具体的に記載する
4. **対応期限**: Critical は即時（最長1週間）、Major は2週間以内、Minor は次回評価時までに設定する
5. **前回指摘のフォロー**: 前回のMinor指摘のフォローアップを必ず含める
