## 1. TSE上場審査基準

### 1.1 形式基準（数値基準）

#### グロース市場

- [ ] 流通株式数が1,000単位以上
  - 根拠: 有価証券上場規程第212条第1号
  - 証跡: 株主名簿、流通株式数計算書
  - リスク: HIGH（未充足で上場申請不可）

- [ ] 流通株式時価総額が5億円以上
  - 根拠: 有価証券上場規程第212条第2号
  - 証跡: 想定発行価格に基づく時価総額算定書
  - リスク: HIGH（未充足で上場申請不可）

- [ ] 流通株式比率が25%以上
  - 根拠: 有価証券上場規程第212条第3号
  - 証跡: 株式分布状況表
  - リスク: HIGH（未充足で上場申請不可）

- [ ] 事業継続年数が1年以上（取締役会設置後1年以上経過）
  - 根拠: 有価証券上場規程第212条第4号
  - 証跡: 登記簿謄本
  - リスク: HIGH（未充足で上場申請不可）

#### スタンダード市場

- [ ] 株主数が400人以上
  - 根拠: 有価証券上場規程第205条第1号
  - 証跡: 株主名簿
  - リスク: HIGH

- [ ] 流通株式数が2,000単位以上
  - 根拠: 有価証券上場規程第205条第2号
  - 証跡: 株主名簿、流通株式数計算書
  - リスク: HIGH

- [ ] 流通株式時価総額が10億円以上
  - 根拠: 有価証券上場規程第205条第3号
  - 証跡: 時価総額算定書
  - リスク: HIGH

- [ ] 流通株式比率が25%以上
  - 根拠: 有価証券上場規程第205条第4号
  - 証跡: 株式分布状況表
  - リスク: HIGH

- [ ] 事業継続年数が3年以上
  - 根拠: 有価証券上場規程第205条第5号
  - 証跡: 登記簿謄本、決算書3期分
  - リスク: HIGH

- [ ] 直前期の利益が1億円以上（経常利益または純利益）
  - 根拠: 有価証券上場規程第205条第6号
  - 証跡: 監査済み財務諸表
  - リスク: HIGH

#### プライム市場

- [ ] 株主数が800人以上
  - 根拠: 有価証券上場規程第204条第1号
  - 証跡: 株主名簿
  - リスク: HIGH

- [ ] 流通株式数が20,000単位以上
  - 根拠: 有価証券上場規程第204条第2号
  - 証跡: 株主名簿、流通株式数計算書
  - リスク: HIGH

- [ ] 流通株式時価総額が100億円以上
  - 根拠: 有価証券上場規程第204条第3号
  - 証跡: 時価総額算定書
  - リスク: HIGH

- [ ] 流通株式比率が35%以上
  - 根拠: 有価証券上場規程第204条第4号
  - 証跡: 株式分布状況表
  - リスク: HIGH

- [ ] 事業継続年数が3年以上
  - 根拠: 有価証券上場規程第204条第5号
  - 証跡: 登記簿謄本、決算書3期分
  - リスク: HIGH

- [ ] 直前2期の利益合計が25億円以上、または直前期の売上高が100億円以上かつ時価総額1,000億円以上
  - 根拠: 有価証券上場規程第204条第6号
  - 証跡: 監査済み財務諸表
  - リスク: HIGH

### 1.2 実質基準（質的基準）

- [ ] 企業の継続性及び収益性（事業計画の合理性）
  - 根拠: 有価証券上場規程第207条第1号/第213条第1号
  - 証跡: 事業計画書、予実管理資料、KPI推移
  - リスク: HIGH（審査の核心項目）

- [ ] 企業経営の健全性（関連当事者取引の適正性、役員構成）
  - 根拠: 有価証券上場規程第207条第2号/第213条第2号
  - 証跡: 関連当事者取引一覧、取締役会議事録、独立役員届出書
  - リスク: HIGH

- [ ] 企業のコーポレート・ガバナンス及び内部管理体制の有効性
  - 根拠: 有価証券上場規程第207条第3号/第213条第3号
  - 証跡: 内部統制報告書、監査役監査報告書、内部監査報告書、規程類一式
  - リスク: HIGH（IT統制はここに含まれる）

- [ ] 企業内容等の開示の適正性
  - 根拠: 有価証券上場規程第207条第4号/第213条第4号
  - 証跡: 有価証券報告書ドラフト、開示体制構築資料、IR体制図
  - リスク: HIGH

- [ ] その他公益又は投資者保護の観点から必要と認める事項
  - 根拠: 有価証券上場規程第207条第5号/第213条第5号
  - 証跡: コンプライアンス体制図、反社チェック結果
  - リスク: HIGH

## 2. J-SOX IT統制要件（2024年改訂版）

> 2024年4月施行の改訂実施基準を反映。IT統制の重要性が明確化され、ITの利用範囲が拡大している企業は評価範囲の適切な決定が求められる。

### 2.1 IT全社統制（IT Entity-Level Controls）

- [ ] IT戦略・IT投資計画が経営戦略と整合しているか
  - 根拠: 財務報告に係る内部統制の評価及び監査の基準（2024年改訂）II.2.(6)
  - 証跡: IT戦略計画書、IT投資計画書、取締役会承認議事録
  - リスク: HIGH（全社統制の不備は全般統制に波及）

- [ ] IT組織体制が明確に定義され、責任と権限が文書化されているか
  - 根拠: 実施基準II.2.(6)①
  - 証跡: IT組織図、職務分掌規程、RACI表
  - リスク: HIGH

- [ ] ITリスク管理方針が策定・運用されているか
  - 根拠: 実施基準II.2.(6)②
  - 証跡: ITリスク管理方針書、ITリスクアセスメント結果、リスク対応計画
  - リスク: HIGH

- [ ] ITに係る全般統制の方針が明確化されているか
  - 根拠: 実施基準II.2.(6)③
  - 証跡: IT統制方針書、IT統制基準書
  - リスク: HIGH

- [ ] IT人材の確保・育成計画があるか
  - 根拠: 実施基準II.2.(6)④
  - 証跡: IT人材育成計画、研修実績、資格取得状況
  - リスク: MEDIUM

- [ ] ITに関する法令・規制の遵守体制があるか
  - 根拠: 実施基準II.2.(6)⑤
  - 証跡: 法令遵守チェックリスト、個人情報保護方針
  - リスク: HIGH

### 2.2 IT全般統制（ITGC: IT General Controls）

#### 2.2.1 プログラムの開発・変更管理

- [ ] システム開発標準（SDLC）が定義・文書化されているか
  - 根拠: 実施基準II.3.(3)①
  - 証跡: システム開発標準書、開発ガイドライン
  - リスク: HIGH

- [ ] 要件定義→設計→開発→テスト→リリースの各段階で承認プロセスがあるか
  - 根拠: 実施基準II.3.(3)①
  - 証跡: 各フェーズの承認記録、レビュー記録
  - リスク: HIGH

- [ ] プログラム変更要求が文書化され、影響分析が実施されているか
  - 根拠: 実施基準II.3.(3)①
  - 証跡: 変更要求書、影響分析書
  - リスク: HIGH

- [ ] テスト計画・テスト結果が文書化され、承認されているか
  - 根拠: 実施基準II.3.(3)①
  - 証跡: テスト計画書、テスト結果報告書、テストエビデンス
  - リスク: HIGH

- [ ] 開発環境とテスト環境と本番環境が分離されているか
  - 根拠: 実施基準II.3.(3)①
  - 証跡: 環境構成図、アクセス制御設定
  - リスク: HIGH

- [ ] 本番環境への変更リリース手順が定義され、承認されているか
  - 根拠: 実施基準II.3.(3)①
  - 証跡: リリース手順書、リリース承認記録、デプロイログ

- [ ] 緊急変更手順が定義され、事後承認プロセスがあるか
  - 根拠: 実施基準II.3.(3)①
  - 証跡: 緊急変更手順書、緊急変更記録、事後承認記録
  - リスク: MEDIUM

  - 証跡: アクセス権限棚卸し結果、是正記録

## 3. アクセス管理チェックリスト

### 3.1 退職者ID管理

- [ ] 退職者のアカウント無効化が退職日当日（または翌営業日）に完了しているか
  - 根拠: 実施基準II.3.(3)②、ISMS A.9.2.6
  - 証跡: 退職日一覧 vs アカウント停止日一覧の突合表
  - リスク: HIGH（退職後のアクセスは重大リスク）

- [ ] 退職者が利用していた全システムのアカウントが漏れなく無効化されているか
  - 根拠: 実施基準II.3.(3)②
  - 証跡: 退職者別システムアクセス権限無効化チェックシート
  - リスク: HIGH

- [ ] 退職者のメールアカウント・クラウドサービスアカウントが無効化されているか
  - 根拠: 実施基準II.3.(3)②
  - 証跡: Google Workspace/Microsoft 365管理コンソールの停止記録
  - リスク: HIGH

- [ ] 退職者のVPN・リモートアクセス権限が無効化されているか
  - 根拠: 実施基準II.3.(3)②
  - 証跡: VPN管理ツールの設定変更記録
  - リスク: HIGH

- [ ] 退職者のSSHキー・APIキー・アクセストークンが失効しているか
  - 根拠: 実施基準II.3.(3)②
  - 証跡: キー管理台帳、失効処理記録
  - リスク: HIGH

### 3.2 特権ID管理

- [ ] 特権ID（root、admin、DBA等）の保有者一覧が最新化されているか

  - リスク: HIGH

- [ ] 特権IDの利用は事前申請・承認制になっているか

  - 証跡: 特権ID利用申請書・承認記録
  - リスク: HIGH

- [ ] 特権IDの操作ログが取得・保存されているか
  - 根拠: 実施基準II.3.(3)②
  - 証跡: 特権ID操作ログ（コマンド履歴含む）
  - リスク: HIGH

- [ ] 特権IDのパスワードが定期的に変更されているか
  - 根拠: 実施基準II.3.(3)②
  - 証跡: パスワード変更記録
  - リスク: MEDIUM

- [ ] 共有特権IDの利用が最小限に抑制されているか（個人特権IDへの移行）
  - 根拠: 実施基準II.3.(3)②
  - 証跡: 特権ID棚卸し結果、共有ID削減計画
  - リスク: HIGH

- [ ] 全システムのアクセス権限棚卸しが四半期以上の頻度で実施されているか
  - 根拠: 実施基準II.3.(3)②

## 8. 証跡管理の要件

### 8.1 4W1H（証跡の必須要素）

すべての証跡は以下の4W1Hを含むこと:

| 要素 | 内容 | 例 |
|------|------|-----|
| Who（誰が） | 操作者・承認者の識別 | ユーザーID、氏名、役職 |
| What（何を） | 実施した操作・判断の内容 | 「アクセス権限変更」「リリース承認」 |
| When（いつ） | 操作・判断の日時 | タイムスタンプ（NTP同期済み） |
| Where（どこで） | 対象システム・環境 | 「本番環境」「AWS ap-northeast-1」 |
| How（どのように） | 操作方法・手順 | 「管理コンソールから実施」「CLIで実行」 |

### 8.2 証跡の保存期間一覧

| 証跡種類 | 最低保存期間 | 推奨保存期間 | 根拠法令・基準 |
|----------|-------------|-------------|---------------|
| 操作ログ（一般） | 1年 | 3年 | 実施基準II.3.(3)③ |
| 認証ログ（ログイン/ログアウト） | 1年 | 3年 | 実施基準II.3.(3)②③ |
| 特権ID操作ログ | 3年 | 5年 | 実施基準II.3.(3)②③ |
| 変更管理記録 | 3年 | 5年 | 実施基準II.3.(3)① |
| アクセス権限棚卸し結果 | 3年 | 5年 | 実施基準II.3.(3)② |
| 監査証跡（内部監査報告書） | 5年 | 7年 | 金商法第193条の2 |
| 財務関連データ | 7年 | 10年 | 法人税法施行規則第59条 |
| 個人情報取扱記録 | 3年 | 5年 | 個人情報保護法第29条 |
| 委託先評価記録 | 3年 | 5年 | 個人情報保護法第25条 |
| インシデント対応記録 | 3年 | 5年 | 実施基準II.3.(3)③ |
| AI利用ログ | 1年 | 3年 | AI事業者ガイドライン |

## 12. IPO審査での頻出指摘事例

### 12.1 アクセス権限管理

| 指摘事例 | リスク | 対策 |
|----------|--------|------|
| 退職者のアカウントが退職後も有効なまま放置されている | HIGH | 退職日当日の自動無効化フロー構築、突合チェック四半期実施 |
| アクセス権限の棚卸しが実施されていない、または形骸化している | HIGH | 四半期棚卸しの実施、不要権限の即時削除、棚卸し結果の承認 |
| 特権IDが個人に紐付かず、共有されている | HIGH | 個人特権IDへの移行、共有ID利用時の利用者記録 |
| 職務分掌に反するアクセス権限が付与されている（開発者が本番DBに直接アクセス可能等） | HIGH | 職務分掌マトリクスの策定・実装、例外承認プロセス |
| パスワードポリシーが未設定または甘い | MEDIUM | 最低12文字、複雑性要件、MFA必須化 |

### 12.2 変更管理

| 指摘事例 | リスク | 対策 |
|----------|--------|------|
| 本番環境への変更がチケット管理されていない | HIGH | 全変更のチケット化必須化、チケットなしデプロイのブロック |
| リリース承認なしで本番デプロイが実施されている | HIGH | CI/CDパイプラインに承認ゲートを組み込む |
| 開発者が直接本番環境を変更できる（職務分掌違反） | HIGH | 本番アクセス権限の分離、デプロイは専用アカウント/パイプラインのみ |
| テスト実施の記録が不十分 | HIGH | テスト自動化、テスト結果の自動保存 |
| 緊急変更の事後承認が形骸化している | MEDIUM | 緊急変更レビュー会議の定例化 |

### 12.3 規程と運用の乖離

| 指摘事例 | リスク | 対策 |
|----------|--------|------|

| 規程の従業員への周知が不十分 | MEDIUM | 年次研修の実施、eラーニングの導入 |

## 14. 認証・規格一覧

### 14.1 ISMS（ISO/IEC 27001:2022）

| 項目 | 内容 |
|------|------|
| 正式名称 | ISO/IEC 27001:2022 情報セキュリティマネジメントシステム |
| 概要 | 情報セキュリティの管理体制を構築・運用・維持するための国際規格 |
| IPOとの関連 | 上場審査で情報セキュリティ体制の証明として高く評価される。取得義務はないが事実上の標準 |
| 取得目安時期 | N-2期での取得開始、N-1期での認証取得が理想 |
| 主な管理策 | 93管理策（2022年版）: 組織的、人的、物理的、技術的 |
| 有効期間 | 3年（年次サーベイランス審査あり） |
| 関連する附属書 | ISO/IEC 27002:2022（管理策の実施ガイダンス） |

### 14.2 SOC2（System and Organization Controls 2）

| 項目 | 内容 |
|------|------|
| 正式名称 | SOC 2 - Trust Services Criteria |
| 概要 | サービス組織の内部統制に関する保証報告書。AICPA基準 |
| IPOとの関連 | SaaS企業のIPOではほぼ必須。委託先管理の証明としても活用 |
| 5つのTrust Services Criteria | Security（必須）、Availability、Processing Integrity、Confidentiality、Privacy |
| Type I vs Type II | Type I: 特定日時点の設計評価。Type II: 期間（6-12ヶ月）における運用有効性評価 |
| 取得目安時期 | N-2期でType I、N-1期でType II |
| 有効期間 | 12ヶ月（年次更新） |

### 14.3 プライバシーマーク（Pマーク）

| 項目 | 内容 |
|------|------|
| 正式名称 | プライバシーマーク（JIS Q 15001:2023） |
| 概要 | 個人情報保護に関する日本独自の認証制度。JIPDEC運営 |
| IPOとの関連 | 個人情報を取り扱う事業ではIPO審査でプラス評価。BtoC事業では事実上必須 |
| 取得目安時期 | N-2期での取得開始 |
| 有効期間 | 2年 |
| ISMSとの違い | Pマークは個人情報保護に特化。ISMSは情報セキュリティ全般 |

### 14.4 ISO/IEC 42001:2023（AI マネジメントシステム）

| 項目 | 内容 |
|------|------|
| 正式名称 | ISO/IEC 42001:2023 人工知能マネジメントシステム |
| 概要 | AI開発・提供・利用の管理体制に関する国際規格。2023年12月発行 |
| IPOとの関連 | AIを事業の中核に据える企業では差別化要因。EU AI Act対応にも有効 |
| 主な要求事項 | AIリスクアセスメント、AI倫理方針、データ品質管理、透明性・説明可能性 |
| 取得目安時期 | AI事業会社はN-2期〜N-1期での取得を検討 |
| 有効期間 | 3年（年次サーベイランス審査あり） |
| 関連規格 | ISO/IEC 23894（AIリスクマネジメント）、ISO/IEC 38507（AIのガバナンス） |

### リスクレベル定義

| レベル | 定義 | 対応方針 |
|--------|------|----------|
| HIGH | 未対応の場合、上場審査での指摘リスクが高い。重大な不備となる可能性 | 最優先で対応。N-1期までに必ず整備完了 |
| MEDIUM | 上場審査で改善要求の可能性あり。軽微な不備と判断される可能性 | 計画的に対応。N-1期中に整備完了を目指す |
