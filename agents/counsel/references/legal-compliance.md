## 1. IT企業に関連する主要法律一覧

### 1.1 不正競争防止法

| 項目 | 内容 |
|------|------|
| 所管 | 経済産業省 |
| 目的 | 事業者間の公正な競争の確保 |

#### 営業秘密の3要件（第2条第6項）

1. **秘密管理性**: アクセス制限、秘密表示、施錠管理等により秘密として管理されていること
2. **有用性**: 事業活動に有用な技術上又は営業上の情報であること（ネガティブ情報も含む）
3. **非公知性**: 公然と知られていないこと（保有者の管理下以外では一般に入手できないこと）

#### 主要な不正行為類型（第2条第1項）

| 号 | 類型 | 概要 |
|----|------|------|
| 4号 | 不正取得 | 窃取・詐欺・強迫等による営業秘密の取得 |
| 5号 | 不正取得後使用・開示 | 不正取得した営業秘密の使用又は開示 |
| 7号 | 正当取得後の背信的使用 | 正当に取得した営業秘密を不正の利益目的等で使用・開示 |
| 11号 | 限定提供データの不正取得 | 2019年改正で追加 |

#### 罰則（第21条・第22条）

| 対象 | 懲役 | 罰金 |
|------|------|------|
| 個人（国内） | 10年以下 | 2,000万円以下 |
| 個人（海外重課） | 10年以下 | 3,000万円以下 |
| 法人（国内） | - | 5億円以下 |
| 法人（海外重課） | - | 10億円以下 |

#### IT企業での実務ポイント

- 退職者による顧客情報・ソースコード持ち出しが最多パターン
- 競業避止義務契約だけでは不十分、技術的アクセス制御が秘密管理性の立証に必須
- クラウド環境のアクセスログ保全が証拠確保に重要

### 1.2 不正アクセス禁止法（不正アクセス行為の禁止等に関する法律）

| 項目 | 内容 |
|------|------|
| 所管 | 総務省・警察庁 |
| 目的 | 電気通信回線を通じた不正アクセスの禁止 |

#### 禁止行為（第3条〜第6条）

| 条文 | 禁止行為 | 罰則 |
|------|----------|------|
| 第3条 | 不正アクセス行為（他人のID・パスワードの無断使用、セキュリティホール攻撃） | 3年以下の懲役又は100万円以下の罰金 |
| 第4条 | 不正アクセス行為を助長する行為（ID・パスワードの無断提供） | 1年以下の懲役又は50万円以下の罰金 |
| 第5条 | 他人の識別符号の不正取得 | 1年以下の懲役又は50万円以下の罰金 |
| 第6条 | 他人の識別符号の不正保管 | 1年以下の懲役又は50万円以下の罰金 |

#### IT企業での実務ポイント

- ペネトレーションテストは事前の書面による同意が必須（同意なしは不正アクセスに該当）
- 退職者のアカウント即時無効化が未実施の場合、会社側の管理体制も問われる
- バグバウンティプログラムの範囲外テストは不正アクセスに該当しうる
- 第8条: アクセス管理者の防御措置義務（努力義務）

### 1.3 電気通信事業法

| 項目 | 内容 |
|------|------|
| 所管 | 総務省 |
| 目的 | 電気通信の健全な発達と利用者の利益保護 |

#### 通信の秘密（第4条）

- 電気通信事業者の取扱中に係る通信の秘密は侵してはならない
- 知得（知ること）・漏洩（他者に漏らすこと）・窃用（自己のために利用すること）全て禁止
- 罰則: 2年以下の懲役又は100万円以下の罰金（第179条）
- 業務従事者の場合: 3年以下の懲役又は200万円以下の罰金（第179条第2項）

#### 外部送信規律（第27条の12、2023年6月施行）

- 利用者の端末から外部に情報を送信させる場合の規律
- 対象: Cookie、広告ID、アクセスログ等の外部送信
- 義務内容（以下のいずれか）:
  1. 通知又は公表（プライバシーポリシー等での公表が一般的）
  2. 同意取得
  3. オプトアウト機会の提供
- 対象事業者: 検索サービス、SNS、ニュース配信、動画配信、オンラインショッピングモール等

#### IT企業での実務ポイント

- SaaS事業者が電気通信事業に該当するかの判断が必要（届出の要否）
- Cookie同意バナーの実装が外部送信規律対応として必要
- CDN事業者、クラウドサービス事業者も対象となりうる

### 1.4 特定電子メール法（特定電子メールの送信の適正化等に関する法律）

| 項目 | 内容 |
|------|------|
| 所管 | 総務省 |
| 目的 | 迷惑メールの規制 |

#### オプトイン規制（第3条）

- 広告宣伝目的の電子メール送信には**事前の同意（オプトイン）**が必要
- 同意の記録を保存する義務あり
- 例外: 取引関係にある者への送信、名刺交換等で自らメールアドレスを通知した者

#### 表示義務（第4条）

- 送信者の氏名又は名称
- 受信拒否の通知先（オプトアウト手段）
- 送信者の住所
- 苦情等を受け付ける連絡先

#### 罰則

| 違反内容 | 罰則 |
|----------|------|
| オプトイン規制違反 | 1年以下の懲役又は100万円以下の罰金 |
| 法人 | 3,000万円以下の罰金 |

#### IT企業での実務ポイント

- メール配信サービス（SendGrid、Amazon SES等）利用時も送信者の責任
- トランザクションメール（注文確認等）は対象外だが、広告要素を含む場合は対象
- メールマガジン登録のダブルオプトイン推奨

### 1.5 著作権法

| 項目 | 内容 |
|------|------|
| 所管 | 文化庁 |
| 目的 | 著作物の公正な利用と著作者の権利保護 |

#### AI関連の重要条文

##### 第30条の4（著作物に表現された思想又は感情の享受を目的としない利用）

- AI学習（機械学習）のためのデータ利用は原則として著作権侵害に当たらない
- ただし「著作権者の利益を不当に害する場合」は適用除外
- 文化審議会の見解（2024年）: 特定のクリエイターの作品のみを大量に学習させる場合は「不当に害する」に該当しうる

##### AI生成物の著作物該当性

| 判断基準 | 著作物に該当 | 著作物に非該当 |
|----------|-------------|---------------|
| 人間の創作的関与 | プロンプトの詳細な指示、生成物の選別・加工、構成の指示 | 単純なプロンプト入力のみ、自動生成そのまま |
| 著作者 | 創作的関与をした人間 | AIそのもの（AIは著作者になれない） |

##### OSSライセンスと著作権

| ライセンス分類 | 代表例 | コピーレフト | 商用利用 | 注意点 |
|---------------|--------|-------------|---------|--------|
| Permissive | MIT, Apache 2.0, BSD | なし | 可 | 著作権表示・ライセンス表示の保持 |
| Weak Copyleft | LGPL, MPL | ファイル単位 | 可 | 改変ファイルのソース公開義務 |
| Strong Copyleft | GPL v2/v3, AGPL | プログラム全体 | 可（条件付） | リンクした全体のソース公開義務 |

##### GPL汚染リスク

- GPLコードをリンクしたプログラム全体がGPLの適用対象となる
- AGPLはネットワーク経由の利用にも適用（SaaS事業者は特に注意）
- GPL汚染の回避策: プロセス間通信、REST API経由の連携

#### 罰則

| 違反内容 | 罰則 |
|----------|------|
| 著作権侵害（親告罪） | 10年以下の懲役又は1,000万円以下の罰金 |
| 法人 | 3億円以下の罰金 |
| 非親告罪（2018年改正） | 海賊版の大規模頒布等に限定 |

### 1.6 下請法（下請代金支払遅延等防止法）

| 項目 | 内容 |
|------|------|
| 所管 | 公正取引委員会・中小企業庁 |
| 目的 | 下請事業者の利益保護 |

#### 適用対象（IT企業関連: 情報成果物作成委託）

| 親事業者 | 下請事業者 |
|----------|-----------|
| 資本金3億円超 | 資本金3億円以下 |
| 資本金1,000万円超〜3億円以下 | 資本金1,000万円以下 |

#### 親事業者の4つの義務

1. **書面の交付義務**（第3条）: 発注時に直ちに書面を交付
2. **書類の作成・保存義務**（第5条）: 2年間保存
3. **支払期日を定める義務**（第2条の2）: 受領日から60日以内
4. **遅延利息の支払義務**（第4条の2）: 年率14.6%

#### 禁止行為（第4条）

| 行為 | 内容 |
|------|------|
| 受領拒否 | 下請事業者に責任がないのに受領を拒否 |
| 下請代金の支払遅延 | 支払期日までに支払わない（60日ルール） |
| 下請代金の減額 | 正当な理由なく下請代金を減額 |
| 返品 | 下請事業者に責任がないのに返品 |
| 買いたたき | 通常の対価に比べ著しく低い下請代金を設定 |
| 購入・利用強制 | 正当な理由なく自己の指定する物の購入等を強制 |
| 報復措置 | 公正取引委員会等への申告を理由とする不利益取扱い |
| 有償支給原材料等の対価の早期決済 | |
| 割引困難な手形の交付 | |
| 不当な経済上の利益の提供要請 | |
| 不当な給付内容の変更・やり直し | 費用を負担せずに仕様変更・やり直しを要求 |

#### IT企業での実務ポイント

- フリーランスエンジニアへの開発委託は下請法の対象となりうる
- SES契約であっても実態が請負の場合は下請法適用
- 2024年11月施行フリーランス保護法との関係整理が必要

### 1.7 公益通報者保護法

| 項目 | 内容 |
|------|------|
| 所管 | 消費者庁 |
| 目的 | 公益通報者の保護 |

#### 2022年改正（施行済み）の主要ポイント

1. **内部通報体制の整備義務化**: 従業員数301人以上の事業者に義務化（300人以下は努力義務）
2. **通報対象の拡大**: 行政罰（過料）の対象行為も追加
3. **保護対象の拡大**: 退職後1年以内の退職者、役員も保護対象
4. **従事者の守秘義務**: 内部通報担当者に刑事罰付きの守秘義務（30万円以下の罰金）

#### 2026年改正（施行予定）の主要ポイント

1. **通報者範囲のさらなる拡大**: 退職後3年以内の退職者、取引先従業員も保護対象
2. **罰則の強化**: 通報を理由とする不利益取扱いに対する行政措置命令違反に罰則追加
3. **是正措置の実効性強化**: 行政機関による調査権限の強化
4. **デジタル通報の整備**: オンライン通報窓口の設置義務化

#### IT企業での実務ポイント

- 匿名通報を受け付ける仕組みの構築が推奨（専用フォーム、外部窓口）
- 通報者の特定につながるログアクセスの制限
- エンジニアがセキュリティ上の問題を通報するケースへの対応体制

### 1.8 労働基準法

| 項目 | 内容 |
|------|------|
| 所管 | 厚生労働省 |
| 目的 | 労働条件の最低基準の確保 |

#### テレワーク関連

- **労働時間管理**: テレワークでも労働時間の把握義務あり（第32条・第108条）
- **事業場外みなし労働時間制**: テレワークへの適用は厳格化（2024年ガイドライン改訂）
  - 適用要件: 使用者の具体的な指揮監督が及ばず、労働時間の算定が困難であること
  - PC操作ログ等で労働時間が把握できる場合は適用不可
- **安全衛生**: 自宅作業環境の整備（照明、机・椅子等の基準）

#### AI利用時の監視制限

- **モニタリングの制限**: 従業員のPC操作、キーストローク、画面キャプチャ等のAI分析は、目的・手段・労働者への周知が必要
- **個人情報保護委員会ガイダンス**: AI分析によるプロファイリングについて利用目的の特定が必要
- **プライバシー権との関係**: 業務上の必要性と従業員のプライバシーの均衡（比例原則）
- **労使協議の推奨**: AIによるモニタリング導入時は労使での協議が望ましい

#### IT企業での実務ポイント

- 裁量労働制（専門業務型）: SE、プログラマーは対象外（研究開発職は対象）
- 高度プロフェッショナル制度: 年収1,075万円以上が要件
- 副業・兼業: 労働時間の通算管理が必要（管理モデルの活用推奨）

## 2. AI利用に関する法的リスク

### 2.1 AI事業者ガイドライン（第1.1版・2024年4月策定）

#### 3主体別の要件

| 主体 | 主要要件 |
|------|----------|
| **AI開発者** | 安全性の確保、公平性の確保、透明性の確保、プライバシー保護、セキュリティの確保、学習データの適正性確認 |
| **AI提供者** | 適切な利用条件の設定、利用者への情報提供、インシデント対応体制、モニタリング、AI生成物である旨の表示検討 |
| **AI利用者** | 利用目的の正当性確認、出力の確認（人間による監督）、利用条件の遵守、個人情報の適正な取扱い |

#### 共通原則

1. 人間中心（Human-centric）
2. 安全性（Safety）
3. 公平性（Fairness）
4. プライバシー保護（Privacy）
5. セキュリティ（Security）
6. 透明性（Transparency）
7. アカウンタビリティ（Accountability）
8. 教育・リテラシー（Education/Literacy）
9. 公正競争確保（Fair Competition）
10. イノベーション（Innovation）

### 2.2 AI生成コードの著作権問題

#### 著作物該当性の判断基準

```
[AI生成コードの著作権判断フロー]

1. 人間がプロンプトで詳細な仕様・構造・アルゴリズムを指示したか？
   ├── YES → 2へ
   └── NO → 著作物に該当しない可能性が高い（著作権なし）

2. 生成されたコードに創作性があるか？（ありふれた表現でないか）
   ├── YES → 3へ
   └── NO → 著作物に該当しない（アイデアの域を出ない）

3. 人間が生成物を選別・加工・修正したか？
   ├── YES → 人間を著作者とする著作物に該当する可能性が高い
   └── NO → 著作物該当性は個別判断（人間の創作的関与の程度による）
```

#### 実務上の対応

- AI生成コードには生成経緯を記録（プロンプト、使用AIツール、人間の修正履歴）
- 重要なコードベースにおいてはAI生成部分と人間作成部分を区別管理
- 第三者の著作物との類似性チェック（AIが学習データを再現するリスク）

### 2.3 OSSライセンスコンプライアンス

#### SBOM（Software Bill of Materials）管理

| 項目 | 内容 |
|------|------|
| 定義 | ソフトウェアを構成するコンポーネントの一覧表 |
| 主要フォーマット | SPDX（ISO/IEC 5962:2021）、CycloneDX |
| 管理ツール | Snyk、FOSSA、Black Duck、Trivy |
| 更新頻度 | リリースごと、最低でも四半期ごと |

#### OpenChain（ISO/IEC 5230:2020）

- OSSコンプライアンスプログラムの国際標準
- 要件: OSSポリシーの策定、責任者の任命、教育、コンプライアンスプロセスの確立
- 日本では富士通、日立、NEC等が認証取得

#### GPL汚染の防止策

1. **技術的分離**: GPLコンポーネントをマイクロサービスとして独立させ、REST API経由で連携
2. **SBOM自動生成**: CI/CDパイプラインにSBOMスキャンを組み込む
3. **ライセンス自動チェック**: dependabot、licensee等のツールでPR時に自動検査
4. **社内OSSリポジトリ**: 承認済みOSSのみ利用可能な内部レジストリの運用

### 2.4 EU AI Act の域外適用

#### 日本企業への影響

| 条件 | 適用対象 |
|------|----------|
| EU域内の利用者にAIシステムを提供 | 日本企業であっても適用 |
| EU域内で生成された出力を利用 | 適用される場合あり |
| EU域内に拠点がなくても | 代理人の選任義務あり |

#### リスク分類と規制内容

| リスクレベル | 規制内容 | 例 |
|-------------|----------|-----|
| 禁止（Unacceptable） | 使用禁止 | ソーシャルスコアリング、リアルタイム公共空間での遠隔生体認証 |
| 高リスク（High） | 適合性評価、登録、モニタリング義務 | 採用AI、信用スコアリング、重要インフラ制御 |
| 限定リスク（Limited） | 透明性義務（AI利用の表示等） | チャットボット、ディープフェイク |
| 最小リスク（Minimal） | 規制なし（行動規範の策定推奨） | スパムフィルター、ゲームAI |

#### 罰則

- 禁止AI使用: 最大3,500万ユーロ又は全世界売上高の7%のいずれか高い方
- 高リスクAI義務違反: 最大1,500万ユーロ又は全世界売上高の3%
- 不正確な情報提供: 最大750万ユーロ又は全世界売上高の1%

#### 施行スケジュール

- 2024年8月: 発効
- 2025年2月: 禁止AIの規定適用
- 2025年8月: 汎用AI（GPAI）の規定適用
- 2026年8月: 高リスクAIの規定の完全適用

### 2.5 経産省「AIの利用・開発に関する契約チェックリスト」要点

#### 契約時の確認事項

1. **データに関する事項**
   - 学習データの権利処理状況
   - 入力データの取扱い（AI事業者側での利用可否）
   - 出力データの権利帰属
   - データの保持期間・削除義務

2. **AI生成物に関する事項**
   - 権利帰属（発注者/受注者/共有）
   - 利用範囲の制限
   - 第三者の権利侵害時の責任分担
   - 精度・品質の保証範囲

3. **責任に関する事項**
   - AI出力の誤りによる損害の責任分担
   - セキュリティインシデント時の対応
   - 法令改正時の対応義務
   - 損害賠償の上限設定

4. **技術的事項**
   - モデルのバージョン管理
   - API仕様変更時の対応
   - SLA（稼働率、応答時間）
   - モデルの継続的改善義務

## 3. 契約管理

### 3.1 NDA（秘密保持契約）の必須条項

| 条項 | 内容 | AI利用時の注意点 |
|------|------|-----------------|
| 秘密情報の定義 | 開示情報の範囲を明確化 | AI学習データとして利用されるリスクを明記 |
| 秘密保持義務 | 目的外利用・第三者開示の禁止 | AIサービスへの入力が「第三者への開示」に該当するか明確化 |
| 例外事由 | 公知情報、独自開発情報等 | AI生成物が独自開発に該当するかの整理 |
| 有効期間 | 通常2〜5年 | AI学習済みモデルからの情報除去困難性を考慮 |
| 返還・廃棄 | 契約終了時の情報返還・廃棄義務 | AIモデルに学習された情報の「廃棄」の定義 |
| 損害賠償 | 違反時の賠償責任 | AI起因の情報漏洩時の責任範囲 |

#### AI利用時の追加条項（推奨）

- 秘密情報をAIサービスに入力することの可否
- AIサービスの学習データとしての利用可否
- AI APIを通じた秘密情報の処理に関する制限
- AI利用に起因する秘密情報漏洩時の責任分担

### 3.2 業務委託契約（AI API利用条項）

#### データ利用範囲の明確化

```
[条項例]
第X条（データの取扱い）
1. 委託者が提供するデータは、本業務の遂行の目的のみに利用するものとする。
2. 受託者は、委託者の事前の書面による同意なく、委託者データを
   AIモデルの学習・訓練に使用してはならない。
3. 受託者が利用するAI APIサービスにおけるデータ取扱い
   （入力データの保持期間、学習への利用有無等）について、
   委託者に書面で情報提供するものとする。
```

#### 権利帰属の整理

| パターン | 内容 | メリット | デメリット |
|----------|------|----------|-----------|
| 発注者帰属 | 成果物の権利は全て発注者に帰属 | 発注者の利用自由度が高い | 受注者の再利用不可、単価が高くなりがち |
| 受注者帰属 | 成果物の権利は受注者に帰属、発注者にライセンス | 受注者のノウハウ蓄積 | 発注者の利用制限 |
| 共有 | 権利を共有 | バランスが取れる | 権利行使に相手方の同意が必要 |

#### 品質保証

- AI出力の精度保証レベル（ベストエフォート/SLA付き）
- AI出力に基づく意思決定の責任範囲
- 人間によるレビュー・承認プロセスの要否

### 3.3 SLA（Service Level Agreement）の主要指標

| 指標 | 定義 | 一般的な水準 |
|------|------|-------------|
| **稼働率（Availability）** | 計画停止を除くサービス利用可能時間の割合 | 99.9%（月間停止43.8分）〜99.99%（月間停止4.38分） |
| **応答時間（Response Time）** | リクエストから応答までの時間 | API: 200ms〜2s、Web: 3s以内 |
| **RTO（Recovery Time Objective）** | 障害発生から復旧までの目標時間 | Tier1: 1時間、Tier2: 4時間、Tier3: 24時間 |
| **RPO（Recovery Point Objective）** | データ損失を許容する最大時間 | Tier1: 0（リアルタイム同期）、Tier2: 1時間、Tier3: 24時間 |
| **MTBF（Mean Time Between Failures）** | 障害間の平均時間 | サービス品質の指標 |
| **MTTR（Mean Time To Repair）** | 障害発生から復旧までの平均時間 | SLAの実績評価に使用 |

#### ペナルティ（SLAクレジット）

| 稼働率 | クレジット率（一般的） |
|--------|----------------------|
| 99.9%未満〜99.0% | 月額料金の10% |
| 99.0%未満〜95.0% | 月額料金の25% |
| 95.0%未満 | 月額料金の50%又は解約権 |

### 3.4 利用規約の法的要件

#### 消費者契約法との関係

| 無効となる条項 | 内容 |
|---------------|------|
| 事業者の損害賠償責任の全部免除 | 第8条第1項第1号 |
| 事業者の故意・重過失による損害賠償の一部免除 | 第8条第1項第2号 |
| 消費者の解除権を放棄させる条項 | 第8条の2 |
| 平均的損害額を超えるキャンセル料 | 第9条第1号 |
| 年14.6%を超える遅延損害金 | 第9条第2号 |

#### 電子消費者契約法（電子消費者契約に関する民法の特例に関する法律）

- 操作ミスによる意思表示の無効（確認画面なしの場合、消費者の重過失があっても無効主張可能）
- 対策: 申込確認画面の設置が事実上必須

#### 特定商取引法（通信販売）

- 広告の表示義務（第11条）: 販売価格、送料、支払方法、返品条件等
- 誇大広告の禁止（第12条）
- 電子メール広告の規制（第12条の3〜5）: オプトイン規制

## 4. 知的財産管理

### 4.1 職務発明規程（特許法第35条）

#### 職務発明の要件

1. 従業者等がした発明であること
2. その性質上、使用者等の業務範囲に属すること
3. 発明をするに至った行為が従業者等の現在又は過去の職務に属すること

#### 権利帰属のパターン（2015年改正後）

| パターン | 内容 | 要件 |
|----------|------|------|
| 原始的使用者帰属 | 発明時点から使用者に特許を受ける権利が帰属 | 契約・勤務規則等での事前定め |
| 原始的発明者帰属（デフォルト） | 発明者に特許を受ける権利が帰属 | 事前定めがない場合 |

#### 相当の利益

- 職務発明について使用者が特許を受ける権利を取得した場合、従業者は「相当の利益」を受ける権利あり
- 相当の利益の内容: 金銭、昇進、留学機会、ストックオプション等
- 算定の合理性: 使用者と従業者の協議状況、基準の開示状況、従業者の意見聴取状況を考慮

#### IT企業での実務ポイント

- AIが支援した発明の発明者認定（人間の創作的関与が必要）
- OSSとして公開する場合の職務発明規程との整合性
- 競合他社からの転職者が保有する発明情報の取扱い

### 4.2 OSS利用ポリシー

#### ライセンス分類と対応

| カテゴリ | ライセンス | 社内利用 | SaaS提供 | 製品組込み | 必要な対応 |
|----------|-----------|---------|----------|-----------|-----------|
| A（自由） | MIT, BSD-2, BSD-3 | ○ | ○ | ○ | 著作権表示・ライセンス文の同梱 |
| B（注意） | Apache 2.0 | ○ | ○ | ○ | 著作権表示・ライセンス文・NOTICE同梱、特許ライセンス条項注意 |
| C（要確認） | LGPL | ○ | ○ | △ | 動的リンクは可、静的リンクはソース公開義務 |
| D（制限） | GPL v2/v3 | ○ | ○ | × | リンクした全体のソース公開義務 |
| E（高リスク） | AGPL | ○ | × | × | ネットワーク経由の利用でもソース公開義務 |
| F（禁止） | SSPL | × | × | × | 利用禁止（事実上のプロプライエタリ） |

#### SBOM管理プロセス

1. **依存関係のスキャン**: ビルド時に全依存ライブラリを自動検出
2. **ライセンス判定**: 各ライブラリのライセンスを自動分類
3. **承認フロー**: カテゴリC以上は法務レビュー必須
4. **SBOM生成**: SPDX又はCycloneDXフォーマットで生成
5. **継続的モニタリング**: 新規脆弱性・ライセンス変更の定期チェック

### 4.3 AI生成物の権利帰属

#### 人間の創作的関与の判断基準

| レベル | 関与の程度 | 権利帰属 | 例 |
|--------|-----------|---------|-----|
| Level 0 | 関与なし | 著作物に非該当（パブリックドメイン的扱い） | AIが自律的に生成 |
| Level 1 | 最小限の関与 | 著作物該当性は低い | 「猫の絵を描いて」程度のプロンプト |
| Level 2 | 中程度の関与 | 個別判断 | 詳細なプロンプト指示、複数回の修正指示 |
| Level 3 | 高度な関与 | 著作物に該当する可能性が高い | 具体的な構成・表現の指示、生成物の選別・加工・編集 |
| Level 4 | AIはツール的利用 | 人間の著作物 | AIの提案を参考にしつつ人間が主体的に創作 |

- [ ] **個人情報管理**
  - [ ] 個人情報台帳の更新
  - [ ] アクセス権限の棚卸し
  - [ ] 委託先の監督状況
  - [ ] 安全管理措置の実施状況

- [ ] **知的財産管理**
  - [ ] 特許出願・維持の状況確認
  - [ ] 商標権の更新管理
  - [ ] OSSライセンスの遵守状況（SBOM更新）
  - [ ] 職務発明の報告・処理状況

- [ ] **労務コンプライアンス**
  - [ ] 労働時間管理の適正性
  - [ ] ハラスメント防止措置の実施状況
  - [ ] 健康診断・ストレスチェックの実施状況
  - [ ] 副業・兼業の管理状況

- [ ] **情報セキュリティ**
  - [ ] セキュリティインシデントの発生状況
  - [ ] 脆弱性対応の状況
  - [ ] セキュリティ研修の実施状況
  - [ ] BCP/DRの見直し状況

- [ ] **内部通報制度**
  - [ ] 通報件数・処理状況の確認
  - [ ] 通報者保護の実施状況
  - [ ] 制度の従業員への周知状況
  - [ ] 外部窓口の運用状況

> **注意**: 本リファレンスは2026年3月時点の法令・ガイドラインに基づく。法令改正や新規ガイドラインの公表により内容が変更される可能性があるため、重要な法的判断に際しては最新の法令・判例を確認すること。
