# comptroller クロスチェックプロトコル

> comptroller（監査役エージェント）が compliance（主査エージェント）の成果物を独立的にクロスチェックするためのプロトコル。監査法人の視点を取り入れ、規程と運用の乖離、証跡の有効性、チェックリスト実行結果の妥当性を検証する。

## 1. クロスチェックの原則

### 1.1 独立性の確保

- comptrollerはcomplianceの成果物を「そのまま信用しない」。独自の検証を必ず行う
- complianceが提出した証跡は、comptrollerが独立して存在確認・内容確認を行う
- complianceの判断（合格/不合格）に対し、comptrollerは独自の判断基準で再評価する
- complianceの実行プロセスに関与しない。結果のみを検証対象とする

### 1.2 懐疑的姿勢（Professional Skepticism）

- 「問題なし」の報告に対して、本当に問題がないか裏取りを行う
- 「実施済み」の報告に対して、実施の痕跡（証跡）を独自に確認する
- サンプリングによる確認ではなく、可能な限り全件確認を行う
- 「規程が存在する」ことと「規程通りに運用されている」ことは別問題として扱う
- 形式的な証跡の存在と、実質的な統制の有効性を区別する

### 1.3 事実ベース検証（Evidence-Based Verification）

- 口頭報告や自己申告は証跡として認めない
- 証跡は原本（またはシステムから直接取得したデータ）で確認する
- タイムスタンプの整合性を重視する（事後作成の証跡を検出する）
- 複数の独立した証跡の突合（Corroboration）により信頼性を高める
- 証跡間の矛盾（日時の不整合、担当者の不一致等）を見逃さない

### 1.4 重要性の原則（Materiality）

- 全ての不備を同等に扱わない。財務報告に与える影響の大きさで優先度を判断する
- 重要な虚偽表示リスクに直結する統制の不備を最優先で指摘する
- 「重要な不備」と「軽微な不備」を明確に区分する

## 2. 監査法人視点でのチェックポイント

> Big4監査法人（EY新日本、PwCあらた、KPMG あずさ、Deloitteトーマツ）が上場審査・J-SOX監査で重視するポイントを反映。

### 2.1 EY新日本 重視ポイント

| チェック観点 | 内容 | comptrollerの検証方法 |
|-------------|------|---------------------|
| ITGCの設計有効性 | 4領域（開発変更・アクセス・運用・委託先）の統制設計が適切か | RCM（リスクコントロールマトリクス）の網羅性確認、統制目標とリスクの対応関係確認 |
| ITGCの運用有効性 | 設計された統制が期間を通じて一貫して運用されているか | 12ヶ月分の証跡のサンプリングテスト（最低25件）、時系列での運用一貫性確認 |
| ITACの正確性 | 自動処理（自動仕訳・自動計算）が正確に機能しているか | テストデータによる再実行テスト、手計算との突合 |
| 職務分掌の実効性 | 開発と運用の分離、承認者と実施者の分離が機能しているか | 実際のデプロイログ vs 承認記録の突合、SoD（Segregation of Duties）マトリクス検証 |

### 2.2 PwCあらた 重視ポイント

| チェック観点 | 内容 | comptrollerの検証方法 |
|-------------|------|---------------------|
| クラウド環境の統制 | IaaS/PaaS/SaaSそれぞれの責任共有モデルに基づく統制 | クラウドベンダーのSOCレポートとCUECsの対応状況確認 |
| DevOps/CI-CD統制 | 自動化されたデプロイパイプラインにおける統制の組込み | CI/CDパイプライン設定の確認、承認ゲートの実装検証 |
| データの完全性 | システム間IF（インターフェース）のデータ整合性 | IF連携ログの件数突合、エラーハンドリングの設計確認 |
| 第三者認証の活用 | ISMS・SOC2等の認証の有効性と範囲の適切性 | 認証範囲が対象システムをカバーしているか確認 |

### 2.3 KPMGあずさ 重視ポイント

| チェック観点 | 内容 | comptrollerの検証方法 |
|-------------|------|---------------------|
| IT環境の理解 | 財務報告に係るITシステムの全体像把握 | システム構成図・データフロー図の正確性検証 |
| 重要なアプリケーションの特定 | 財務報告に直接関係するシステムの範囲決定の適切性 | 業務プロセスとシステムの対応関係確認、スコーピングの妥当性検証 |
| キーコントロールの特定 | 重要な統制が適切に特定されているか | リスクの大きさとコントロールの対応関係、代替統制の有無確認 |
| 不備の評価 | 発見された不備が適切に評価・分類されているか | 不備の影響度分析の妥当性確認、「重要な不備」の見落としがないか |

### 2.4 Deloitteトーマツ 重視ポイント

| チェック観点 | 内容 | comptrollerの検証方法 |
|-------------|------|---------------------|
| セキュリティガバナンス | 経営層のサイバーセキュリティへの関与度 | 取締役会議事録におけるセキュリティ議題の確認、CISO報告の定期性 |
| インシデント対応体制 | セキュリティインシデント発生時の対応能力 | インシデント対応訓練の実施記録、Playbook整備状況 |
| AI・新技術のリスク | AI利用に伴う新たなリスクへの対応 | AI利用ガバナンスの整備状況、シャドーAI対策の実効性 |
| 規程の実効性 | 規程が形骸化していないか | 規程と実運用のウォークスルーテスト、従業員へのヒアリング結果確認 |

### 2.5 共通重点チェック事項

以下の項目は全監査法人が共通して重視するため、comptrollerは必ず検証すること:

1. **退職者アカウント管理**: 退職日 vs アカウント無効化日の突合（1日以上の遅延は指摘対象）
2. **本番デプロイの承認**: 承認記録なしのデプロイが存在しないか全件確認
3. **特権IDの利用**: 特権ID操作の全件ログ確認、業務上の必要性との突合
4. **バックアップの有効性**: リストアテスト結果の確認（テスト未実施は重要な不備）
5. **規程の版管理**: 規程の最終改訂日が1年以上前でないか
6. **証跡の連続性**: 12ヶ月間で証跡が欠落している月がないか

## 3. complianceの成果物に対する検証手順

### 3.1 チェックリスト実行結果の妥当性検証

#### Step 1: 網羅性の確認

```
検証項目:
- [ ] ipo-audit-checklist.md の全セクションがカバーされているか
- [ ] 各セクションのチェック項目が一つも漏れなく評価されているか
- [ ] 「対象外」と判断された項目に合理的な理由が付されているか
- [ ] 新たに追加すべきチェック項目がないか（直近の法改正、ガイドライン更新）
```

#### Step 2: 判断の妥当性検証

```
検証項目:
- [ ] 「合格」判断された項目について、判断根拠が十分か
- [ ] 「合格」判断が甘すぎないか（形式的な存在確認だけで合格としていないか）
- [ ] 「不合格」判断された項目について、不備の内容が具体的に記載されているか
- [ ] リスクレベル（HIGH/MEDIUM/LOW）の評価が適切か
- [ ] 改善勧告の内容が具体的かつ実行可能か
```

#### Step 3: サンプリング再検証

```
手順:
1. complianceが「合格」とした項目から、HIGH リスク項目を全件抽出
2. 上記のうち最低30%（最低5件）を無作為に選択
3. 選択した項目について、comptrollerが独自に証跡を確認
4. complianceの判断と一致するか検証
5. 不一致がある場合は全件再検証に切り替え
```

### 3.2 証跡の存在確認と有効性評価

#### 証跡の存在確認

```
検証項目:
- [ ] complianceが参照した証跡が実際に存在するか
- [ ] 証跡のファイルパス/URLが正確か
- [ ] 証跡が最新のものか（古いバージョンを参照していないか）
- [ ] 証跡へのアクセスが可能か（リンク切れ、権限不足がないか）
```

#### 証跡の有効性評価

```
検証項目:
- [ ] 証跡に4W1H（Who/What/When/Where/How）が含まれているか
- [ ] タイムスタンプが改ざんされた形跡はないか
- [ ] 証跡の作成日時が合理的か（チェック実施日より前に作成されていないか — 事後作成の疑い）
- [ ] 複数の証跡間で日時・担当者に矛盾がないか
- [ ] 証跡のフォーマットが社内基準に準拠しているか
- [ ] 証跡が対象期間（通常12ヶ月）をカバーしているか
- [ ] 証跡に承認者の署名/承認記録が含まれているか（必要な場合）
```

#### 証跡の突合テスト

```
手順:
1. 退職者リスト（HR部門提供）vs アカウント無効化記録（IT部門提供）の突合
2. 変更要求チケット vs コードレビュー記録 vs デプロイログの突合
3. アクセス権限棚卸し結果 vs 実際のシステム設定の突合
4. バックアップ実行ログ vs バックアップポリシー（頻度・世代数）の突合
5. 委託先評価実施日 vs 評価サイクル（年次等）の突合
```

### 3.3 規程と運用の一致確認方法

#### ウォークスルーテスト

```
手順:
1. 対象プロセスの規程（手順書）を入手
2. 規程に記載された手順を1件のトランザクションで追跡
3. 各ステップで規程通りの操作が行われているか確認
4. 規程に記載されていない追加ステップがないか確認
5. 規程に記載されているが実施されていないステップがないか確認

対象プロセス（最低限）:
- アカウント登録/削除プロセス
- 変更要求→承認→リリースプロセス
- インシデント対応プロセス
- バックアップ/リストアプロセス
- 委託先評価プロセス
```

#### 例外事項の検出

```
検証項目:
- [ ] 規程に「例外」として認められている運用がないか（例外が常態化していないか）
- [ ] 緊急変更の頻度が全体の10%を超えていないか（超えている場合は通常プロセスに問題あり）
- [ ] 規程の適用除外申請がないか（ある場合は妥当性を確認）
- [ ] 規程改訂日と運用変更日に乖離がないか
```

## 4. 差し戻し基準

### 4.1 即時差し戻し（必ず差し戻す）

以下のいずれかに該当する場合、comptrollerは成果物をcomplianceに即時差し戻す:

| 差し戻し条件 | 理由 | 差し戻し時の指示 |
|-------------|------|-----------------|
| HIGHリスク項目が未評価（チェック漏れ） | 重要な統制の見落としは監査上許容できない | 該当項目の評価実施と証跡提出を要求 |
| 証跡が存在しない項目を「合格」としている | 証跡なしの合格判断は根拠不十分 | 証跡の取得・提出、または「不合格」への判断変更を要求 |
| 証跡間のタイムスタンプに矛盾がある | 証跡の信頼性に疑義がある | 矛盾の原因究明と正確な証跡の再提出を要求 |
| 規程と運用の重大な乖離が見過ごされている | 統制の実効性が確認できない | 乖離の詳細な調査と改善勧告の追加を要求 |
| 退職者アカウントの残存を見逃している | アクセス管理の根幹に関わる不備 | 全退職者の突合チェック再実施を要求 |
| 本番デプロイの承認記録欠如を見逃している | 変更管理の根幹に関わる不備 | 承認記録なしデプロイの全件調査を要求 |
| サンプリング再検証でcomplianceの判断と不一致が20%以上 | 判断の信頼性が低い | 全HIGH項目の再評価を要求 |
| J-SOXの4領域のいずれかが全くカバーされていない | 評価範囲の重大な欠陥 | 欠落領域の評価実施を要求 |

### 4.2 条件付き差し戻し（改善指示付き）

以下に該当する場合、改善指示を付けてcomplianceに差し戻す:

| 差し戻し条件 | 改善指示 |
|-------------|----------|
| MEDIUMリスク項目の評価が不十分 | 該当項目の詳細評価と根拠の補強を要求 |
| 証跡の4W1Hが不完全 | 不足要素の特定と補完を要求 |
| 改善勧告が抽象的で実行可能性が低い | 具体的なアクション・期限・担当者を明記した勧告への改訂を要求 |
| 証跡の対象期間にギャップがある（1-2ヶ月の欠落） | ギャップ期間の証跡取得、または取得不能理由の説明を要求 |
| リスクレベルの評価が不適切（過小評価） | リスク評価の再実施と根拠の明記を要求 |

### 4.3 差し戻し時の手順

```
1. 差し戻し理由を明確に記載（該当箇所の特定、問題点の具体的記述）
2. 期待する改善内容を具体的に記載
3. 再提出期限を設定（即時差し戻し: 3営業日以内、条件付き差し戻し: 5営業日以内）
4. 差し戻し回数をカウント（同一項目の3回目の差し戻しはエスカレーション）
5. 差し戻し記録をクロスチェックレポートに記載
```

## 5. 承認基準

### 5.1 承認の前提条件（全て満たすこと）

```
必須条件:
- [ ] ipo-audit-checklist.md の全セクションがカバーされている
- [ ] HIGHリスク項目が全て評価済みである
- [ ] 全ての「合格」判断に有効な証跡が紐付いている
- [ ] サンプリング再検証での不一致率が10%未満である
- [ ] 差し戻し事項が全て解消されている
- [ ] 規程と運用のウォークスルーテストが最低3プロセスで実施されている
- [ ] 証跡間の重大な矛盾がない
```

### 5.2 承認レベル

#### 完全承認（Full Approval）

以下の全条件を満たす場合:

- 承認の前提条件を全て充足
- HIGHリスク項目の合格率が90%以上
- MEDIUMリスク項目の合格率が80%以上
- 不合格項目に対する改善計画が提示されている
- サンプリング再検証での不一致率が5%未満

#### 条件付き承認（Conditional Approval）

以下の条件に該当する場合:

- 承認の前提条件を全て充足
- HIGHリスク項目の合格率が70%以上90%未満
- 不合格項目に対する改善計画と期限が提示されている
- 条件付き承認の有効期限: 30日間（期限内に改善完了確認が必要）

#### 不承認（Rejection）

以下のいずれかに該当する場合:

- 承認の前提条件を1つ以上満たさない
- HIGHリスク項目の合格率が70%未満
- 差し戻し事項が未解消
- 証跡の信頼性に重大な疑義がある

### 5.3 承認時の記載事項

```
承認記録に含める事項:
1. 承認レベル（完全承認/条件付き承認）
2. 承認日時
3. 検証範囲の概要
4. 主要な検証結果のサマリ
5. 条件付き承認の場合: 条件の内容と期限
6. 残存リスクの認識
7. 次回検証の推奨時期
```

## 付録: クロスチェック実施ガイド

### 検証の優先順位

クロスチェックは以下の優先順位で実施する:

| 優先度 | 検証対象 | 理由 |
|--------|---------|------|
| 1 | アクセス管理（退職者ID、特権ID） | 監査法人の最重要確認事項。不備時の影響が最大 |
| 2 | 変更管理（承認記録、職務分掌） | 本番環境への不正変更リスクに直結 |
| 3 | 証跡の存在・有効性 | 証跡がなければ統制の有効性を証明できない |
| 4 | 規程と運用の一致 | 形骸化した規程は統制として無効 |
| 5 | 委託先管理・クラウド管理 | 外部依存リスクの評価 |
| 6 | AI利用ガバナンス | 新興リスク領域。整備途上でも方向性の確認が重要 |

### サンプリング基準

| 母集団サイズ | 最低サンプル数 | 備考 |
|-------------|---------------|------|
| 1-10件 | 全件 | 全件テスト |
| 11-50件 | 10件 | 無作為抽出 |
| 51-250件 | 25件 | 無作為抽出 |
| 251件以上 | 30件 | 無作為抽出+リスクベース追加 |

※ 不一致を1件でも検出した場合、サンプルサイズを2倍に拡大して再検証する。
