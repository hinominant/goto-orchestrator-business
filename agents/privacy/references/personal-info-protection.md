# 個人情報保護法 包括リファレンス

> 本ドキュメントは privacy エージェントが個人情報保護法に基づく評価・監査を行う際の基盤リファレンスである。
> 令和4年4月施行（令和3年改正法）準拠。条文番号は改正後の番号体系に基づく。

---

## 1. 主要条文一覧と実務要件

### 第17条（利用目的の特定）

**条文要旨**: 個人情報取扱事業者は、個人情報を取り扱うに当たっては、その利用の目的をできる限り特定しなければならない。

**具体的要件**:
- 利用目的は「本人が合理的に予測できる程度」に具体的に特定する
- 「事業活動に用いるため」「マーケティング活動に用いるため」等の抽象的記載は不十分
- 利用目的の変更は「変更前の利用目的と関連性を有すると合理的に認められる範囲」に限定

**実務上の注意点**:
- プライバシーポリシーに利用目的を明記する際は、サービスごと・データ種別ごとに分けて記載する
- AI/機械学習への利用を想定する場合、その旨を利用目的に含める必要がある
- 利用目的の変更時は本人への通知または公表が必要（第21条第3項）

### 第18条（利用目的による制限）

**条文要旨**: あらかじめ本人の同意を得ないで、特定された利用目的の達成に必要な範囲を超えて個人情報を取り扱ってはならない。

**具体的要件**:
- 目的外利用には原則として本人の同意が必要
- 例外: 法令に基づく場合、人の生命・身体・財産の保護に必要な場合、公衆衛生・児童の健全育成に必要な場合、国の機関等への協力

**実務上の注意点**:
- 同意取得は「本人の意思に基づく」ものであり、デフォルトオプトインは不適切
- 合併・事業承継時は変更前の利用目的の範囲内で引き続き利用可能
- 同意の記録を保持し、撤回手続きも整備すること

### 第20条（適正な取得）

**条文要旨**: 偽りその他不正の手段により個人情報を取得してはならない。要配慮個人情報はあらかじめ本人の同意を得て取得する。

**具体的要件**:
- 不正な手段: 他の事業者が不正に取得した個人情報と知りながらの取得、本人を欺く行為による取得
- 要配慮個人情報の取得には原則として事前同意が必要（オプトアウトによる取得不可）
- 第三者から提供を受ける場合、提供元の取得経緯を確認する義務

**実務上の注意点**:
- Webスクレイピングで個人情報を収集する場合、利用規約違反や不正アクセスに注意
- Cookie等で取得する情報が個人関連情報に該当する場合、第31条の規律に注意
- AI学習データの収集元の適法性を確認すること

### 第23条（安全管理措置）

**条文要旨**: 個人データの漏えい、滅失又は毀損の防止その他の安全管理のために必要かつ適切な措置を講じなければならない。

→ 詳細は「2. 安全管理措置の4分類」セクションで解説

### 第24条（従業者の監督）

**条文要旨**: 従業者に個人データを取り扱わせるに当たっては、安全管理措置が適切に実施されるよう必要かつ適切な監督を行わなければならない。

**具体的要件**:
- 「従業者」には正社員のほか、パート、アルバイト、派遣社員、取締役、監査役も含む
- 雇用契約時に秘密保持に関する事項を含む契約を締結
- 定期的な教育・研修の実施
- 内部規程の周知徹底

**実務上の注意点**:
- 退職後の秘密保持義務も契約で明確化すること
- リモートワーク環境での従業者監督体制も整備が必要
- 監督の記録（研修実施記録、誓約書等）を保持すること

### 第25条（委託先の監督）

**条文要旨**: 個人データの取扱いを委託する場合、委託先に対して必要かつ適切な監督を行わなければならない。

**具体的要件**:
- 適切な委託先の選定（個人情報保護体制の確認）
- 委託契約における安全管理措置の明記
- 委託先における個人データ取扱状況の把握
- 再委託の制限または管理

**実務上の注意点**:
- クラウドサービス（AWS, GCP, Azure等）利用時、当該サービスが「委託」に該当するか検討が必要
- SaaS利用時の利用規約で個人データの取扱いについて確認すること
- 外部AI APIへの個人データ送信は委託または第三者提供に該当しうる
- 委託先の定期監査を実施すること

### 第26条（漏えい等の報告）

**条文要旨**: 個人データの漏えい等が発生した場合、個人情報保護委員会への報告及び本人への通知を行わなければならない。

→ 詳細は「4. 漏えい時の対応フロー」セクションで解説

### 第27条（第三者提供の制限）

**条文要旨**: あらかじめ本人の同意を得ないで、個人データを第三者に提供してはならない。

**具体的要件**:
- 原則: 本人の事前同意が必要
- 例外: 法令に基づく場合、人の生命・身体・財産保護、公衆衛生・児童育成、国の機関等への協力
- オプトアウト手続き: 本人の求めに応じて提供を停止する旨等を通知し、個人情報保護委員会に届出
- 委託・事業承継・共同利用は「第三者」に該当しない

**実務上の注意点**:
- 第三者提供には提供側・受領側ともに記録義務あり（第29条・第30条）
- オプトアウトでは要配慮個人情報・不正取得データ・他のオプトアウトから得たデータの提供は不可
- API連携で他社にデータを渡す場合、第三者提供に該当するか慎重に検討すること
- グループ会社間でも原則として「第三者」に該当する（共同利用の要件を満たす場合を除く）

### 第28条（外国にある第三者への提供 / 越境移転）

**条文要旨**: 外国にある第三者に個人データを提供する場合、本人の同意を得る際に当該外国の個人情報保護制度等の情報を提供しなければならない。

**具体的要件**:
- 同意取得時に提供すべき情報:
  - 当該外国の名称
  - 当該外国の個人情報保護制度の概要
  - 当該第三者が講じる個人情報保護のための措置の概要
- 基準適合体制（APEC CBPRシステム認証等）の整備による代替
- 個人情報保護委員会が認めた国（EU、英国等）への移転は日本国内と同等

**実務上の注意点**:
- 海外クラウドサービスのデータセンター所在地を確認すること
- 外部AI API（OpenAI等）利用時、データの処理・保存場所に注意
- 越境移転に関する同意は個別かつ明確に取得すること
- 定期的に移転先の制度変更を確認し、情報提供を更新すること

### 第33条（開示）

**条文要旨**: 本人から保有個人データの開示請求があった場合、遅滞なく開示しなければならない。

**具体的要件**:
- 電磁的記録の提供を含む本人が指定する方法による開示
- 第三者提供記録の開示も対象
- 手数料の徴収は可能（実費を勘案して合理的な範囲）

**実務上の注意点**:
- 開示請求への対応手続きを整備し、窓口を明示すること
- 本人確認の手続きを適切に行うこと
- 開示しない旨の決定をした場合、その理由を通知すること

### 第34条（訂正等）

**条文要旨**: 本人から保有個人データの訂正・追加・削除の請求があった場合、利用目的の達成に必要な範囲内で遅滞なく調査を行い、必要な措置を講じなければならない。

**実務上の注意点**:
- 訂正等を行った場合・行わない場合ともに本人に通知する義務がある
- AI学習済みモデルに組み込まれた個人情報の訂正・削除要求への対応方法を事前に検討しておくこと

### 第35条（利用停止等）

**条文要旨**: 本人から保有個人データの利用停止・消去・第三者提供の停止の請求があった場合、違反がある場合または本人の権利利益が害されるおそれがある場合に、必要な措置を講じなければならない。

**具体的要件**:
- 利用停止・消去の請求事由（令和3年改正で拡大）:
  - 利用目的による制限違反
  - 適正な取得違反
  - 利用する必要がなくなった場合
  - 漏えい等が発生した場合
  - 本人の権利又は正当な利益が害されるおそれがある場合

**実務上の注意点**:
- データ削除のプロセスを技術的に整備しておくこと（バックアップからの削除を含む）
- AI学習データからの削除要求への対応手順を事前に定めること

## 2. 安全管理措置（第23条）の4分類

### 2.1 組織的安全管理措置

個人データの取扱いに関する組織体制の整備。

| チェック項目 | 内容 | 確認ポイント |
|---|---|---|
| 組織体制の整備 | 個人データ管理責任者の設置、取扱部署・担当者の明確化 | 責任者が任命されているか。組織図に反映されているか |
| 個人データの取扱規程 | 取得、利用、保存、提供、削除・廃棄の各段階の規程 | 規程が文書化され、最新版が管理されているか |
| 取扱状況の確認手段 | 個人データ取扱台帳の整備、定期的な棚卸し | 台帳が最新か。年1回以上の棚卸しを実施しているか |
| 漏えい等事案への対応体制 | インシデント対応手順、報告ルートの整備 | 手順書が存在するか。訓練を実施しているか |
| 取扱状況の定期的な見直し | 監査計画、是正措置のフォロー | 年1回以上の内部監査を実施しているか |

### 2.2 人的安全管理措置

従業者に対する教育・監督。

| チェック項目 | 内容 | 確認ポイント |
|---|---|---|
| 教育研修の実施 | 個人情報保護に関する定期研修 | 年1回以上実施しているか。受講記録があるか |
| 秘密保持契約 | 従業者との秘密保持に関する契約・誓約 | 全従業者から取得しているか。退職時の対応は |
| 就業規則への反映 | 個人情報保護義務の就業規則への明記 | 違反時の懲戒規定が含まれているか |
| 委託先従業者の管理 | 委託先の従業者教育状況の確認 | 委託契約に教育義務が含まれているか |

### 2.3 物理的安全管理措置

物理的な管理区域・機器の管理。

| チェック項目 | 内容 | 確認ポイント |
|---|---|---|
| 管理区域の設定 | サーバールーム等の入退室管理 | ICカード・生体認証等で制御されているか |
| 取扱区域の管理 | 事務室での覗き見防止、離席時の画面ロック | クリアデスク・クリアスクリーンが徹底されているか |
| 盗難防止措置 | 機器・記録媒体の施錠管理 | ワイヤーロック、施錠キャビネットを使用しているか |
| 持ち運び時の対策 | 外部持ち出し時の暗号化、承認手続き | 暗号化が必須化されているか。持出記録があるか |
| 削除・廃棄 | 復元不可能な方法での削除・廃棄 | データ消去証明書を取得しているか。廃棄記録があるか |

### 2.4 技術的安全管理措置

情報システムにおける技術的対策。

| チェック項目 | 内容 | 確認ポイント |
|---|---|---|
| アクセス制御 | 担当者・権限の最小限の設定 | 最小権限の原則が適用されているか。定期的な棚卸しがあるか |
| アクセス者の識別と認証 | ID・パスワード管理、多要素認証 | MFAが導入されているか。パスワードポリシーは適切か |
| 外部からの不正アクセス防止 | ファイアウォール、IDS/IPS、WAF | セキュリティパッチが適時適用されているか |
| 情報システムの使用に伴う漏えい防止 | 暗号化通信（TLS）、データ暗号化 | 保存時暗号化（AES-256等）、通信暗号化（TLS 1.2以上）が適用されているか |
| ログ管理 | アクセスログ、操作ログの取得・保存 | ログの保存期間は十分か。改ざん防止措置があるか |

## 3. AI/IT開発における個人情報保護

### 3.1 AI学習データの個人情報取り扱い

#### 仮名加工情報（第41条・第42条）

- **定義**: 他の情報と照合しない限り特定の個人を識別できないように加工した情報
- **メリット**: 利用目的の変更が可能、漏えい報告義務なし、開示等請求への対応不要
- **制約**: 第三者提供は原則禁止、識別行為の禁止、内部利用に限定
- **AI開発での活用**: 学習データとして社内利用する場合に有効。目的外利用が可能になるため、当初想定していなかったAI開発にも活用可能

#### 匿名加工情報（第43条-第46条）

- **定義**: 特定の個人を識別できないように、かつ復元できないように加工した情報
- **メリット**: 第三者提供が可能（本人同意不要）、目的外利用が可能
- **制約**: 加工基準の遵守、加工方法の安全管理、識別行為の禁止、公表義務
- **AI開発での活用**: 外部提供を含むオープンイノベーションでの利用に有効

#### 加工基準の実務ポイント

```
仮名加工情報:
  - 特定の個人を識別できる記述（氏名等）の削除または置換
  - 個人識別符号の削除
  - 不正利用で財産的被害が生じるおそれのある記述の削除

匿名加工情報:
  - 上記に加え、特異な記述の削除
  - 個人データとの照合性を断つ措置
```

### 3.2 外部AI API利用時の法的リスク

#### 第三者提供の該当性

- AI APIにプロンプトとして個人データを送信する行為が第三者提供に該当する可能性
- API提供事業者との間に「委託」関係が成立する場合は第三者に該当しない
- 判断基準: API提供事業者が送信データを自社目的で利用するか否か
  - 学習に利用する場合 → 第三者提供の可能性が高い
  - 学習に利用しない（API処理のみ） → 委託として整理できる可能性

#### 越境移転リスク

- 海外AI API（OpenAI, Anthropic, Google等）利用時、第28条の越境移転規制に該当
- データ処理サーバーの所在国を確認し、当該国の個人情報保護制度を把握する
- 同意取得時に移転先国の情報を提供する義務

#### 安全管理措置の確保

- API利用規約で個人データの安全管理に関する条項を確認
- 通信の暗号化（TLS 1.2以上）の確認
- データ保持期間・削除ポリシーの確認
- SOC 2 Type II等のセキュリティ認証の確認

### 3.3 プロンプトインジェクション経由の漏洩リスク

#### リスク概要

- 攻撃者がプロンプトインジェクションにより、他のユーザーの個人情報をAIに出力させるリスク
- システムプロンプトに含まれる個人情報の抽出リスク
- RAG（検索拡張生成）で参照されるデータベース内の個人情報の漏洩リスク

#### 対策

- 個人情報をプロンプトやシステムプロンプトに直接含めない設計
- 入力バリデーションによるインジェクション攻撃の検知・遮断
- 出力フィルタリングによる個人情報の検知・マスキング
- RAGの検索範囲をユーザー権限に応じて制限
- 定期的なペネトレーションテストの実施

### 3.4 AI生成コードへの個人情報混入リスク

#### リスク概要

- AIコード生成ツールが学習データに含まれる個人情報をコード内に出力するリスク
- テストデータとして実在の個人情報がハードコードされるリスク
- ログ出力やエラーメッセージに個人情報が含まれるリスク

#### 対策

- コードレビューにおける個人情報混入チェックの義務化
- CI/CDパイプラインでの個人情報パターン検出ツールの導入
- テストデータには必ずダミーデータを使用するルールの徹底
- ログ出力の個人情報マスキング（氏名、メールアドレス、電話番号等のパターン検知）

## 4. 漏えい時の対応フロー

### 4.1 報告対象となる4類型

以下のいずれかに該当する漏えい等が発生し、または発生したおそれがある場合に報告義務が発生する。

| 類型 | 内容 | 具体例 |
|---|---|---|
| 要配慮個人情報の漏えい | 要配慮個人情報が含まれる場合（件数不問） | 健康診断結果の流出、犯罪歴の漏洩 |
| 財産的被害のおそれ | 不正利用で財産的被害が生じるおそれ（件数不問） | クレジットカード番号の流出、口座情報の漏洩 |
| 不正の目的によるおそれ | 不正アクセス等の不正な目的による漏えい（件数不問） | サイバー攻撃によるデータ窃取、内部不正 |
| 1,000件超の漏えい | 1,000件を超える個人データの漏えい | 大規模なデータベースの流出 |

### 4.2 速報と確報の要件

#### 速報（事態発覚後3-5日以内）

- **報告先**: 個人情報保護委員会（委員会のウェブサイト経由）
- **期限**: 速やかに（概ね3-5日以内）
- **報告事項**: 判明している範囲で以下を報告
  - 概要
  - 漏えい等が発生し、又は発生したおそれがある個人データの項目
  - 漏えい等が発生し、又は発生したおそれがある個人データに係る本人の数
  - 原因
  - 二次被害又はそのおそれの有無及びその内容
  - その他参考となる事項

#### 確報（30日以内 / 不正目的の場合60日以内）

- **期限**: 事態を知った日から30日以内（不正目的による場合は60日以内）
- **報告事項**: 速報の事項に加え、以下を確定して報告
  - 再発防止のための措置
  - 本人への対応状況

### 4.3 本人通知義務

- **原則**: 報告対象の漏えい等が発生した場合、本人に通知しなければならない
- **通知内容**: 事態の概要、漏えい等が発生した個人データの項目、原因、二次被害防止のための対応
- **代替措置**: 本人への通知が困難な場合（連絡先不明等）、事態の公表等の代替措置

### 4.4 インシデント対応手順書テンプレート

```
=== 個人情報漏えいインシデント対応手順書 ===

■ Phase 1: 初動対応（発覚後即時）
  □ 漏えい等の事実確認
  □ 被害拡大防止の応急措置
    - 該当システムの遮断・隔離
    - アカウントの停止・パスワードリセット
    - ネットワークの切断
  □ 個人データ管理責任者への報告
  □ 経営層への報告
  □ 対応チームの招集

■ Phase 2: 速報（発覚後3-5日以内）
  □ 個人情報保護委員会への速報
  □ 影響範囲の暫定把握
    - 対象データの種類と件数
    - 影響を受ける本人の範囲
  □ 報告書（速報版）の作成・提出

■ Phase 3: 調査・原因究明（確報期限まで）
  □ フォレンジック調査の実施
  □ ログ分析による影響範囲の確定
  □ 原因の特定（技術的原因、人的原因）
  □ 証拠の保全

■ Phase 4: 本人通知
  □ 通知対象者の確定
  □ 通知文書の作成
  □ 通知手段の決定（個別通知/公表）
  □ 問い合わせ窓口の設置

■ Phase 5: 確報（30日/60日以内）
  □ 調査結果の取りまとめ
  □ 再発防止策の策定
  □ 個人情報保護委員会への確報
  □ 報告書（確報版）の作成・提出

■ Phase 6: 再発防止・フォローアップ
  □ 再発防止策の実施
  □ 実施状況のモニタリング
  □ 対応プロセスの振り返り・改善
  □ 訓練への反映
```

## 6. 要配慮個人情報の一覧と取扱い

### 要配慮個人情報の類型

| 類型 | 具体例 |
|---|---|
| 人種 | 人種、民族的出身 |
| 信条 | 宗教、思想、政治的見解 |
| 社会的身分 | 出生により決定される社会的地位（被差別部落出身等） |
| 病歴 | 過去・現在の疾病に関する情報 |
| 犯罪の経歴 | 前科（有罪判決を受けた事実） |
| 犯罪により害を被った事実 | 犯罪被害の事実 |
| 身体障害・知的障害・精神障害等 | 障害者手帳の交付、障害の等級 |
| 健康診断等の結果 | 定期健康診断、ストレスチェックの結果 |
| 医師等による指導・診療・調剤 | 診療録、処方箋の情報 |
| 刑事事件に関する手続き | 逮捕、捜索、起訴等の事実 |
| 少年の保護事件に関する手続き | 少年審判に関する事実 |
| ゲノムデータ | 本人を識別できるゲノム情報 |

### 取扱いの特則

- **取得**: 原則として本人の事前同意が必要（第20条第2項）
- **オプトアウト**: 要配慮個人情報はオプトアウトによる第三者提供は不可（第27条第2項）
- **AI学習データ**: 要配慮個人情報を学習データに含める場合、取得段階で当該目的の同意が必要

## 7. GDPR/APPI 比較表

| 項目 | APPI（日本） | GDPR（EU） |
|---|---|---|
| **適用範囲** | 日本国内で個人情報を取り扱う事業者 | EU域内のデータ主体に関するデータを処理する全事業者（域外適用あり） |
| **個人情報の定義** | 生存する個人に関する情報で特定の個人を識別できるもの | 識別された又は識別可能な自然人に関するすべての情報 |
| **法的根拠** | 利用目的の特定・通知/公表が中心。同意は目的外利用・第三者提供時 | 6つの法的根拠（同意、契約、法的義務、生命利益、公共利益、正当利益）から選択 |
| **同意の要件** | 「本人の同意」（明確な定義なし） | 自由意思による、特定的、説明を受けた上での、明確な意思表示 |
| **DPO設置** | 義務なし（努力義務的に管理責任者の設置を推奨） | 一定の要件に該当する場合、DPO設置が義務 |
| **DPIA** | 義務なし | 高リスクな処理を行う場合にDPIA実施が義務 |
| **データポータビリティ** | 開示請求の方法として電磁的記録を追加（令和3年改正） | データポータビリティの権利として明確に規定 |
| **忘れられる権利** | 利用停止・消去の請求権（令和3年改正で拡大） | 消去の権利として明確に規定 |
| **越境移転** | 本人同意＋情報提供、または基準適合体制 | 十分性認定、SCCs、BCRs等 |
| **漏えい報告** | 個人情報保護委員会に速報（3-5日）・確報（30/60日） | 監督当局に72時間以内に報告 |
| **制裁金** | 最大1億円（法人） | 最大2,000万ユーロ又は全世界売上高の4%のいずれか高い方 |
| **プロファイリング** | 明確な規定なし | 自動処理のみに基づく決定に服さない権利 |

### 日本固有の要件（GDPR にないもの）

- **仮名加工情報制度**: 社内利用に限り利活用を柔軟化する日本独自の制度
- **個人関連情報（第31条）**: Cookie等の個人関連情報の第三者提供規制
- **オプトアウト制度**: 届出制によるオプトアウトでの第三者提供（GDPRには同等制度なし）
- **4類型の報告基準**: 漏えい報告の対象を4類型で具体的に規定（GDPRは「リスクをもたらす可能性が高い」場合）

## 8. チェックリスト（個人情報取扱い全フロー準拠確認用）

### Phase A: 取得段階

- [ ] 利用目的が具体的に特定されているか（第17条）
- [ ] 利用目的が本人に通知又は公表されているか（第21条）
- [ ] 適正な手段で取得しているか（第20条第1項）
- [ ] 要配慮個人情報の場合、本人の事前同意を得ているか（第20条第2項）
- [ ] 個人関連情報の取得の場合、提供元の確認義務を果たしているか（第31条）
- [ ] 第三者から取得する場合、取得経緯の確認を行っているか（第30条）

### Phase B: 利用段階

- [ ] 特定した利用目的の範囲内で利用しているか（第18条）
- [ ] 目的外利用の場合、本人同意を得ているか（第18条例外規定確認）
- [ ] データの正確性・最新性が確保されているか（第22条）
- [ ] 安全管理措置が講じられているか（第23条）
  - [ ] 組織的安全管理措置
  - [ ] 人的安全管理措置
  - [ ] 物理的安全管理措置
  - [ ] 技術的安全管理措置
- [ ] 従業者の監督が適切に行われているか（第24条）

### Phase C: 提供段階

- [ ] 第三者提供の場合、本人同意を得ているか（第27条）
- [ ] オプトアウトの場合、届出・公表がされているか（第27条第2項）
- [ ] 委託の場合、委託先の監督が適切か（第25条）
- [ ] 共同利用の場合、所定の事項が本人に通知又は公表されているか（第27条第5項第3号）
- [ ] 越境移転の場合、情報提供義務を果たしているか（第28条）
- [ ] 提供記録・受領記録が作成・保存されているか（第29条・第30条）

### Phase D: 保存・管理段階

- [ ] 保存期間が定められ、不要になったデータは削除されているか
- [ ] バックアップデータの管理が適切か
- [ ] アクセス権限が最小限に設定されているか
- [ ] ログの取得・保存が適切に行われているか
- [ ] 定期的な棚卸し・監査が実施されているか

### Phase E: 開示等請求対応

- [ ] 開示請求への対応手続きが整備されているか（第33条）
- [ ] 訂正等の請求への対応手続きが整備されているか（第34条）
- [ ] 利用停止等の請求への対応手続きが整備されているか（第35条）
- [ ] 苦情処理の体制が整備されているか（第40条）
- [ ] 請求窓口が本人に明示されているか

### Phase F: 漏えい等対応

- [ ] インシデント対応手順書が整備されているか
- [ ] 報告対象4類型を理解し、判断基準が明確か
- [ ] 速報・確報の手続きが整備されているか
- [ ] 本人通知の手続きが整備されているか
- [ ] 定期的な訓練を実施しているか

### Phase G: AI/IT固有チェック

- [ ] AI学習データに個人情報が含まれる場合、適法な取得・利用か
- [ ] 外部AI APIへの個人データ送信が第三者提供/越境移転に該当しないか確認したか
- [ ] プロンプトインジェクション対策を講じているか
- [ ] AI生成コードへの個人情報混入チェックを行っているか
- [ ] テストデータにダミーデータを使用しているか
- [ ] ログへの個人情報出力のマスキングを実装しているか
