## 1. 自動修復システム概要

自動修復システムは、Claude Code 実行中に発生するエラー・異常・品質低下を自動検知し、可能な限り人手を介さずに回復する仕組みである。

### 1.1 設計思想

```
         検知              診断              修復              検証
    ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
    │  Monitor │ ──→ │ Diagnose │ ──→ │  Repair  │ ──→ │  Verify  │
    │          │     │          │     │          │     │          │
    │ Hook検知 │     │ パターン │     │ 自動修復 │     │ テスト   │
    │ 定期監視 │     │ 照合     │     │ or 提案  │     │ 再実行   │
    │ 閾値超過 │     │ 重大度   │     │ or 報告  │     │ 整合性   │
    └──────────┘     └──────────┘     └──────────┘     └──────────┘
```

**4つの原則:**

1. **検知 → 診断 → 修復 → 検証** の4フェーズを必ず通す（修復だけして検証しない、は許さない）
2. **自動修復可能なもの**は即座に修復し、ログを残す
3. **人間の判断が必要なもの**はフラグ（YELLOW/RED）で通知し、修復案を添える
4. **修復履歴**を蓄積し、パターン学習に活用する（同じ失敗を2度繰り返さない）

### 1.2 スコープ

| 対象 | 含む | 含まない |
|------|------|---------|
| 実行環境 | ローカル開発環境、Codespaces | 本番環境（別途インシデント対応） |
| 操作種別 | ファイル操作、Git、テスト、ビルド、lint | DB直接操作、インフラ変更 |
| エージェント | 全65エージェント + Rally並列実行 | 外部CI/CDパイプライン |

## 2. 自動修復レベル

| Level | 名称 | 説明 | 人間の介入 | 応答時間目標 |
|-------|------|------|-----------|------------|
| **AR-L0** | 予防（Prevention） | 問題が発生する前に防ぐ | 不要 | 即時（ツール実行前） |
| **AR-L1** | 即時修復（Instant Fix） | 検知後、自動で修復 | 不要 | 5秒以内 |
| **AR-L2** | ガイド付き修復（Guided Fix） | 修復案を提示し、確認後に実行 | 確認のみ | 30秒以内（提案まで） |
| **AR-L3** | 手動修復（Manual Fix） | 問題と原因を報告し、人間が修復 | 必須 | 即時報告 |

### 2.1 レベル判定フロー

```
問題検知
    │
    ├─ 既知パターン？
    │   ├─ Yes → パターンDBから修復レベルを取得
    │   └─ No  → 重大度判定へ
    │
    ├─ 重大度判定
    │   ├─ データ損失リスクあり？ ──→ AR-L3（手動修復）
    │   ├─ セキュリティリスクあり？ ──→ AR-L2以上
    │   ├─ 可逆操作で修復可能？ ──→ AR-L1（即時修復）
    │   └─ 事前防止可能？ ──→ AR-L0（予防）
    │
    └─ 修復可能性判定
        ├─ 自動修復の成功率 > 95% → AR-L1
        ├─ 自動修復の成功率 > 70% → AR-L2（確認付き）
        └─ 自動修復の成功率 < 70% → AR-L3（手動）
```

## 3. 予防メカニズム（AR-L0）

問題が発生する前にブロック・修正する。最もコスト効率が高い。

| ID | 予防対象 | メカニズム | 実装箇所 | 検知方法 |
|----|---------|-----------|---------|---------|
| AR-L0-001 | .envのコミット | .gitignore 自動チェック + 追加 | install.sh | PreToolUse Hook で `git add .env` 検知 |
| AR-L0-002 | シークレットのハードコード | PreToolUse Hook でパターン検知 | tool-risk.js | 正規表現: `AKIA[0-9A-Z]{16}`, `sk-[a-zA-Z0-9]{48}`, `ghp_[a-zA-Z0-9]{36}` |
| AR-L0-003 | 破壊的コマンド実行 | Permissions deny ルール | settings.json | `rm -rf /`, `git push --force`, `DROP TABLE` 等のパターンマッチ |
| AR-L0-004 | ファイルオーナーシップ競合 | 並列実行前のロック取得 | PARALLEL プロトコル | ownership_map との照合 |
| AR-L0-005 | コンテキスト肥大化 | SLIM_CONTEXT プロトコルの自動発動 | SLIM_CONTEXT.md | トークン予算の閾値監視（80%超過で圧縮開始） |
| AR-L0-006 | Sandbox 脱出 | allowUnsandboxedCommands: false | settings.json | Sandbox設定の整合性チェック |
| AR-L0-007 | テスト未実行での完了宣言 | TEST_POLICY プロトコル | TEST_POLICY.md | PostToolUse Hook でテスト実行有無を追跡 |
| AR-L0-008 | 仕様なしの実装開始 | SPEC_FIRST プロトコル | SPEC_FIRST.md | 仕様ファイルの存在チェック |
| AR-L0-009 | 機密ファイルの読み取り | Permissions + Sandbox | settings.json | `.env`, `*.pem`, `*.key`, `credentials.json` へのアクセス検知 |
| AR-L0-010 | 無限ループ/コスト暴走 | Safety Gate パターン | tool-risk.js | `while true`, 大量ループのパターン検知 |

```
│                                        │

```

### 8.4 メトリクス

| メトリクス | 計算方法 | 健全な範囲 |
|-----------|---------|-----------|
| 修復成功率 | 成功修復数 / 全修復試行数 | > 85% |
| 平均修復時間 | 全修復の duration_ms 平均 | < 10秒（AR-L1） |
| エスカレーション率 | エスカレーション数 / 全検知数 | < 20% |
| 再発率 | 同一パターン再発 / 全検知数 | < 10% |
| 偽陽性率 | 不要な修復 / 全修復数 | < 5% |

## 9. Guardrail 連携

Auto-Repair システムは Guardrail プロトコル（`GUARDRAIL.md`）と密接に連携する。

### 9.1 マッピング

| Guardrail Level | Auto-Repair Level | 動作 | トリガー例 |
|----------------|-------------------|------|-----------|
| L1 (MONITORING) | AR-L1 | 自動修復 + ログ記録 | lint警告、軽微な非推奨 |
| L2 (CHECKPOINT) | AR-L1 or AR-L2 | 自動検証 → 修復 | テスト失敗<20%、型エラー |
| L3 (PAUSE) | AR-L2 or AR-L3 | 自動回復 or ユーザー待機 | テスト失敗>50%、破壊的変更 |
| L4 (ABORT) | AR-L3 + 即時停止 | エスカレーション | セキュリティ、データ整合性 |

### 9.2 Guardrail Auto-Recovery との統合

```
Guardrail L2 Recovery             Auto-Repair
┌──────────────────┐      ┌────────────────────┐
│ test_failure_minor│ ───→ │ AR-L1-002 リトライ │
│ type_error        │ ───→ │ AR-L1-008 型修正   │
│ lint_error        │ ───→ │ AR-L1-003 auto-fix │
└──────────────────┘      └────────────────────┘

Guardrail L3 Recovery             Auto-Repair
┌──────────────────┐      ┌────────────────────┐
│ test_failure_major│ ───→ │ AR-L2-005 breaking │
│ breaking_change   │ ───→ │ AR-L2-005 影響分析 │
│ build_failure     │ ───→ │ AR-L1-009 キャッシュ│
│                   │      │ or AR-L2-003 デプロイ│
└──────────────────┘      └────────────────────┘
```

### 9.3 Task Type 別のデフォルト修復方針

```yaml
FEATURE:
  guardrail: L2
  auto_repair:
    lint_error: AR-L1-003    # 即時修復
    type_error: AR-L1-008    # 即時修復（2回まで）
    test_failure: AR-L1-002  # リトライ後 AR-L2

SECURITY:
  guardrail: L2
  auto_repair:
    secret_leak: AR-L2-001   # 必ずガイド付き
    vulnerability: AR-L3-003 # 手動対応
    lint_error: AR-L1-003    # 即時修復

REFACTOR:
  guardrail: L2
  auto_repair:
    test_regression: AR-L2-005  # Breaking change として扱う
    type_error: AR-L1-008       # 即時修復
    lint_error: AR-L1-003       # 即時修復

INCIDENT:
  guardrail: L3
  auto_repair:
    all_errors: AR-L2 以上      # 安全側に倒す
    data_issue: AR-L3-001       # 手動対応必須
```

## 10. Hook 連携

### 10.1 Hook → Auto-Repair 接続マップ

| Hook | Phase | Auto-Repair Connection | 発火条件 |
|------|-------|----------------------|---------|
| PreToolUse (tool-risk.js) | 実行前 | AR-L0 予防（ブロック/警告） | 危険なコマンド・シークレット検知 |
| PostToolUse (post-tool-use.js) | 実行後 | AR-L1 即時修復トリガー | エラー検知・異常出力 |
| Stop (stop-hook.js) | 終了時 | 修復ログサマリ + 未修復問題一覧 | セッション終了 |

### 10.2 PreToolUse Hook（AR-L0 発動）

```
ツール実行リクエスト
    │
    ▼
tool-risk.js
    │
    ├─ Safety Gate パターン検知？
    │   ├─ シークレット送信 → BLOCK + AR-L0-002 記録
    │   ├─ .env コミット → BLOCK + AR-L0-001 発動（.gitignore修復）
    │   ├─ 破壊的コマンド → BLOCK + AR-L0-003 記録
    │   └─ 無限ループ → BLOCK + AR-L0-010 記録
    │
    ├─ HIGH リスク？
    │   └─ 確認ダイアログ表示 + additionalContext 注入
    │
    ├─ MEDIUM リスク？
    │   └─ 説明表示 + additionalContext 注入
    │
    └─ LOW リスク？
        └─ サイレント通過
```

### 10.3 PostToolUse Hook（AR-L1 トリガー）

```
ツール実行完了
    │
    ▼
post-tool-use.js
    │
    ├─ 実行結果を記録（ログ）
    │
    ├─ エラー検知？
    │   ├─ exit code != 0
    │   ├─ stderr に ERROR/FATAL
    │   └─ 期待出力との不一致
    │
    ├─ エラー種別判定
    │   ├─ lint エラー → AR-L1-003 発動
    │   ├─ テスト失敗 → AR-L1-002 発動
    │   ├─ 型エラー → AR-L1-008 発動
    │   ├─ import 未解決 → AR-L1-004 発動
    │   ├─ ビルドエラー → AR-L1-009 試行 → AR-L2-003
    │   ├─ ポート競合 → AR-L1-010 発動
    │   └─ 不明エラー → ログ記録 + 分析待ち
    │
    └─ ファイル変更の妥当性チェック
        ├─ .env が変更された → AR-L0-001 チェック
        └─ オーナーシップ外のファイル変更 → 警告
```

### 10.4 Stop Hook（修復サマリ）

```
セッション終了
    │
    ▼
stop-hook.js
    │
    ├─ 修復ログサマリ生成
    │   ├─ 修復成功: N件
    │   ├─ 修復失敗: N件
    │   ├─ エスカレーション: N件
    │   └─ 未修復（保留）: N件
    │
    ├─ 未修復問題の一覧
    │   └─ パターンID + 説明 + 推奨アクション
    │
    ├─ lessons.md への書き込み
    │   └─ 新規発見されたパターン
    │
    └─ PROJECT.md Activity Log 更新
        └─ 修復アクティビティのサマリ
```

```yaml
      command: "docker compose ps --format json"
```

## 12. フラグシステムとの連携

### 12.1 フラグ → Auto-Repair マッピング

```
┌─────────┬──────────────────────┬──────────────────────┐
│  Flag   │  Auto-Repair Level   │  動作                │
├─────────┼──────────────────────┼──────────────────────┤
│  RED    │  AR-L3 or 即時停止   │  人間の介入必須      │
│  FLAG   │                      │  操作の即時停止      │
│         │                      │  エスカレーション    │
├─────────┼──────────────────────┼──────────────────────┤
│ YELLOW  │  AR-L2               │  ガイド付き修復      │
│  FLAG   │  （ガイド付き修復）  │  修復案を提示        │
│         │                      │  ユーザー確認を待つ  │
├─────────┼──────────────────────┼──────────────────────┤
│  GREEN  │  AR-L1 or 情報のみ   │  即時修復 or ログ    │
│  FLAG   │                      │  自動修復を試行      │
│         │                      │  結果をログに記録    │
└─────────┴──────────────────────┴──────────────────────┘
```

### 12.2 フラグ発行基準

```yaml
RED_FLAG:
  triggers:
    - pattern: "AR-L3-*"             # 全ての AR-L3 パターン
    - severity: "CRITICAL"            # 重大度 CRITICAL
    - secret_leak: true               # シークレット漏洩
    - data_loss: true                 # データ損失
    - force_push_shared_branch: true  # 共有ブランチへの force push
  action: "STOP + ESCALATE"

YELLOW_FLAG:
  triggers:
    - pattern: "AR-L2-*"             # 全ての AR-L2 パターン
    - severity: "HIGH"                # 重大度 HIGH
    - ar_l1_escalation: true          # AR-L1 からのエスカレーション
    - test_failure_rate: "> 50%"      # テスト失敗率50%超
    - performance_degradation: "> 200%"  # パフォーマンス劣化200%超
  action: "PAUSE + PROPOSE_FIX"

GREEN_FLAG:
  triggers:
    - pattern: "AR-L1-*"             # 全ての AR-L1 パターン（修復成功時）
    - severity: "LOW"                 # 重大度 LOW
    - auto_fix_success: true          # 自動修正成功
  action: "LOG + CONTINUE"
```

```
    ▼

         │
    ┌────┴────┬────────────┐
    │         │            │
    ▼         ▼            ▼
 GREEN     YELLOW        RED
 FLAG      FLAG          FLAG
    │         │            │
    ▼         ▼            ▼
Guardrail  Guardrail   Guardrail
 L1-L2     L2-L3       L3-L4
    │         │            │
    ▼         ▼            ▼
 AR-L1     AR-L2        AR-L3
 即時修復  ガイド付き    手動修復
    │         │            │
    ▼         ▼            ▼

    │         │            │
    ▼         ▼            ▼
  完了      修復/却下    人間が対応
```

## 15. 設計決定の根拠

| 決定事項 | 根拠 |
|---------|------|
| AR-L1 の自動修復は検証必須 | 修復が新たな問題を引き起こす可能性を排除するため |
| AR-L1-005（Git コンフリクト）の成功率が低い（60%） | ロジック変更の自動判定は困難。安全側に倒し、積極的に AR-L2 にエスカレーション |
| シークレット関連は全て AR-L2 以上 | 自動修復の誤判定リスクが高く、人間の確認が必須 |
| パフォーマンス劣化の閾値は 200% | 正常な変動（150%以内）と真の劣化を区別するため |
| パターン学習は週次レビュー | MAINTENANCE プロトコルと同期。日次では変更頻度が高すぎる |
| kill -9 は使わない（AR-L1-010） | graceful shutdown を優先。データ破損リスク回避 |
| AR-L3 は報告のみ | クリティカルな問題の自動修復は二次被害のリスクが高い |
