# AIで開発チームを作る ── LM Orchestrator 初学者ガイド

> この文書は「ChatGPTで雑談したことはあるけど、それ以外はよくわからない」という方向けに書かれています。
> 専門用語はできるだけ使わず、出てきたときは必ず説明します。

---

## 1. AIは「会話」だけじゃない

普段使っている ChatGPT は、AIの一種です。

質問すると答えてくれる。相談に乗ってくれる。文章を考えてくれる。
でも、**AIにできることは「会話」だけじゃありません**。

| よく知られているAIの使い方 | 実はAIにできること |
|---|---|
| 質問に答える | プログラムを書く |
| 相談に乗る | ファイルを作る・編集する |
| 文章を考える | バグ（プログラムの不具合）を見つけて直す |
| 翻訳する | ウェブサイトを自動で操作する |
| | データを分析してグラフを作る |
| | チームを組んで大きな仕事を分担する |

右側の「実はできること」を実現するためのAIが、このプロジェクトのテーマです。

---

## 2. ソフトウェア開発の流れ

**開発** = ソフトウェアを作ること。

私たちが日常的に使っている LINE、YouTube、Instagram、業務システム。
これらはすべて、誰かが **プログラム（コンピュータへの命令書）** を書いて作っています。

```
開発の基本的な流れ:

  企画する → 設計する → コードを書く → テストする → リリースする
  「何を         「どういう      「実際に       「ちゃんと      「ユーザーが
   作るか」      仕組みに       命令を        動くか       使える状態に
               するか」      書く」       確認する」     する」
```

この「コードを書く」「テストする」という部分を、**AIに任せる**のが最近の流れです。
任せるどころか、AIがほぼ全部やってくれるケースも増えています。

## 3. Claude Code とは

**Claude Code（クロード・コード）** は、Anthropic という会社が作った「開発ができるAI」です。

普通のChatGPTとの違いを整理します。

| | ChatGPT（一般的なAI） | Claude Code（開発AI） |
|---|---|---|
| イメージ | 何でも答えてくれる相談役 | 一緒に手を動かしてくれるメンバー |
| できること | 「教えて」→「答えるよ」 | 「これ作って」→「はい、作りました」 |
| ファイル操作 | できない | **ファイルを作る・読む・書き換えられる** |
| コマンド実行 | できない | **コンピュータに直接命令できる** |
| 操作画面 | ブラウザのチャット画面 | **ターミナル**（黒い画面）で動く |

つまり、Claude Code は **「話すだけ」ではなく「実際に作業してくれる」AI** です。

「ログイン画面を作って」と依頼すれば、本当にファイルを作ってコードを書いてくれます。

### 安全性について知っておくべきこと

Claude Code は強力な分、**安全に使う仕組み**が重要です。
LM Orchestrator には以下の安全装置が組み込まれています:

- **Tool Risk Hook**: AIがコマンドを実行する前に、「これは危険な操作か？」を自動判定
- **Permissions設定**: 実行を許可するコマンドのホワイトリスト管理
- **Secret保護**: APIキーやパスワードが誤ってコードに書き込まれたり、外部に送信されたりすることを防止

初心者は `--with-hooks` オプション付きでインストールすることを強く推奨します。

## 4. 1つのAIだけでは限界がある

Claude Code は強力ですが、1つのAIだけだと困ることがあります。

たとえば、こんな状況を想像してください。

> **依頼者**: 「新規サービスの立ち上げ、全部やって！」
>
> **1つのAI**: 「画面を作って、データベースを設計して、APIを組んで、テストして、
>              セキュリティもチェックして......全部1人では手が回りません」

人間のチームと同じで、**AIも1つで全部こなすのは難しい**んです。
特に：

- 作業量が多いと、前にやったことの文脈を忘れる
- 得意な領域と苦手な領域がある
- 同時に複数のことは処理できない

ではどうするか？

**→ AIでチームを組めばいい。**

## 5. 「オーケストラ」という考え方

ここでこのプロジェクトの名前「**LM Orchestrator（エルエム・オーケストレーター）**」の話です。

### オーケストラの仕組み

音楽のオーケストラを思い浮かべてください。

```
              指揮者
            ／  |  ＼
       バイオリン  トランペット  ドラム
       （弦楽器）   （管楽器）   （打楽器）
```

- **指揮者**は自分では楽器を弾かない。でも、誰がいつ演奏するかを決める
- **各パート**は自分の担当楽器に集中する
- 全員がバラバラに弾いたら騒音。指揮者がまとめるから**美しい音楽**になる

このプロジェクトは、これと同じことを **AIで** やります。

### AIのオーケストラ

```
              Nexus（指揮者AI）
              ／    |    ＼
           Scout   Builder   Radar
          （調査役） （開発役） （テスト役）
```

- **Nexus（ネクサス）** = 指揮者。自分ではコードを書かない。「誰に何を任せるか」を判断する
- **Scout（スカウト）** = 調査役。バグ（不具合）の原因を突き止める専門家
- **Builder（ビルダー）** = 開発役。実際にコードを書く実装担当
- **Radar（レーダー）** = テスト役。作ったものが正しく動くか検証する

他にも **73種類のAI専門家** がいます（セキュリティ、デザイン、データ分析など）。

## 6. 実際の動き方

たとえば「ログイン画面にバグがある」と報告があったとします。

### ステップ 1: 指揮者が受け取る

```
依頼者: 「ログイン画面でエラーが出ています。修正をお願いします」
Nexus:  「了解。まず原因調査が必要ですね。Scout、調べてください」
```

### ステップ 2: 調査役が調べる

```
Scout: 「原因を特定しました。パスワードの文字数チェックにバグがあります。
        該当箇所は login.js の42行目です」
Nexus: 「ありがとうございます。Builder、修正をお願いします」
```

### ステップ 3: 開発役が修正する

```
Builder: 「修正完了です。login.js の42行目を書き換えました」
Nexus:   「Radar、テストをお願いします」
```

### ステップ 4: テスト役が検証する

```
Radar: 「全テスト通過。問題ありません」
Nexus: 「修正完了です。全工程OKです」
```

**依頼者がやったことは最初の一言だけ**。あとはAIチームが対応してくれます。

## 8. なぜチームにするのか

1つのAIに全部やらせることもできます。それでもチーム体制にする理由があります。

### 理由 1: 専門特化のほうが精度が高い

バグを直すとき、「バグを探すのが得意なAI」と「コードを書くのが得意なAI」を分けたほうが、それぞれの精度が上がります。

### 理由 2: 並行処理で速くなる

```
1つだと:  調査 → 開発 → テスト → レビュー （順番にやるから遅い）

チームだと: 調査（Scout）
           開発（Builder）  ← 同時に進められる
           テスト（Radar）
```

### 理由 3: クロスチェックでミスが減る

Builder が書いたコードを Judge がチェックする。
「自分で書いた文章の誤りには気づきにくい」のと同じで、別のAIが確認することでミスを減らせます。

### 理由 4: 安全装置がある

このシステムには **ガードレール（安全装置）** が組み込まれています。

| レベル | 発生条件 | 対応 |
|--------|---------|------|
| L1 | 軽微な警告 | ログに記録して続行 |
| L2 | テストが一部失敗 | 自動で修正を試みる（最大3回） |
| L3 | テストが大量に失敗 | 変更を元に戻して再計画 |
| L4 | セキュリティ上の問題 | **即座に全停止** |

暴走しないように、段階的なブレーキが設計されています。

さらに、**Tool Risk Hook** がツール実行前に危険度を自動判定し、破壊的な操作（ファイル削除、本番デプロイなど）を事前に警告します。

## 9. 一般的なAIとの違い

まとめると、こういう違いがあります。

```
【一般的なAI（ChatGPTなど）】

  あなた ←→ AI（1つ）

  「〇〇って何？」「〇〇について教えて」
  → 回答が返ってくる
  → ただし、実際に何かを「作る」ことはしない


【LM Orchestrator】

  あなた → Nexus（指揮者）→ 専門家チーム（73種類）→ 成果物
                                                      ↑
                                              実際にファイルが作られる
                                              コードが書かれる
                                              テストが実行される
                                              バグが修正される
```

**「教えてもらう」から「作ってもらう」へ。**

これが、このシステムの本質です。

## 10. 具体的な活用例

### 例 1: キャンペーンページを作りたい

```
依頼: 「新しいキャンペーンのLPを作ってほしい」

→ Nexus が作業計画を立てる
→ Builder がHTML/CSSを実装する
→ Artisan がデザインを調整する
→ Radar がスマホ表示を含めてテストする
→ 完成
```

### 例 2: データを分析したい

```
依頼: 「先月のユーザーデータを集計してレポートにまとめてほしい」

→ Nexus が Analyst に任せる
→ Analyst がデータを分析する
→ Canvas がグラフを作成する
→ 完成
```

### 例 3: アプリの不具合を直したい

```
依頼: 「ボタンを押しても反応しない」

→ Nexus が Scout に調査を依頼する
→ Scout が原因を特定する
→ Builder が修正する
→ Radar がテストする
→ Sentinel がセキュリティを確認する
→ 完成
```

## 11. 利用に必要なもの

現時点では、このシステムを利用するには以下が必要です。

1. **Claude Code** がインストールされたコンピュータ
2. **ターミナル**（コンピュータに文字で命令を出す画面）の基本操作
3. 開発業務の基本的な理解

技術的な詳細を深く知らなくても、**このシステムの概念を理解していること自体が重要**です。

AIは「質問に答えてくれる便利ツール」から「一緒に仕事をするチームメンバー」へと進化しています。
この流れを知っておくことで、今後のAI活用の判断がしやすくなるはずです。

## 12. 用語集

この文書に登場した専門用語をまとめます。

| 用語 | 意味 |
|------|------|
| **AI** | 人工知能。人間のように判断・推論するコンピュータの仕組み |
| **プログラム** | コンピュータへの命令書。これを書く作業が「プログラミング」 |
| **コード** | プログラムの中身（命令文）のこと |
| **バグ** | プログラムの不具合。意図しない動作をすること |
| **開発** | ソフトウェア（アプリやウェブサービス）を作ること |
| **ターミナル** | コンピュータにテキストで命令を出す画面 |
| **Claude Code** | Anthropic社が開発した、ソフトウェア開発が可能なAI |
| **エージェント** | 特定の役割を持ったAI。チームの一員として機能する |
| **オーケストレーター** | 複数のエージェントを統括する指揮者的な存在 |
| **リポジトリ** | プログラムのソースコードを保管・管理する場所 |
| **テスト** | 作ったものが正しく動くか検証すること |
| **セキュリティ** | 情報やシステムを不正アクセスなどから守ること |
| **Git** | ファイルの変更履歴を記録する仕組み。「いつ誰が何を変えたか」を追跡できる |
| **Hook** | ツール実行の前後に自動で走る安全チェック機構 |
