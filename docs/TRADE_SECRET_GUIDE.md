# 営業秘密保護ガイドライン

> 本文書は、LM-orchestrator-business の開発・運用における営業秘密の保護方針を、不正競争防止法および経済産業省「営業秘密管理指針」に基づき整理したものである。

---

## 1. 営業秘密の3要件（不正競争防止法 第2条第6項）

不正競争防止法上の「営業秘密」として保護を受けるためには、以下の3要件を全て満たす必要がある。

### 1.1 秘密管理性

| 要素 | 内容 | 具体的な判断基準 |
|------|------|-----------------|
| **アクセス制限** | 秘密情報にアクセスできる者が制限されていること | 物理的・技術的なアクセス制御の実施 |
| **客観的認識可能性** | 情報にアクセスした者が秘密であると認識できること | 秘密表示、秘密保持契約の締結 |

**秘密管理性が認められる措置の例:**
- ファイルやディレクトリに「秘密」「Confidential」の表示
- アクセス制御（パーミッション設定、暗号化）
- 秘密保持契約（NDA）の締結
- 秘密情報管理規程の策定と周知
- 施錠管理、入退室管理

### 1.2 有用性

| 要素 | 内容 |
|------|------|
| **事業活動に有用** | 客観的に事業活動に役立つ情報であること |
| **広義の解釈** | 直接的な利益だけでなく、失敗情報・ネガティブ情報も含む |
| **反社会的なものは除外** | 脱税情報等は有用性を満たさない |

### 1.3 非公知性

| 要素 | 内容 |
|------|------|
| **公然と知られていない** | 保有者の管理下以外で一般的に入手できないこと |
| **判断基準** | 刊行物、インターネット、特許公報等で公開されていないこと |
| **部分的公知** | 個々の情報が公知でも、組合せに独自性があれば非公知性を満たし得る |

## 2. `.agents/` ディレクトリの営業秘密管理

### 2.1 `.agents/` ディレクトリに格納される情報の分類

`.agents/` ディレクトリには、プロジェクト固有のエージェント設定やナレッジが格納される。これらは営業秘密に該当し得る情報を含む。

| 情報カテゴリ | 例 | 営業秘密該当性 | 根拠 |
|-------------|-----|---------------|------|
| **エージェント設計** | エージェントの役割定義、プロンプト設計 | 該当し得る | 独自のノウハウ・設計思想 |
| **ワークフロー設計** | エージェント間連携フロー、判断ロジック | 該当し得る | 業務プロセスの独自性 |
| **ナレッジベース** | プロジェクト固有の技術知見、失敗事例 | 該当し得る | 蓄積された有用な情報 |
| **設定ファイル** | API キー、認証情報 | 該当する | アクセス制御に関わる秘密情報 |
| **プロジェクトデータ** | 顧客情報、取引条件 | 該当する | 事業上の秘密情報 |

### 2.2 必要な管理措置

```
.agents/
├── README.md          ← 秘密表示を記載
├── agent_config/      ← エージェント設計（秘密管理対象）
├── knowledge/         ← ナレッジベース（秘密管理対象）
├── workflows/         ← ワークフロー設計（秘密管理対象）
└── credentials/       ← 認証情報（最高秘密管理対象）
    └── .gitignore     ← Git管理から除外
```

## 3. 秘密表示の方法

### 3.1 ファイルヘッダーへの秘密表示

営業秘密に該当するファイルには、以下のヘッダーを付与する。

```
# ==============================================================================
# 秘密情報 / CONFIDENTIAL
# 本ファイルには営業秘密に該当する情報が含まれています。
# 不正競争防止法第2条第6項に基づき、秘密として管理されています。
# 権限のない者による閲覧・複製・開示は禁止されています。
# ==============================================================================
```

### 3.2 ディレクトリの README による秘密表示

秘密情報を含むディレクトリには、以下の内容の README を配置する。

```markdown
# 秘密情報ディレクトリ

本ディレクトリには、不正競争防止法第2条第6項に定める営業秘密に
該当する情報が含まれています。

## 取扱注意事項

1. 本ディレクトリ内の情報は、業務上の必要がある者のみアクセスできます
2. 権限のない者への開示・複製・送信は禁止されています
3. 本ディレクトリの内容を AI API に送信することは禁止されています
4. 退職・異動時には本ディレクトリ内の情報を返却・削除してください

## 管理者
[管理部署・連絡先]

## 最終更新日
YYYY年MM月DD日
```

### 3.3 秘密度レベルの定義

| レベル | 表示 | 定義 | 管理措置 |
|--------|------|------|----------|
| **極秘** | `[極秘/TOP SECRET]` | 漏洩により事業に重大な損害を与える情報 | 暗号化必須、アクセスログ必須、印刷禁止 |
| **秘密** | `[秘密/CONFIDENTIAL]` | 漏洩により事業に損害を与える情報 | アクセス制限、`.claudeignore` 登録 |
| **社内限定** | `[社内限定/INTERNAL]` | 社外への開示を制限する情報 | 社外送信時の承認 |
| **公開可** | 表示なし | 公開しても支障のない情報 | 特になし |

## 4. LM-orchestrator-business が提供する技術的保護措置

### 4.1 アクセス制御

| 機能 | 説明 | 営業秘密保護への寄与 |
|------|------|---------------------|
| **`.claudeignore`** | 指定ファイル・ディレクトリを AI のコンテキストから除外 | 秘密情報の AI への送信を防止 |
| **`deny` 設定** | 指定ファイル・ディレクトリへの AI のアクセスを禁止 | 秘密情報へのアクセスを技術的に制限 |
| **ガードレール** | AI の動作範囲を制限 | 秘密情報の意図しない処理を防止 |

### 4.2 データ保護

| 機能 | 説明 | 営業秘密保護への寄与 |
|------|------|---------------------|
| **`DATA_PROTECTION_REMINDER`** | プロンプトに注意喚起を挿入 | 秘密情報の取扱いに関する意識向上 |
| **フラグシステム** | ファイルの機密度に応じたフラグ設定 | 秘密情報の分類と管理の自動化 |
| **自動修復** | 設定不備の自動検出と修正 | 保護措置の漏れを防止 |

### 4.3 監査・追跡

| 機能 | 説明 | 営業秘密保護への寄与 |
|------|------|---------------------|
| **操作ログ** | AI が実行した操作の記録 | 秘密情報へのアクセス履歴の追跡 |
| **セキュリティアーキテクチャ** | 多層防御の設計 | 秘密情報の保護を体系的に実施 |

### 4.4 推奨設定例

```yaml
# CLAUDE.md での設定例

# 営業秘密を含むディレクトリへのアクセスを禁止
deny: .agents/credentials/**
deny: data/confidential/**
deny: docs/internal/**

# .claudeignore での設定例
.agents/credentials/
data/confidential/
*.secret
*.credential
```

## 5. 営業秘密管理指針（経済産業省 2023年改訂）との対応

### 5.1 管理指針の主要項目と対応状況

| 管理指針の項目 | 内容 | LM-orchestrator-business での対応 |
|---------------|------|-----------------------------------|
| **秘密管理措置の実施** | 情報にアクセスした者が秘密と認識できる措置 | 秘密表示（ヘッダー、README） |
| **アクセス制限** | 秘密情報にアクセスできる者の制限 | `.claudeignore`、`deny` 設定 |
| **物理的管理** | 施錠、入退室管理等 | （ソフトウェアのため非該当。開発環境のセキュリティで代替） |
| **技術的管理** | 暗号化、アクセス制御、ログ管理 | 暗号化、アクセス制御、操作ログ |
| **人的管理** | NDA 締結、教育・研修 | NDA テンプレート、利用ガイドライン |
| **退職・転職時の管理** | 秘密情報の返却・削除、競業避止義務 | 退職時チェックリスト |

### 5.2 AI ツール利用における追加的な管理措置

営業秘密管理指針は AI ツールの利用を明示的に想定していないため、以下の追加措置を講じる。

| 追加措置 | 内容 | 実施方法 |
|----------|------|----------|
| **AI 送信前の分類確認** | 秘密情報を AI API に送信しないことの確認 | `.claudeignore`、`deny` による自動制御 |
| **AI 出力の秘密管理** | AI が生成した秘密性のある出力の管理 | 出力ファイルの秘密度分類 |
| **プロンプトの秘密管理** | 秘密情報を含むプロンプトの管理 | プロンプト内の秘密情報のマスキング |
| **コンテキスト管理** | AI のコンテキストに含まれる情報の管理 | コンテキストウィンドウの制御 |

### 5.3 秘密情報の漏洩が発生した場合の対応

1. **即時対応**
   - 漏洩経路の遮断（API キーの失効、アクセス権の停止）
   - Anthropic へのデータ削除依頼（AI_INCIDENT_RESPONSE.md 参照）
   - 影響範囲の特定

2. **法的対応の検討**
   - 不正競争防止法に基づく差止請求・損害賠償請求の可否
   - 刑事告訴の可否（営業秘密侵害罪: 10年以下の懲役、2,000万円以下の罰金）
   - 弁護士への相談

3. **再発防止**
   - 漏洩原因の分析と対策の実施
   - 管理措置の見直し
   - 従業員への再教育

## 6. 営業秘密管理チェックリスト

### 6.1 初期設定時

- [ ] 秘密情報の洗い出しと分類（極秘/秘密/社内限定/公開可）
- [ ] 秘密表示の付与（ファイルヘッダー、ディレクトリ README）
- [ ] `.claudeignore` への秘密情報ファイル・ディレクトリの登録
- [ ] `deny` 設定による極秘情報へのアクセス禁止
- [ ] 秘密保持契約（NDA）の締結状況の確認
- [ ] 営業秘密管理規程の策定

### 6.2 定期レビュー（四半期ごと）

- [ ] 秘密情報の分類の見直し
- [ ] `.claudeignore`、`deny` 設定の棚卸し
- [ ] 秘密表示の付与漏れの確認
- [ ] アクセス権限の棚卸し
- [ ] NDA の有効期限の確認
- [ ] インシデントの有無の確認

### 6.3 人事異動・退職時

- [ ] 秘密情報へのアクセス権限の削除
- [ ] 秘密情報の返却・削除の確認
- [ ] 退職後の秘密保持義務の確認（NDA の存続条項）
- [ ] 競業避止義務の確認（該当する場合）
