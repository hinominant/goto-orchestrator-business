# LM Orchestrator 失敗パターン辞書

Claude Code 利用時に発生しうる失敗パターンを分類・整理し、予防と対応を標準化する。

> **運用方針:** 新規パターン検出時は本辞書に追記し、重複チェックを必ず行うこと。同一根本原因のパターンが既存であれば更新で対応する。

| ENV-P | 環境構築 | 環境差異・依存関係・構成管理 |
| QUA-P | 品質管理 | テスト・lint・デプロイ品質 |

## SEC-P001: .envファイルのGitコミット

**カテゴリ:** SEC（セキュリティ）
**重大度:** CRITICAL
**発生頻度:** 高
**出自:** [確定] Claude Code が `git add .` を実行し、.env がステージングされる事例が複数報告

### 症状
- `.env` ファイルがGitリポジトリにコミットされる
- APIキー、DBパスワード、トークン等のシークレットがGit履歴に永続化される
- リポジトリがpublicの場合、即座に外部からアクセス可能になる
- privateリポジトリでも、clone権限を持つ全員にシークレットが露出する

### 根本原因
1. `.gitignore` に `.env` が記載されていない（初期化漏れ）
2. Claude Code が `git add .` や `git add -A` を使用し、意図しないファイルをステージングする
3. `.env.example` と `.env` の区別が曖昧で、誤って実体ファイルをコミットする
4. サブディレクトリ内の `.env`（例: `services/api/.env`）が `.gitignore` のパターンから漏れる

### 検知トリガー
- PreToolUse Hook: `git add` コマンドの引数に `.env` を含むパターンを検出
- PostToolUse Hook: `git status` の出力に `.env` がステージング済みとして表示される
- Safety Gate: `git add .` または `git add -A` の実行自体を警告対象とする
- CI/CD: `git diff --cached --name-only` で `.env` パターンをチェック

### 予防策
1. `install.sh` がプロジェクト初期化時に `.gitignore` の `.env` 記載を自動チェック・追記する
2. CLAUDE.md に「`git add .` 禁止、ファイル名を明示的に指定」をルールとして記載
3. PreToolUse Hook で `git add` の引数をバリデーションし、`.env` を含む場合はブロック
4. `pre-commit` フックで `.env` パターンのステージングを検出・拒否
5. `.gitignore` テンプレートに以下を必ず含める:
   ```
   .env
   .env.*
   !.env.example
   ```

### 修復手順
1. **即時対応:** `git reset HEAD .env` でステージングから除外（未プッシュの場合）
2. **プッシュ済みの場合:**
   ```bash
   # Git履歴から.envを完全除去
   git filter-branch --force --index-filter \
     'git rm --cached --ignore-unmatch .env' \
     --prune-empty --tag-name-filter cat -- --all
   git push --force-with-lease --all
   ```
3. **シークレットローテーション:** `.env` に含まれていた全てのシークレットを無効化し、再発行する。これが最も重要なステップ
4. **監査:** GitHubの場合、Secret Scanning アラートを確認する
5. **再発防止:** `.gitignore` を修正し、pre-commit フックを追加する

### 関連フラグ
SEC-F002（.env コミット検出フラグ）

### 関連ADR
ADR-001（セキュリティを第一設計原則とする）

## SEC-P002: APIキーのハードコード

**カテゴリ:** SEC（セキュリティ）
**重大度:** CRITICAL
**発生頻度:** 高
**出自:** [確定] Claude Code がサンプルコード生成時にプレースホルダではなく実APIキーを埋め込む事例

### 症状
- ソースコード内にAPIキーやトークンが文字列リテラルとして直接記述される
- `const API_KEY = "sk-..."` のような記述がコミットされる
- コードレビューで発見されても、Git履歴に残存する

### 根本原因
1. Claude Code が環境変数参照ではなく直接値を記述するコードを生成する
2. 開発者がデバッグ時に一時的に埋め込んだキーを戻し忘れる
3. `.env` から読み込む設計が最初から行われていない
4. テストコード内に実際のAPIキーが記述される

### 検知トリガー
- PreToolUse Hook: ファイル書き込み時に以下の正規表現パターンでスキャン
  ```
  (sk-[a-zA-Z0-9]{20,})           # OpenAI
  (ghp_[a-zA-Z0-9]{36})           # GitHub PAT
  (xoxb-[0-9]{10,}-[a-zA-Z0-9]+)  # Slack Bot Token
  (AIza[0-9A-Za-z_-]{35})          # Google API Key
  (AKIA[0-9A-Z]{16})               # AWS Access Key
  ```
- PostToolUse Hook: 新規作成・編集されたファイル内容をパターンマッチ
- CI/CD: `gitleaks` や `truffleHog` による自動スキャン

### 予防策
1. CLAUDE.md にAPIキーの環境変数参照を必須とするルールを記載
2. PreToolUse Hook でファイル書き込み内容のシークレットパターンマッチを実装
3. プロジェクトテンプレートに `config.ts` / `config.py` 等の環境変数読み込み層を用意
4. `.env.example` にキー名のみ（値なし）を記載し、設定すべき変数を明示

### 修復手順
1. ソースコードからハードコードされたキーを環境変数参照に置換
2. Git履歴からの除去（SEC-P001の修復手順と同様）
3. 該当APIキーの無効化と再発行
4. pre-commit フックにシークレット検出ルールを追加

### 関連フラグ
SEC-F001（シークレット検出フラグ）

### 関連ADR
ADR-001（セキュリティを第一設計原則とする）

## SEC-P003: stdoutへのシークレット出力

**カテゴリ:** SEC（セキュリティ）
**重大度:** HIGH
**発生頻度:** 中
**出自:** [確定] デバッグ時に `echo $API_KEY` が実行され、Claude Code のログにシークレットが残存する事例

### 症状
- `echo`, `printf`, `console.log` 等でシークレットが標準出力に出力される
- Claude Code のセッションログにシークレットが平文で記録される
- CI/CDのビルドログにシークレットが露出する

### 根本原因
1. Claude Code がデバッグ目的で環境変数を表示するコマンドを実行する
2. エラーハンドリング時にシークレットを含むオブジェクトをそのままログ出力する
3. `env` や `printenv` コマンドで全環境変数が出力される

### 検知トリガー
- Safety Gate: 以下のパターンをブロック対象
  ```
  echo.*\$(.*KEY|.*TOKEN|.*SECRET|.*PASSWORD)
  echo.*\$\{.*KEY|.*TOKEN|.*SECRET|.*PASSWORD\}
  printenv
  env | grep -v
  ```
- PreToolUse Hook: Bash コマンドの引数解析でシークレット変数参照を検出

### 予防策
1. Safety Gate に `echo $SECRET_*` パターンのdenyルールを設定
2. PreToolUse Hook でシークレット変数名パターン（`*_KEY`, `*_TOKEN`, `*_SECRET`, `*_PASSWORD`）の出力コマンドをブロック
3. CLAUDE.md に「環境変数のデバッグ出力禁止」ルールを明記
4. アプリケーションログにシークレットマスキング機能を実装

### 修復手順
1. Claude Code セッションのログファイルを特定し、シークレット含有部分を確認
2. 該当シークレットのローテーション（出力されたものは漏洩済みとして扱う）
3. CI/CDログの場合、ログ保持ポリシーに従い削除依頼

### 関連フラグ
SEC-F004（シークレット stdout 出力フラグ）

### 関連ADR
ADR-001（セキュリティを第一設計原則とする）

## SEC-P004: curl/wgetでの認証情報送信

**カテゴリ:** SEC（セキュリティ）
**重大度:** CRITICAL
**発生頻度:** 低
**出自:** [推定] Claude Code が外部APIテスト時に認証ヘッダ付きでリクエストを送信するリスク

### 症状
- `curl -H "Authorization: Bearer $TOKEN" https://外部サービス` が実行される
- シークレットがHTTPリクエストとして外部に送信される
- 中間者攻撃やログ記録により認証情報が漏洩する
- 意図しないエンドポイントにシークレットが送信される

### 根本原因
1. Claude Code がAPI動作確認のためにcurlコマンドを生成・実行する
2. 外部サービスのURLが誤っており、意図しない宛先に送信される
3. HTTPSではなくHTTPで送信され、平文で認証情報が流れる

### 検知トリガー
- Safety Gate: `curl` / `wget` コマンドに以下を含む場合にブロック
  ```
  -H.*[Aa]uthorization
  -H.*Bearer
  -u.*:.*@
  --header.*[Tt]oken
  ```
- PreToolUse Hook: 外部URLへのリクエスト発行を検出
- denyルール: 許可されたドメインリスト以外へのリクエストを拒否

### 予防策
1. denyルールで `curl` / `wget` の認証ヘッダ付きリクエストをブロック
2. 許可ドメインリスト（allowlist）を `PROJECT_CONTEXT.md` に定義
3. テスト用のモックサーバーを用意し、外部APIへの直接リクエストを不要にする
4. CLAUDE.md に「外部APIへの直接リクエスト禁止」ルールを記載

### 修復手順
1. 送信先サービスのアクセスログを確認（可能な場合）
2. 送信された認証情報の即時ローテーション
3. 送信先が攻撃者管理の場合、インシデント対応プロセスを開始

### 関連フラグ
SEC-F003（認証情報外部送信フラグ）

### 関連ADR
ADR-002（多層防御アーキテクチャの採用）

## SEC-P005: MCP Elicitation インジェクション

**カテゴリ:** SEC
**重大度:** CRITICAL
**発生頻度:** 中
**出自:** [確定]（Unit 42: MCPサーバーの82%に脆弱性）

### 症状
MCPサーバーからの情報要求（Elicitation）に隠し指示が含まれており、
AI が知らずに認証情報の送信やコマンド実行を行う。

### 根本原因
MCPサーバーは Claude Code から信頼されているため、
Elicitation（情報要求）が無検査で処理される設計の隙間。

### 検知トリガー
- Elicitation テキストに「execute the following」「以下を実行」が含まれる
- Elicitation テキストに外部URL + send/transmit が含まれる
- Elicitation テキストに環境変数アクセス指示が含まれる
- base64 デコード後に curl/exec が含まれる

### 予防策
- elicitation-guard.js フックの導入
- MCPサーバーのソースコードを確認してからインストール
- 不要な MCP サーバーを削除

### 修復手順
1. 侵害されたMCPサーバーを無効化
2. 漏洩した認証情報を rotate
3. MCPサーバーのコードを精査

### 関連フラグ
ELI-F001

### 関連ADR
ADR-007（4-Hook体制（PreToolUse / PostToolUse / Elicitation / Stop））

## SEC-P006: ANTHROPIC_BASE_URL 書き換えによる API キー窃取

**カテゴリ:** SEC
**重大度:** CRITICAL
**発生頻度:** 低
**出自:** [確定]（CVE-2026-21852, CVSS 5.3、v2.0.65で修正）

### 症状
悪意ある設定ファイルが ANTHROPIC_BASE_URL を攻撃者エンドポイントに変更し、
API キーを含む全リクエストが外部に送信される。

### 根本原因
Claude Code が設定ファイルを読み込む際、環境変数オーバーライドの検証が
不十分だった（v2.0.65以前）。

### 検知トリガー
- コマンドに ANTHROPIC_BASE_URL の設定が含まれる
- .claude/settings.json 内に未知の API エンドポイントが含まれる
- 信頼できないリポジトリのクローン後に ANTHROPIC_BASE_URL が変更される

### 予防策
- Claude Code を v2.0.65 以上に更新
- Safety Gate での ANTHROPIC_BASE_URL 変更検知
- 信頼できないリポジトリを開かない

### 修復手順
1. API キーを即座に rotate
2. Anthropic のダッシュボードで使用履歴を確認
3. Claude Code を最新バージョンに更新

### 関連フラグ
SEC-F006（ANTHROPIC_BASE_URL 書き換えフラグ）

### 関連ADR
ADR-008（Safety Gate パターン（自動ブロック））

## SEC-P007: allow リスト経由のバイパス攻撃

**カテゴリ:** SEC
**重大度:** HIGH
**発生頻度:** 中
**出自:** [確定]（複数の実証報告）

### 症状
`python3 *` や `node *` などの広すぎる allow ルールを経由して、
deny されているネットワーク通信や機密ファイルアクセスが実行される。

### 根本原因
deny ルールはコマンドパターンで検知するが、
python3/node のようなインタープリタは任意のネットワーク操作を内包できる。

### 検知トリガー
- python3 -c "import urllib..." または node -e "require('https')..."
- cat コマンドが Bash allow に入っており .env ファイルに使用される
- grep/awk/sed で機密ファイルが引数になる

### 予防策
- allow は最小限のサブコマンドのみ（python3 --version 等）
- python3 *, node *, curl -s * などのワイルドカードを排除
- cat/head/tail を Bash allow から除外

### 修復手順
1. settings.json の allow ルールを精査・絞り込み
2. 広すぎる allow を削除し、必要なものだけ再追加
3. /permissions コマンドで設定確認

### 関連フラグ
SEC-F007（allow リストバイパス試行フラグ）

### 関連ADR
ADR-004（deny ルール最優先評価）

## SEC-P008: AI 生成コードへのシークレット出力埋め込み

**カテゴリ:** SEC
**重大度:** HIGH
**発生頻度:** 高
**出自:** [確定]（実証済み: NF0000 による Google Ads アカウント侵害事例）

### 症状
プロンプトインジェクション攻撃により、AI が生成するコードに
一見無害なデバッグコードとしてシークレット出力が埋め込まれる。
外部送信はないため、curl 検知をすり抜ける。

例:
```python
config = dotenv_values(".env")
for key, value in config.items():
    print(f"  {key} = {value}")  # 全シークレットが stdout に流出
```

### 根本原因
curl/wget による外部送信は検知されるが、
print/console.log/stdout への出力は検知されにくい。
CI/CD ログ、共有ターミナル、ペアプログラミング画面で漏洩する。

### 検知トリガー
- dotenv/load_dotenv + print/console.log の組み合わせ
- for key, value in config.items() の後に print が続く
- process.env のループ出力

### 予防策
- Safety Gate での stdout シークレット出力検知
- AI 生成コードの人的レビュー（print/log 出力箇所の確認）
- CLAUDE.md にデバッグコードでもシークレット出力禁止を明記

### 修復手順
1. 問題のあるコードを削除
2. CI/CD ログを確認し、漏洩した認証情報を rotate
3. ログ収集サービス（Datadog等）に漏洩していないか確認

### 関連フラグ
SEC-F004（シークレット stdout 出力フラグ）

### 関連ADR
ADR-008（Safety Gate パターン（自動ブロック））

## DES-P001: rm -rfによる重要ファイル削除

**カテゴリ:** DES（破壊的操作）
**重大度:** CRITICAL
**発生頻度:** 中
**出自:** [確定] Claude Code が不要ファイル削除時に広範囲のパスを `rm -rf` で削除する事例

### 症状
- プロジェクトディレクトリ、設定ファイル、データファイルが不可逆に削除される
- `rm -rf /` や `rm -rf ~` 等の致命的なコマンドが実行される
- Git管理外のファイル（ローカル設定、未コミットの作業）が失われる

### 根本原因
1. Claude Code がディレクトリクリーンアップ時に過剰な範囲を削除対象にする
2. 変数展開の失敗で意図しないパスが削除される（例: `rm -rf $EMPTY_VAR/`）
3. 相対パスの解決ミスで作業ディレクトリ外のファイルが削除される

### 検知トリガー
- denyルール: 以下のパターンを完全ブロック
  ```
  rm -rf /
  rm -rf ~
  rm -rf /*
  rm -rf $HOME
  ```
- PreToolUse Hook: `rm` コマンドの `-rf` フラグ使用を検出し、対象パスを検証
- 確認プロンプト: `rm -rf` は常にユーザー確認を要求

### 予防策
1. denyルールで危険な `rm -rf` パターンを完全ブロック
2. PreToolUse Hook で `rm` コマンド実行前にユーザー確認を強制
3. CLAUDE.md に「`rm -rf` はプロジェクトルート配下の特定パスのみ許可」ルールを記載
4. 重要ファイルのバックアップ戦略を定義

### 修復手順
1. **Git管理下のファイル:** `git checkout -- <path>` または `git restore <path>` で復元
2. **Git管理外のファイル:** Time Machine等のバックアップから復元
3. **復元不可能な場合:** 影響範囲を特定し、再作成の優先順位を決定
4. **事後分析:** 削除されたファイルの一覧を `git status` で確認し、漏れなく復元

### 関連フラグ
DES-F001（破壊的削除検出フラグ）

### 関連ADR
ADR-003（Hub-spoke パターンの採用）

## DES-P002: git push --forceによる履歴破壊

**カテゴリ:** DES（破壊的操作）
**重大度:** CRITICAL
**発生頻度:** 低
**出自:** [確定] rebase後のforce pushで他メンバーのコミットが消失する事例

### 症状

## ENV-P002: 依存関係の不整合

**カテゴリ:** ENV（環境構築）
**重大度:** MEDIUM
**発生頻度:** 高
**出自:** [確定] lockファイルの更新漏れ、または `npm install` と `npm ci` の挙動差異による問題

### 症状
- `npm ci` / `pip install -r requirements.txt` がエラーで失敗する
- ローカルでは動くがクリーンインストールで失敗する
- 依存パッケージのバージョン不整合でランタイムエラーが発生する
- `package.json` と `package-lock.json` の内容が矛盾する

### 根本原因
1. `package.json` を手動編集後に `npm install` を実行せずlockファイルが未更新
2. Claude Code が `npm install <pkg>` を実行せず `package.json` を直接編集する
3. 異なるNode.js/npmバージョンでlockファイルが生成され、フォーマットが不一致
4. `requirements.txt` にバージョン指定がなく、環境ごとに異なるバージョンが入る

### 検知トリガー
- CI/CD: `npm ci` / `pip install` の失敗
- PostToolUse Hook: `package.json` 変更時にlockファイルの同期チェック
- `npm ls` でpeerDependency警告を検出

### 予防策
1. パッケージ追加は必ずコマンド経由（`npm install <pkg>`, `pip install <pkg>`）で行う
2. `package.json` の直接編集後は `npm install` を必ず実行
3. lockファイルを `.gitignore` に入れない（必ずコミットする）
4. CIで `npm ci`（`package-lock.json` 厳密モード）を使用
5. Python: `pip freeze > requirements.txt` でバージョンを固定

### 修復手順
1. `rm -rf node_modules package-lock.json && npm install` でlockファイルを再生成
2. Python: `pip install -r requirements.txt` でバージョン固定を確認
3. 再生成したlockファイルをコミット
4. CIで正常に動作することを確認

### 関連フラグ
QUA-F001（テストスキップフラグ）

### 関連ADR
ADR-010（フラグシステムと自動修復の統合）

## QUA-P001: テストなしデプロイ

**カテゴリ:** QUA（品質管理）
**重大度:** HIGH
**発生頻度:** 中
**出自:** [確定] 急ぎの修正でテストをスキップしてデプロイし、本番障害が発生する事例

### 症状
- テスト未実行の状態でコードがデプロイされる
- 本番環境でランタイムエラーが発生する
- ユーザー影響のあるバグがリリースされる
- ロールバックが必要になる

### 根本原因
1. CI/CDパイプラインにテスト実行ステップがない
2. テストステップが `allow_failure: true` で失敗を許容している
3. 手動デプロイプロセスでテストがスキップされる
4. `--no-verify` フラグでpre-pushフックがスキップされる
5. QUA-F001フラグ（テスト必須）が設定されていない

### 検知トリガー
- QUA-F001フラグ: テスト実行結果の確認
- CI/CD: テストステップの成功を必須条件とする
- /quality-gate スキル: デプロイ前チェック

### 予防策
1. CI/CDパイプラインでテスト成功をデプロイの必須条件にする
2. `main` / `master` へのマージにテスト通過を必須とするブランチ保護ルール
3. `/quality-gate` スキルをデプロイ前に必ず実行
4. CLAUDE.md に「テスト未実行でのデプロイ禁止」を明記
5. デプロイスクリプトにテスト実行を組み込む

### 修復手順
1. **即時対応:** 本番障害の場合、前バージョンにロールバック
2. **原因特定:** テストを実行し、失敗するテストケースを特定
3. **修正:** バグ修正→テスト通過→再デプロイ
4. **再発防止:** CI/CDゲートの追加・強化

### 関連フラグ
QUA-F001（テスト必須フラグ）

### 関連ADR
ADR-011（Sandbox + Permissions + Hooks の三重防御をデフォルト推奨）

## QUA-P002: lint違反の蓄積

**カテゴリ:** QUA（品質管理）
**重大度:** LOW
**発生頻度:** 高
**出自:** [確定] lint警告が放置され続け、数百件に達してから対応が困難になる事例

### 症状
- lintエラー・警告が大量に蓄積される
- 新規の重要なlintエラーがノイズに埋もれて見逃される
- コードの可読性・保守性が低下する
- チーム内のコードスタイルが不統一になる

### 根本原因
1. lint実行がCI/CDに組み込まれていない
2. lint警告を「後で直す」として放置する文化
3. lintルールが厳しすぎて全対応が非現実的
4. 自動修正可能なルールが手動修正として放置される
5. QUA-F003フラグ（lint必須）が設定されていない

### 検知トリガー
- QUA-F003フラグ: lint実行結果の確認
- CI/CD: lintステップの実行
- AR-L1エージェント: 自動修正の実行状況

### 予防策
1. AR-L1エージェントによる自動修正を有効化（`eslint --fix`, `prettier --write`）
2. CI/CDでlintエラーゼロをマージ条件にする
3. pre-commitフックでlint自動実行
4. 既存のlint違反は一括修正PRで対応し、ベースラインをゼロにする
5. lintルールの定期レビュー（不要なルールの無効化）

### 修復手順
1. `npx eslint --fix .` / `npx prettier --write .` で自動修正可能な違反を一括修正
2. 自動修正不可能な違反をリスト化し、優先順位をつけて対応
3. 修正後のlintエラーゼロを確認
4. CI/CDゲートを追加して今後の蓄積を防止

### 関連フラグ
QUA-F003（lint必須フラグ）

### 関連ADR
ADR-011（Sandbox + Permissions + Hooks の三重防御をデフォルト推奨）
