## ADR-001: セキュリティを第一設計原則とする

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-002, ADR-007, ADR-008, ADR-011

### 背景（Context）

Claude Code はファイル操作・コマンド実行・Git 操作・外部通信など、開発環境に対して非常に強力なツール実行能力を持つ。この能力は熟練者にとって生産性の源泉だが、初心者が意図せず破壊的操作（`rm -rf /`、`git push --force`、本番 DB への `DROP TABLE`、認証情報のログ露出）を実行してしまうリスクが現実的に存在する。

フレームワークの設計原則として「何を最上位に置くか」を決める必要があった。候補はパフォーマンス・機能の豊富さ・セキュリティの3つ。

### 決定（Decision）

セキュリティを最上位の設計原則（Core Principle #1）に据える。全コンポーネントの設計において、セキュリティ要件を機能要件・パフォーマンス要件より優先する。具体的には:

- Nexus のタスク実行前に Phase 0: RISK_ASSESSMENT を必須化
- 全エージェント共通の GUARDRAIL プロトコル（L1-L4）を適用
- Tool Risk Hooks による実行前リスク分類を提供
- シークレット保護（`.env` 自動 `.gitignore` 追加、コード内ハードコード検出）を組み込み
- 破壊的コマンドの日本語警告表示を標準化

### 根拠（Rationale）

1. **ターゲットユーザー**: 初心者エンジニアが安心して Claude Code を使えることがフレームワークの存在意義。安全でなければ「使いやすい」は無意味
2. **非可逆性**: セキュリティ事故（データ削除、認証情報漏洩）は発生後の回復が困難または不可能。パフォーマンス低下は後から改善できる
3. **信頼構築**: 「このフレームワークを使えば安全」という信頼がなければ採用されない。Security-First は差別化ポイントでもある
4. **設計の一貫性**: 最上位原則が明確であれば、個別の設計判断で迷いが減る

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **パフォーマンス優先** | 速度を重視するとセキュリティチェックがスキップされがち。「速いが危ない」フレームワークは初心者に推奨できない |
| **機能性優先** | 機能の豊富さを追求すると攻撃面が広がる。多機能だが安全でないツールは事故を招く |
| **セキュリティをオプション扱い** | オプションは無効化される。デフォルトで安全でなければ、ほとんどのユーザーは設定しない |

### 影響（Consequences）

**正の影響:**
- 初心者が致命的な操作を実行するリスクが大幅に低減
- フレームワーク全体の設計に一貫性が生まれる
- 「安全に使える」という明確な差別化ポイント

**負の影響:**
- 初回セットアップが若干複雑化（`--with-hooks` オプションの理解が必要）
- 熟練者にとっては確認ダイアログが煩わしい場合がある（ただし段階的に緩和可能）
- セキュリティチェックによるわずかなレイテンシ増加

### 更新履歴
- 2026-03-19: 初版

## ADR-002: 多層防御アーキテクチャの採用

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-001, ADR-004, ADR-007, ADR-008, ADR-011

### 背景（Context）

ADR-001 でセキュリティを第一原則と定めたが、単一のセキュリティ機構（例: Hooks だけ、Permissions だけ）では、その機構が迂回・突破された場合に防御線がゼロになる。実際のセキュリティインシデントの多くは「単一防御の突破」が原因。

Claude Code が提供するセキュリティ機構（Sandbox、Permissions、Hooks）と、フレームワーク側で実装する機構（CLAUDE.md のルール、Guardrails）をどう組み合わせるかを決める必要があった。

### 決定（Decision）

5層の多層防御（Defense in Depth）アーキテクチャを採用する。各層が独立して機能し、1層が突破されても次の層で防御する。

```
Layer 1: CLAUDE.md / _framework.md（ルール・ポリシー定義）
  ↓ ルールを破ろうとした場合
Layer 2: Sandbox（ファイルシステム・ネットワークの物理的制限）
  ↓ Sandbox外の操作を試みた場合
Layer 3: Permissions（allow/deny リストによるツール単位の制御）
  ↓ 許可されたツールでの危険な操作の場合
Layer 4: Hooks（PreToolUse / PostToolUse / Stop の3フェーズ制御）
  ↓ Hook を通過した操作の品質検証
Layer 5: Guardrails（L1-L4 の段階的品質・安全チェック + 自動回復）
```

### 根拠（Rationale）

1. **独立性**: 各層は異なる機構で動作するため、1つの脆弱性が全層に波及しない
2. **段階的採用**: ユーザーのスキルレベルに応じて層を追加・削減できる（初心者は全層、熟練者は一部緩和）
3. **実績**: 多層防御はネットワークセキュリティ・物理セキュリティで実績のある設計パターン
4. **Claude Code の機能との整合**: Claude Code が提供する Sandbox / Permissions / Hooks の3機能に、フレームワーク側の CLAUDE.md と Guardrails を加えることで、プラットフォーム機能を最大限活用

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **Hooks のみに依存** | Hook の実装にバグがあった場合、防御がゼロになる。Node.js スクリプトの信頼性だけに全体のセキュリティを依存させるのは危険 |
| **Permissions のみに依存** | Permissions は allow/deny リストの静的定義。コマンドの引数レベルの危険性（例: `bash` は許可するが `rm -rf /` は禁止）を制御できない |
| **CLAUDE.md のルールのみ** | LLM はルールを完全に遵守する保証がない。プロンプトインジェクションや長いコンテキストでのルール忘却のリスクがある |
| **3層構成（Sandbox + Permissions + Hooks）** | 検討したが、CLAUDE.md によるポリシー定義層と Guardrails による品質検証層を加えることで、「防御」と「検証」の両面をカバーできる |

### 影響（Consequences）

**正の影響:**
- 各層が独立して機能するため、1層の突破で全体が崩れない
- ユーザーのスキルレベルに応じた柔軟な構成が可能
- 層ごとにデバッグ・改善できるため保守性が高い

**負の影響:**
- 層間の設定の一貫性維持が必要（例: Permissions で許可した操作を Hook でブロックする矛盾）
- 全層を理解するための学習コスト
- 設定ファイルが複数に分散する（CLAUDE.md, settings.json, hooks/*.js, _common/GUARDRAIL.md）

## ADR-003: Hub-spoke パターンの採用

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-005

### 背景（Context）

67エージェントが存在するフレームワークで、エージェント間の通信パターンを定義する必要があった。67体が自由に相互通信すると、通信経路は理論上 67 x 66 / 2 = 2,211 本になり、デバッグ・監視・制御が事実上不可能になる。

通信パターンの選択肢は P2P（全対全）、Hub-spoke（中央集権）、メッセージキュー（非同期）の3つ。

### 決定（Decision）

全エージェント間通信を Nexus（統括オーケストレーター）および Rally（並列オーケストレーター）経由に限定する Hub-spoke パターンを採用する。

```
          Nexus (Sequential)
         / | | | \
       A1  A2 A3 A4 A5

          Rally (Parallel)
         / | | | \
       T1  T2 T3 T4 T5
```

- **Sequential チェーン**: Nexus が Agent1 → Agent2 → Agent3 の順に実行（ロールシミュレーション）
- **Parallel チェーン**: Rally が TeamCreate/Task API で複数 Claude インスタンスを並列管理
- **エージェント間の直接通信は禁止**: 必ず Nexus または Rally を経由する

### 根拠（Rationale）

1. **追跡可能性**: 全通信が中央ハブを通るため、ログ・監視・デバッグが容易
2. **制御性**: Nexus/Rally がボトルネックにはなるが、同時にゲートキーパーとして機能。不正な呼び出しやリソース過剰消費を検知・制御できる
3. **シンプルさ**: エージェント開発者は「Nexus に結果を返す」ことだけ考えればよい。他エージェントの API を知る必要がない
4. **File ownership**: 並列実行時のファイルオーナーシップ管理が Hub-spoke でないと困難（PARALLEL プロトコル参照）

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **P2P 通信** | 2,211本の通信経路の管理が不可能。循環呼び出し・デッドロックのリスク。エージェント追加時の影響が全体に波及 |
| **メッセージキュー（Kafka/RabbitMQ 的パターン）** | Claude Code のエージェント実行モデル（同一プロセス内のロールシミュレーションまたは Task API）に合わない。外部インフラ依存が発生し、オフライン動作ができない |
| **階層型（Nexus → サブオーケストレーター → エージェント）** | 67体の規模では過剰。サブオーケストレーターの責務定義が曖昧になり、管理コストが増える |

### 影響（Consequences）

**正の影響:**
- 通信フローの完全な可視化・追跡が可能
- ファイルオーナーシップの一元管理
- エージェント追加時の影響が Nexus の設定変更に限定される
- チェーンテンプレート（Scout → Builder → Radar 等）の定義が直感的

**負の影響:**
- Nexus/Rally がシングルポイントオブフェイルアになる可能性（ただし Claude Code のセッション管理で軽減）
- 大量の並列タスクでボトルネックになりうる（Rally の TeamCreate/Task API の並列数に依存）
- エージェント間の密結合なコラボレーション（例: リアルタイムのペアプログラミング）には不向き

- 2026-03-19: 初版

## ADR-004: deny ルール最優先評価

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-001, ADR-002, ADR-008

### 背景（Context）

Claude Code の Permissions 機構で allow ルールと deny ルールが競合した場合の評価順序を定義する必要があった。例えば、以下のような設定が共存する場合:

```json
{
  "allow": ["Bash(git *)"],
  "deny": ["Bash(git push --force*)"]
}
```

`git push --force` は allow（`git *` にマッチ）にも deny にもマッチする。この場合にどちらを優先するかで、システムの安全性が決定的に変わる。

### 決定（Decision）

評価順序を **deny → ask → allow** の3段階とし、deny が常に最優先で評価されることを定める。

```
操作リクエスト
  ↓
[1] deny リストにマッチするか？ → YES → ブロック（理由表示）
  ↓ NO
[2] ask リストにマッチするか？ → YES → ユーザーに確認ダイアログ
  ↓ NO
[3] allow リストにマッチするか？ → YES → 実行許可
  ↓ NO
[4] デフォルト動作 → ask（未定義の操作は確認を求める）
```

### 根拠（Rationale）

1. **安全側フォールバック（Fail-safe）**: セキュリティにおける鉄則は「迷ったらブロック」。deny 優先により、設定ミスがあっても危険な操作は実行されない
2. **明示的な意図**: deny に記載された操作は「絶対に実行してはならない」という強い意思の表明。allow の広いパターンマッチで偶然上書きされるべきではない
3. **ファイアウォールの慣例**: ネットワークファイアウォールの deny-first 評価と同じ考え方。セキュリティ業界の標準プラクティス
4. **デバッグ容易性**: 「なぜブロックされたか」が明確（deny リストのどのルールにマッチしたかを表示）

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **allow 優先** | allow の広いパターン（`Bash(git *)`）が deny の狭いパターン（`Bash(git push --force*)`）を上書きしてしまう。意図せず危険な操作が許可されるリスクが高い |
| **最後のマッチ優先** | 設定ファイルの記述順序に依存するため、設定の追加・変更時に予期しない動作変更が発生しやすい |
| **最も具体的なルール優先** | 「具体性」の定義が曖昧。パターンの長さ？ワイルドカードの数？実装の複雑性が増す割にメリットが薄い |

### 影響（Consequences）

**正の影響:**
- 安全側に倒れる設計により、設定ミスによる事故を防止
- deny リストの追加だけで確実にブロックできる安心感
- ルール評価のデバッグが容易（deny → ask → allow の順に確認すればよい）

**負の影響:**
- 正当な操作が deny ルールに意図せずマッチしてブロックされる可能性（false positive）
- false positive 発生時は deny ルールの修正または allow ルールの追加で対応が必要（ワンステップの手間）
- 新しいツールや操作を追加する際、deny リストとの競合を確認する必要がある

### 更新履歴

## ADR-005: 67エージェント体制（CEOエージェント除外）

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-003

### 背景（Context）

Goto Orchestrator の原型である agent-orchestrator（オーケストラシステム）は68エージェント体制で、CEO / Analyst / Auditor を含む68体が存在した。CEO エージェントは特定企業の経営判断・戦略立案を担当するものであり、汎用フレームワークとしての配布には適さなかった。

Analyst と Auditor は汎用的な用途（データ分析、品質監査）にも使えるが、CEO は企業固有の知識・判断基準に強く依存していた。

### 決定（Decision）

- **CEO エージェントを除外**し、67エージェント体制とする（68 - CEO = 67）
- **Analyst** を汎用データ分析エージェントとして再定義し残す（Redash API 連携、指標定義、誤読防止）
- **Auditor** を汎用品質監査エージェントとして再定義し残す（プロセス準拠、SPEC 準拠監査）
- 企業固有のエージェントが必要な場合は、各プロジェクト側で `agents/` に追加する拡張ポイントを提供

### 根拠（Rationale）

1. **汎用性**: CEO エージェントは特定企業の戦略文脈に依存しており、他プロジェクトで利用できない。フレームワークとしての汎用性を損なう
2. **責務の明確化**: 「経営判断」はフレームワークの責務範囲外。フレームワークは「開発作業の安全な自動化」に集中すべき
3. **Analyst/Auditor の汎用価値**: データ分析と品質監査はどのプロジェクトでも必要。Redash 連携を外部 API 連携の汎用例として、SPEC 準拠監査をプロセス品質の汎用例として残す価値がある
4. **拡張性の確保**: 除外ではなく「プロジェクト側で追加可能」とすることで、企業固有エージェントのニーズにも対応

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **全68体制維持** | CEO エージェントが特定企業依存であり、OSS / 汎用フレームワークとして配布する際にノイズになる。使わないエージェントがあると「何のためにあるのか」という混乱を招く |
| **CEO を汎用化して残す** | 「汎用 CEO」の責務定義が曖昧になる。Bridge（ビジネス⇔技術翻訳）や Magi（3視点意思決定）が既にその領域をカバーしており、重複する |
| **Analyst/Auditor も除外して62体制** | Analyst と Auditor は汎用的な価値があり、除外するメリットがない。データ分析と品質監査は開発フレームワークの基本機能 |

### 影響（Consequences）

**正の影響:**
- 特定企業依存がなくなり、どのプロジェクトでも利用可能な汎用フレームワークに
- エージェント一覧が明確で、各エージェントの存在理由が説明可能
- `install.sh` の `ALL_AGENTS` リストが汎用的な構成に

**負の影響:**
- 元の agent-orchestrator から移行するユーザーは CEO エージェントを自分で再構築する必要がある
- 「67」という数は覚えにくいかもしれないが、機能的な問題ではない

### 更新履歴
- 2026-03-19: 初版

## ADR-006: install.sh によるレジストリパターン

**Status:** 採用
**Date:** 2026-03-19
**Related:** なし

### 背景（Context）

67エージェント + 27+プロトコル + 7コマンド + 7スキル + Hook 実装 + 設定テンプレートという大量のファイル群を、各プロジェクトにどう配布するかを決める必要があった。

配布方法の要件:
- プロジェクトごとに必要なエージェントだけ選択できること
- バージョン固定が容易なこと
- オフライン環境でも動作すること
- 依存関係（npm, pip 等）を最小限にすること
- Claude Code の `.claude/agents/` ディレクトリ構造と互換性があること

### 決定（Decision）

中央リポジトリ（`hinominant/goto-orchestrator`）を「レジストリ」として扱い、`install.sh` でプロジェクトごとにファイルをコピーする方式を採用する。

```bash
# リモートから（curl ワンライナー）
curl -sL https://raw.githubusercontent.com/hinominant/goto-orchestrator/main/install.sh | bash -s -- --with-hooks

# ローカルクローンから
git clone https://github.com/hinominant/goto-orchestrator.git /tmp/goto-orchestrator
cd your-project && /tmp/goto-orchestrator/install.sh --with-hooks

# 選択インストール
./install.sh --with-hooks nexus rally builder radar
```

install.sh は12ステップで以下を実行:
1. エージェント定義のコピー（SKILL.md + references/）
2. カスタムコマンドのコピー
3. フレームワークプロトコルのコピー
4. 共通プロトコルのコピー
5. スキルのコピー
6. 共有知識ファイルの初期化
7. PROJECT_CONTEXT.md の初期化
8. MCP スクリプト・テンプレートのコピー
9. CLAUDE.md へのフレームワーク参照追記
10. MCP セットアップ（`--with-mcp` 時）
11. Permissions セットアップ（`--with-permissions` 時）
12. Hooks セットアップ（`--with-hooks` 時）

### 根拠（Rationale）

1. **ゼロ依存**: bash と git のみで動作。Node.js / Python / パッケージマネージャ不要
2. **コピーの独立性**: インストール後はレジストリとの接続が不要。ファイルはプロジェクトにコピーされるため、レジストリの変更がプロジェクトに影響しない
3. **バージョン固定**: コピーした時点のバージョンが固定される。意図しないアップデートが発生しない
4. **選択インストール**: エージェント名を引数に渡すことで、必要なエージェントだけインストール可能
5. **オフライン対応**: ローカルクローンから実行すれば、ネットワーク不要
6. **Claude Code の規約準拠**: `.claude/agents/` にフラットファイルとして配置するため、Claude Code のエージェント検出機構とそのまま互換

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **npm パッケージ** | Node.js 依存。非 Node.js プロジェクト（Python, Go, Rust 等）での利用にハードルがある。`node_modules` にエージェント定義が埋もれ、Claude Code が検出しにくい |
| **git submodule** | サブモジュールの更新管理が煩雑。`.claude/agents/` にシンボリックリンクを張る必要があり、プラットフォーム依存の問題が発生しやすい。選択インストールが困難 |
| **monorepo（各プロジェクトに直接配置）** | プロジェクト間でのエージェント定義の同期が手動になる。67エージェント分の変更を各プロジェクトに反映する運用コストが高い |
| **GitHub Actions / CI でのインストール** | CI 環境では動作するが、ローカル開発環境への配布が別途必要。二重管理になる |

### 影響（Consequences）

**正の影響:**
- curl ワンライナーで即座にセットアップ可能
- 依存関係なし。どの技術スタックのプロジェクトでも利用可能
- バージョン固定が自然に実現される
- 選択インストールにより、軽量な構成も可能

**負の影響:**
- バージョンアップが自動化されない（再実行が必要）
- プロジェクト間でのエージェント定義の差分管理はユーザー責任
- install.sh 自体のバグが全ユーザーに影響する（ただし冪等性を確保し、既存ファイルはスキップする設計で軽減）

### 更新履歴
- 2026-03-19: 初版

## ADR-007: 4-Hook体制（PreToolUse / PostToolUse / Elicitation / Stop）

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-001, ADR-002, ADR-008

### 背景（Context）

Claude Code のツール実行ライフサイクル全体を安全に管理するために、Hook の配置戦略を決める必要があった。Claude Code は Hook の挿入ポイントとして PreToolUse / PostToolUse / Elicitation / Stop の4つを提供している。

問題は「どの Hook をどの目的で使うか」「全部使うべきか、一部で十分か」という設計判断。

### 決定（Decision）

4つの Hook 全てを活用する **4-Hook 体制** を採用し、ツール実行の完全なライフサイクル管理を実現する。

| Hook | 実装ファイル | 目的 |
|------|-------------|------|
| **PreToolUse** | `tool-risk.js` | 実行前リスク分類。HIGH/BLOCK レベルの操作を事前警告・ブロック |
| **PostToolUse** | `post-tool-use.js` | 実行後ログ記録。操作の監査証跡を残す |
| **Elicitation** | `elicitation-guard.js` | MCP Elicitation インジェクション検出・ブロック |
| **Stop** | `stop-hook.js` | セッション終了時のサマリ永続化。メモリへの学習内容保存 |

各 Hook の責務:

**PreToolUse（tool-risk.js）:**
- ツール名とパラメータからリスクレベルを自動分類（LOW / MEDIUM / HIGH / BLOCK）
- HIGH: 確認ダイアログを表示（日本語で操作内容と危険性を説明）
- BLOCK: 自動ブロック（Safety Gate パターン、ADR-008 参照）
- パターンマッチ対象: `rm -rf`, `git push --force`, `DROP TABLE`, 認証情報の外部送信、`npm publish` 等

**PostToolUse（post-tool-use.js）:**
- 実行されたツール・パラメータ・結果の概要をログに記録
- シークレット漏洩の事後検出（stdout に API キー等が含まれていないか）

**Elicitation（elicitation-guard.js）:**
- MCP サーバーからの Elicitation（情報要求）を監視し、プロンプトインジェクション攻撃を検知・ブロック
- 攻撃パターン: MCPサーバーが正当な情報要求を装い、隠しコマンド実行指示を埋め込む（逆方向インジェクション）

**Stop（stop-hook.js）:**
- セッションで学習した内容のメモリ永続化
- 実行サマリ（成功/失敗/ブロック数）の出力

### 根拠（Rationale）

1. **完全なライフサイクル**: 実行前（防御）→ 実行後（記録）→ MCP インジェクション防御（Elicitation）→ 終了時（学習）の4フェーズで隙間なく管理
2. **責務の分離**: 各 Hook が単一責務。デバッグ・修正・無効化が独立して可能
3. **監査可能性**: PostToolUse のログにより、「何がいつ実行されたか」の完全な監査証跡を確保
4. **学習ループ**: Stop Hook でのメモリ永続化により、セッション間で学習が蓄積される
5. **MCP攻撃対策**: Elicitation Hook により、MCP エコシステム拡大に伴うインジェクションリスクを体系的に防御

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **PreToolUse のみ** | 実行前の防御はできるが、実行後の異常検知（シークレット漏洩等）ができない。セッション終了時の学習永続化も失われる |
| **カスタムミドルウェア（独自実装）** | Claude Code の Hook 機構を使わず独自のミドルウェアを実装する案。Claude Code のアップデートとの互換性維持が困難。車輪の再発明 |
| **Elicitation Hook なしの3-Hook** | MCP サーバーを使うプロジェクトでのインジェクション攻撃を防げない。MCP エコシステムの拡大により Elicitation ベースの攻撃リスクは増大している |

### 影響（Consequences）

**正の影響:**
- ツール実行の完全なライフサイクル管理
- 実行前の防御 + 実行後の検証 + MCP インジェクション防御 + 終了時の学習という4段階のフィードバックループ
- 各 Hook が独立しているため、段階的な導入が可能（まず PreToolUse だけ、慣れたら全4 Hook）

**負の影響:**
- 全ツール実行に Hook 処理のオーバーヘッドが加わる（ただし Node.js で軽量に実装）
- Hook スクリプトのメンテナンスコスト（4ファイル）
- Hook 間の状態共有が必要な場合の設計複雑性（現状はファイルベースで共有）

### 更新履歴
- 2026-03-19: 初版
- 2026-03-20: Elicitation Hook 追加により4-Hook体制に更新

## ADR-008: Safety Gate パターン（自動ブロック）

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-001, ADR-004, ADR-007

### 背景（Context）

ADR-007 の PreToolUse Hook で HIGH レベルの操作に確認ダイアログを表示する設計を採用したが、一部の操作は確認ダイアログすら表示せず即座にブロックすべきものがある。

例えば:
- `rm -rf /`（ルートディレクトリの削除）
- 認証情報を含むファイルの外部送信
- 本番データベースへの `DROP DATABASE`

これらは「確認して実行」の選択肢を与えること自体がリスク。ユーザーが誤って「Y」を押す可能性がある。

### 決定（Decision）

HIGH レベルの上位に **BLOCK レベル** を設け、Safety Gate パターンに該当する操作は確認ダイアログなしで自動ブロックする。

```
リスクレベル判定:
  LOW    → サイレント通過（git status, Read, Grep 等）
  MEDIUM → 操作説明を表示（git push, npm publish, ファイル編集等）
  HIGH   → 確認ダイアログ表示（rm -rf ./dist, DROP TABLE users 等）
  BLOCK  → 自動ブロック（以下の Safety Gate パターン）
```

**Safety Gate パターン（BLOCK 対象）:**

| パターン | 理由 |
|---------|------|
| `rm -rf /` または `rm -rf ~` | システム/ホームディレクトリの全削除 |
| `chmod -R 777 /` | システム全体のパーミッション破壊 |
| `:(){:\|:&};:` (fork bomb) | システムリソースの枯渇 |
| 認証情報のHTTP送信（curl/wget + API_KEY/TOKEN/SECRET） | シークレットの外部漏洩 |
| `DROP DATABASE` | データベースの完全削除 |
| `git push --force` to main/master | メインブランチの履歴破壊 |
| `> /dev/sda` 等のデバイス直接書き込み | ストレージの破壊 |

### 根拠（Rationale）

1. **ゼロトレランス**: 致命的な操作に対して「実行する余地」を残さない。確認ダイアログは人間のミスを防げない
2. **初心者保護**: Claude Code の出力を無条件に信頼して「Y」を押す初心者を保護
3. **不可逆性**: Safety Gate パターンの操作は全て不可逆または回復コストが極めて高い
4. **パターンベースの拡張性**: 新しい危険パターンの追加が容易（tool-risk.js にパターンを追加するだけ）

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **全操作を確認ダイアログで対応** | 致命的操作に対して「Y/N」の選択肢を与えること自体がリスク。ダイアログ疲れで無意識に Y を押す「確認バイアス」も問題 |
| **全操作を allow（制限なし）** | 論外。フレームワークの存在意義に反する |
| **ホワイトリスト方式（許可された操作のみ実行）** | 柔軟性が著しく低下。新しいツールや操作を使うたびにホワイトリストの更新が必要で、開発体験が悪化 |

### 影響（Consequences）

**正の影響:**
- 致命的な操作を確実に防止。ユーザーの判断ミスに依存しない
- 「このフレームワークを使っていれば安全」という強い信頼を構築
- パターンの追加・修正が容易

**負の影響:**
- false positive（正当な操作のブロック）発生時、ユーザーはパターンを修正する必要がある
- BLOCK された操作の代替手段をユーザーが自分で見つける必要がある（ただし Hook が代替手段を提案する設計）
- パターンマッチの精度に依存（正規表現のエッジケース）

### 更新履歴
- 2026-03-19: 初版

## ADR-009: PROJECT_CONTEXT.md による汎用ビジネス文脈

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-005

### 背景（Context）

エージェントが技術的に正しいコードを生成しても、ビジネス文脈を理解していなければ適切な判断ができない。例えば:

- 「本番 DB に直接アクセスしてデータを修正」が許されるか否かはプロジェクトのセキュリティポリシー次第
- 「このフィールドを nullable にする」がビジネスルール上許容されるかはドメイン知識次第
- 「新機能の優先度」はビジネス目標次第

ビジネス文脈をエージェントに伝える仕組みが必要だった。

### 決定（Decision）

`.agents/PROJECT_CONTEXT.md` として汎用テンプレートを提供し、プロジェクトごとにカスタマイズする方式を採用する。

テンプレートの構造（8セクション）:

```markdown
1. Mission Statement     - プロジェクトの目的（1文）
2. Core Values          - 判断基準となる価値観（優先順位付き）
3. Target Users         - ユーザーセグメント・特徴・ニーズ
4. Technical Constraints - 言語/FW・インフラ・DB・外部API
5. Security Policy      - 本番DBアクセス・シークレット管理・外部通信・デプロイ権限
6. Business Rules       - プロダクト固有のビジネスルール
7. Out of Scope         - エージェントが手を出してはいけない領域
8. External Systems     - 外部連携システム・用途・アクセス方法
```

### 根拠（Rationale）

1. **構造化による品質**: 自由記述ではなく8セクションの構造を提供することで、記入漏れを防止
2. **Security Policy の明示**: セキュリティポリシーを独立セクションとして設け、本番環境への影響を防ぐ（ADR-001 の具現化）
3. **Out of Scope の重要性**: 「やらないこと」を明示することで、エージェントの過剰な自律行動を制限
4. **分離**: CLAUDE.md にはフレームワークの技術的ルールを、PROJECT_CONTEXT.md にはビジネス文脈を分離。責務が明確

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **固定フォーマット（入力フォームのような必須項目）** | プロジェクトの多様性に対応できない。SaaS / OSS / 社内ツール / データ分析等、プロジェクトの性質によって必要な情報が異なる |
| **自由記述（ノーフォーマット）** | 何を書けばよいか分からず、空のままになりがち。セキュリティポリシーのような重要項目が漏れるリスク |
| **CLAUDE.md に統合** | CLAUDE.md はフレームワークの技術的ルール（install.sh が自動追記する部分）とプロジェクト固有のルールが混在する。ビジネス文脈は別ファイルに分離すべき |
| **環境変数で注入** | 構造化された文脈情報を環境変数で管理するのは不適切。可読性・保守性が著しく低い |

### 影響（Consequences）

**正の影響:**
- エージェントがビジネス文脈を理解した上で判断できる（例: 「本番 DB は READ ONLY」を知っていれば、DELETE 文を生成しない）
- 新メンバーのオンボーディングにも利用可能（プロジェクトの全体像を1ファイルで把握）
- セキュリティポリシーの明示により、事故防止効果が高まる

**負の影響:**
- テンプレートの記入はユーザーの手動作業。未記入のまま使われる可能性
- 内容が古くなるリスク（プロジェクトの変化に合わせた更新が必要）
- エージェントが PROJECT_CONTEXT.md の内容を完全に理解・遵守する保証はない（LLM の限界）

### 更新履歴
- 2026-03-19: 初版

## ADR-010: フラグシステムと自動修復の統合

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-001, ADR-002

### 背景（Context）

Guardrail プロトコル（L1-L4）でエラーを検知した後の対応フローを設計する必要があった。検知（フラグ）と修復（リカバリ）を別々のシステムとして設計すると、以下の問題が発生する:

- 検知されたエラーの修復が遅れる（手動介入待ち）
- 同じエラーに対して検知と修復で異なる判断がされる
- エラーの重大度に応じたエスカレーション制御が困難

### 決定（Decision）

フラグシステム（検知・通知）と自動修復システム（回復）を統合設計し、検知→修復の一貫したフローを実現する。修復レベル（L0-L3）でエスカレーション制御を行う。

**統合フロー:**

```
エラー発生
  ↓
[検知] Guardrail が Level を判定
  ↓
[通知] PROGRESS プロトコルで即時表示
  ↓
[修復判定] Level に応じた自動修復の可否判断
  ↓
L1 (MONITORING): ログのみ。修復不要
L2 (CHECKPOINT): 自動修復を試行（最大3回）
  - test_failure_minor → Builder 修正 → 再テスト
  - type_error → Builder 型強化
  - lint_error → auto-fix
L3 (PAUSE): 自動修復を試行（最大2回）、失敗時は待機
  - test_failure_major → ロールバック + Sherpa 再分解
  - breaking_change → Architect 影響分析 + マイグレーション
  - build_failure → ロールバック + 修正
L4 (ABORT): 即時停止。ロールバック実行。手動介入要求
  ↓
[結果記録] 修復の成否をメモリに永続化
```

### 根拠（Rationale）

1. **一貫性**: 検知と修復が同一フロー内で行われるため、「検知されたが修復されない」状態が発生しない
2. **段階的エスカレーション**: L2 の自動修復が失敗したら L3 に、L3 が失敗したら L4 にエスカレーションする明確なパス
3. **自律性と安全性のバランス**: L1-L2 は自律的に回復、L3 で人間の確認を求め、L4 で強制停止。完全自動でも完全手動でもない
4. **学習ループ**: 修復結果をメモリに記録することで、同じエラーへの対応が改善される

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **検知のみ（修復は全て手動）** | 自律実行（AUTORUN_FULL モード）の価値が激減。L1/L2 レベルのエラーまで手動対応では生産性が低すぎる |
| **完全自動修復（全レベル）** | L4 レベル（クリティカルセキュリティ、データ整合性）の自動修復は危険。誤った修復がさらなる障害を引き起こすリスク |
| **検知と修復を別システム** | 検知システムが出力するフラグを修復システムが読み取る二段構成。レイテンシ増加、状態不整合、コードの重複が発生 |

### 影響（Consequences）

**正の影響:**
- 検知→修復の一貫したフローにより、エラー対応の平均時間（MTTR）が短縮
- 修復レベル（L0-L3）による明確なエスカレーション制御
- 自律実行モードでの安全性と生産性の両立
- 修復結果のメモリ永続化による学習効果

**負の影響:**
- 自動修復ロジックの実装・テスト・保守コスト
- 自動修復の誤判断リスク（特に L3 レベルのロールバック判断）
- 修復試行回数の上限設定が不適切だと、無限ループのリスク（上限で制御）

### 更新履歴
- 2026-03-19: 初版

## ADR-011: Sandbox + Permissions + Hooks の三重防御をデフォルト推奨

**Status:** 採用
**Date:** 2026-03-19
**Related:** ADR-001, ADR-002, ADR-007

### 背景（Context）

Claude Code の3つのセキュリティ機構（Sandbox / Permissions / Hooks）をどの組み合わせでデフォルト推奨するかを決める必要があった。初心者向けフレームワークとして、デフォルトが安全でなければ意味がない。一方、全てを強制するとセットアップの複雑性が増し、導入障壁になる。

### 決定（Decision）

以下の構成をデフォルト推奨とする:

| 機構 | 有効化方法 | デフォルト推奨 |
|------|-----------|---------------|
| **Hooks** | `install.sh --with-hooks` | 強く推奨（初心者は必須） |
| **Permissions** | `install.sh --with-permissions` | 推奨 |
| **Sandbox** | `settings.json` で有効化 | 推奨 |

install.sh のデフォルト推奨コマンド:

```bash
# 初心者向け（全オプション）
./install.sh --with-hooks --with-mcp --with-permissions

# 最低限（Hooks は必ず）
./install.sh --with-hooks
```

- `--with-hooks` なしでインストールした場合、インストーラーが警告メッセージを表示する
- ドキュメント・README の全インストール例に `--with-hooks` を付ける
- QUICKSTART.md で Hooks の重要性を冒頭で説明

### 根拠（Rationale）

1. **段階的な導入**: 全てを強制するのではなく、最低限の `--with-hooks` から始めて、慣れたら Permissions / Sandbox を追加する段階的な学習パスを提供
2. **最大の防御効果**: Hooks はツール実行前のリスク分類で最も直接的な防御効果がある。Permissions は静的な allow/deny、Sandbox は物理的な制限
3. **セットアップの容易さ**: `--with-hooks` は install.sh にフラグを追加するだけ。settings.json の手動編集が不要
4. **オプトアウト可能**: 熟練者は Hook を無効化できる。強制ではなく推奨とすることで、ユーザーの選択権を尊重

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **最小構成をデフォルト（セキュリティ機構なし）** | 初心者がセキュリティ機構なしで使い始めるリスクが高すぎる。「デフォルトが安全」でなければフレームワークの存在意義がない |
| **全部入りを強制（オプトアウト不可）** | 熟練者にとって過度な制限。確認ダイアログの頻出で生産性低下。「邪魔だから使わない」と判断されるリスク |
| **Permissions のみを推奨** | Permissions は静的リストで、コマンド引数レベルの制御ができない。`bash` を許可すると `rm -rf /` も許可される |

### 影響（Consequences）

**正の影響:**
- 初心者が安全なデフォルト構成で使い始められる
- 段階的な学習パスにより、セキュリティ機構の理解が深まる
- 熟練者は自分のニーズに合わせて調整可能

**負の影響:**
- `--with-hooks` の指定を忘れるユーザーがいる可能性（警告メッセージで軽減）
- 全オプション有効時のセットアップ時間がわずかに増加
- Hooks + Permissions + Sandbox の3つの設定ファイルを理解する学習コスト

### 更新履歴
- 2026-03-19: 初版

## ADR-012: 日本語ファーストの設計

**Status:** 採用
**Date:** 2026-03-19
**Related:** なし

### 背景（Context）

フレームワークの主要出力言語を決める必要があった。ドキュメント・警告メッセージ・エージェントの出力・エラーメッセージ・コメントなど、多くのテキスト出力の言語設計を統一する必要がある。

ターゲットユーザーの主要言語は日本語。Claude Code 自体は多言語対応だが、フレームワーク側で言語方針を定めないと、エージェントごとに出力言語がバラバラになるリスクがある。

### 決定（Decision）

日本語ファーストの設計とし、以下の方針を定める:

| 対象 | 言語 |
|------|------|
| ドキュメント（docs/, README.md） | 日本語 |
| CLAUDE.md / フレームワークプロトコル | 日本語（技術用語は英語可） |
| エージェントの出力 | 日本語 |
| 警告・エラーメッセージ | 日本語 |
| Tool Risk Hooks の警告表示 | 日本語 |
| コード内コメント | プロジェクトの方針に従う |
| 変数名・関数名 | 英語（プログラミング慣例） |
| コミットメッセージ | Conventional Commits（英語ヘッダ + 日本語本文可） |

CLAUDE.md の Core Principle に「All outputs in Japanese」を明記。`_framework.md` でエージェントの出力言語を日本語に指定。

### 根拠（Rationale）

1. **ユーザー理解速度**: 母語での情報処理速度は第二言語の2-3倍。特に警告メッセージ（セキュリティ関連）は瞬時に理解される必要がある
2. **初心者フレンドリー**: 英語のエラーメッセージは初心者にとってハードルになる。日本語であれば対応策の理解も速い
3. **フレームワークの一貫性**: 全エージェントが日本語で出力することで、チェーン実行時のコンテキスト切り替えコストが減る
4. **ターゲット市場**: 日本語圏のエンジニア向けフレームワークとしてのポジショニング

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **英語オンリー** | ターゲットユーザー（日本語話者の初心者エンジニア）にとってハードルが高い。特にセキュリティ警告が英語では、緊急時の理解が遅れる |
| **バイリンガル（日英併記）** | ドキュメントの分量が2倍になり、保守コストが倍増。「どちらが正か」の混乱も発生しうる。現時点ではオーバーエンジニアリング |
| **ユーザー設定で切り替え** | 実装コストが高い。全ドキュメント・全エージェント出力・全警告メッセージの二言語対応が必要。初期段階では日本語に集中し、需要が出たら国際化を検討 |

### 影響（Consequences）

**正の影響:**
- 日本語圏ユーザーの理解速度・学習速度が向上
- セキュリティ警告が母語で即座に理解される
- フレームワーク全体の出力が統一される

**負の影響:**
- 国際展開時に翻訳コストが発生
- 英語圏のユーザーは利用困難（ただし現時点ではターゲット外）
- 技術用語の日本語訳が統一されない場合の混乱リスク（用語集の整備で対応）

- 2026-03-19: 初版

## ADR-013: GlassWorm/Trojan Source 対策を Safety Gate BLOCK レベルで実装

**Status:** 採用
**Date:** 2026-03-21
**Related:** ADR-008（Safety Gate パターン）

### 背景（Context）

GlassWorm は Unicode の制御文字やゼロ幅スペース（U+200B 等）を悪用し、「見えないコード」をリポジトリに混入させるマルウェア。400 以上の GitHub/npm/VSCode 拡張が被害を受けた実績がある。従来のコードレビューでは検出困難であり、AI エージェントがプロンプトインジェクション経由で不可視文字入りのコードを生成するリスクもある。

goto-orchestrator の既存 Safety Gate は、シェルコマンドのパターンマッチによる検知だが、文字レベルの不正は検知対象外だった。

### 決定（Decision）

不可視 Unicode 文字の検知を **BLOCK（自動ブロック）** レベルで実装する。

1. **Bash コマンド**: Safety Gate パターンチェックの前段で `INVISIBLE_UNICODE_RE` による検査を実行。不可視文字を含むコマンドは即座に BLOCK
2. **Write/Edit ツール**: `content` / `new_string` に不可視文字が含まれる場合も BLOCK
3. **検知対象**: U+200B-200F, U+2060-2064, U+FEFF, U+202A-202E, U+2066-206F, U+00AD, U+034F, U+061C 等、30 種以上の不可視/制御文字

### 根拠（Rationale）

1. **HIGH ではなく BLOCK にした理由**: 不可視文字がコマンドやコードに含まれる正当なユースケースは事実上存在しない。誤って承認するリスクを排除するため BLOCK が適切
2. **Bash と Write/Edit の両方をカバー**: Bash コマンドだけでなく、AI が生成するコード（Write/Edit）にも混入の可能性があるため、両方で検知する
3. **偽陽性リスク**: 日本語（CJK）、絵文字、通常の ASCII 文字は検知対象外。検知対象は明確に「不可視」かつ「制御目的」の文字のみ

### 却下した代替案（Alternatives Rejected）

| 代替案 | 却下理由 |
|--------|---------|
| **HIGH（確認ダイアログ）** | 不可視文字は目視できないため、確認ダイアログで表示しても意味がない。BLOCK で即座に止めるべき |
| **外部ツール（ufoscan 等）依存** | 外部依存ゼロの設計原則に反する。正規表現で十分に検知可能 |
| **PostToolUse での事後検知** | 実行後では手遅れ。PreToolUse での事前ブロックが必要 |

### 影響（Consequences）

**正の影響:**
- GlassWorm / Trojan Source 攻撃を自動的にブロック
- AI 生成コードへの不可視文字混入も防止
- 外部ツール不要で検知可能（ゼロ依存維持）

**負の影響:**
- 極めて特殊なケース（Unicode 研究、テスト用途）で正当な不可視文字使用がブロックされる可能性（ただし実運用で発生する確率は極小）

### 更新履歴
- 2026-03-21: 初版
